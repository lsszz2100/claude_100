# 039. 버그 조사 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

버그 조사는 수정이 아니라 진단에서 시작합니다. Claude Code에게 바로 “고쳐줘”라고 하면 증상만 덮는 수정이 나올 수 있습니다.

좋은 버그 조사 프롬프트는 증상, 기대 동작, 실제 동작, 재현 절차, 로그, 최근 변경, 조사 범위를 포함합니다.

## 버그 리포트 구조

```text
증상:
[무슨 문제가 보이는가]

기대 동작:
[정상이라면 어떻게 되어야 하는가]

실제 동작:
[현재는 어떻게 되는가]

재현 절차:
1. [단계]
2. [단계]
3. [단계]

환경:
- OS:
- 브라우저/런타임:
- 브랜치:
- 관련 설정:

로그:
[오류 메시지 또는 스택 트레이스]

최근 변경:
[관련될 수 있는 최근 수정]
```

이 정도만 정리해도 Claude Code의 조사 품질이 크게 좋아집니다.

## 조사 전용 요청

```text
아래 버그를 조사해줘.

제약:
- 아직 파일은 수정하지 마.
- 먼저 재현 가능성을 판단해줘.
- 관련 파일과 로그를 근거로 원인 후보를 3개 이하로 좁혀줘.
- 각 후보마다 확인 방법을 제안해줘.
- 마지막에 가장 가능성 높은 수정 계획만 작성해줘.
```

“아직 파일은 수정하지 마”를 명시하면 조사와 수정이 분리됩니다.

## 로그를 줄 때 주의할 점

로그는 전체를 붙여 넣기보다 핵심 구간을 주는 것이 좋습니다. 단, 너무 짧게 잘라 최초 원인을 없애면 안 됩니다.

좋은 로그 제공:

```text
아래는 실패한 테스트의 처음 실패 지점부터 요약까지의 로그다.
최초 원인과 파생 오류를 구분해줘.
```

민감 정보가 있으면 값은 가립니다.

```text
DATABASE_URL=[redacted]
API_KEY=[redacted]
```

## 원인 후보를 검증하게 하기

Claude Code에게 원인 후보만 나열하게 하지 말고 검증 방법을 요구하세요.

```text
원인 후보별로 다음을 표로 정리해줘.

열:
- 후보 원인
- 근거
- 확인할 파일
- 확인할 코드 위치
- 맞다면 나타날 증거
- 아니라면 배제할 증거
```

이 표는 추측을 줄이고 다음 행동을 명확히 만듭니다.

## 수정 승인

조사 후 수정은 좁게 승인합니다.

```text
가장 가능성 높은 원인 1번만 수정해줘.

범위:
- [파일명]

제약:
- 다른 리팩터링은 하지 마.
- 공개 API는 바꾸지 마.
- 수정 후 재현 테스트 또는 관련 단위 테스트를 실행해줘.
```

버그 수정과 리팩터링을 섞지 않는 것이 핵심입니다.

## 실습

아래 상황을 버그 조사 프롬프트로 바꿔 보세요.

```text
로그인 후 새로고침하면 가끔 로그아웃 상태가 된다.
```

포함할 것:

- 재현 절차
- 기대 동작
- 실제 동작
- 최근 변경
- 조사 범위
- 수정 금지 조건

## Claude Code에 입력할 프롬프트

```text
아래 버그를 조사해줘.

증상:
[증상]

기대 동작:
[기대 동작]

실제 동작:
[실제 동작]

재현 절차:
[단계별 절차]

로그:
[오류 메시지 또는 없음]

최근 변경:
[관련 가능 변경 또는 모름]

제약:
- 아직 파일은 수정하지 마.
- 원인 후보를 3개 이하로 좁혀줘.
- 각 후보의 근거 파일과 확인 방법을 적어줘.
- 최초 원인과 파생 증상을 구분해줘.
- 마지막에 수정 계획을 제안해줘.
```

## 체크리스트

- [ ] 증상과 기대 동작이 분리되어 있다.
- [ ] 재현 절차가 있다.
- [ ] 로그와 최근 변경을 제공한다.
- [ ] 조사 단계에서는 파일 수정을 막는다.
- [ ] 원인 후보마다 검증 방법이 있다.

## 흔한 실수

- “안 돼요”만 말하고 재현 절차를 주지 않는다.
- 로그의 마지막 줄만 보고 원인을 판단한다.
- 조사 없이 바로 수정한다.
- 한 번에 여러 후보를 모두 고치려 한다.
