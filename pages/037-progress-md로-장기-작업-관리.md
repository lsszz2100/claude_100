# 037. progress.md로 장기 작업 관리

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

긴 작업은 대화 기록만으로 관리하면 위험합니다. 세션이 길어지거나 `/compact`, `/clear`, resume을 거치면 세부 맥락이 흐려질 수 있습니다. `progress.md`는 작업의 현재 상태를 파일로 고정해 다음 세션에서도 이어갈 수 있게 하는 작업 장부입니다.

`progress.md`는 공식 Claude Code 기능이라기보다 실무 운영 패턴입니다. Claude Code가 읽고 갱신하기 쉬운 Markdown 체크포인트를 만드는 것이 목적입니다.

## 언제 필요한가

다음 상황에서는 `progress.md`를 권장합니다.

- 하루 이상 이어지는 작업
- 여러 파일과 테스트가 관련된 기능
- 중간에 사용자 승인 단계가 많은 작업
- 여러 에이전트나 세션으로 나눠 진행하는 작업
- 실패한 테스트와 남은 TODO를 추적해야 하는 작업

작은 오타 수정처럼 10분 안에 끝나는 일에는 필요하지 않을 수 있습니다.

## 기본 템플릿

````markdown
# Progress: [작업 이름]

Last updated: YYYY-MM-DD HH:mm

## Goal

- [최종 목표]

## Current Status

- Status: Not started / In progress / Blocked / Ready for review / Done
- Current focus:
- Branch:
- Related issue/PR:

## Decisions

- [날짜] [결정 내용과 이유]

## Completed

- [x] [완료한 작업]

## In Progress

- [ ] [현재 진행 중인 작업]

## Next Steps

1. [다음 작업]
2. [다음 작업]
3. [다음 작업]

## Changed Files

- `[path]`: [변경 이유]

## Commands Run

```text
[command]
[result summary]
```

## Test Status

- Last passing:
- Failing:
- Not run:

## Risks / Open Questions

- [리스크 또는 질문]
````

위 템플릿은 길어 보이지만, 매번 모든 항목을 채울 필요는 없습니다. 중요한 것은 다음 사람이 이어받을 수 있는 최소 정보입니다.

## Claude Code에게 갱신시키기

작업 중간에 다음처럼 요청할 수 있습니다.

```text
지금까지의 작업을 progress.md에 갱신할 수 있게 요약해줘.

포함할 것:
1. 완료한 일
2. 변경한 파일
3. 실행한 명령과 결과
4. 남은 TODO
5. 실패했거나 실행하지 못한 테스트
6. 다음 세션에서 바로 시작할 작업

아직 파일은 수정하지 말고 먼저 초안만 보여줘.
```

초안을 확인한 뒤 파일 갱신을 승인합니다.

```text
좋아. 방금 초안 기준으로 progress.md를 갱신해줘.
기존 기록은 지우지 말고 날짜별로 누적해줘.
```

## /compact와 함께 쓰기

긴 세션에서는 `/compact` 전후로 `progress.md`를 활용하면 좋습니다.

```text
progress.md와 현재 변경사항을 기준으로
/compact에 넣을 요약을 만들어줘.
중요한 결정, 변경 파일, 테스트 상태, 다음 작업만 포함해줘.
```

이렇게 하면 대화 요약과 파일 기록이 서로 보완됩니다.

## 좋은 기록과 나쁜 기록

나쁜 기록:

```markdown
오늘 많이 수정함. 테스트는 나중에.
```

좋은 기록:

```markdown
- `src/auth/session.ts`: 만료된 refresh token 처리 분기 추가.
- `src/auth/session.test.ts`: refresh 실패 시 logout되는 테스트 추가.
- 실행: `npm run test:unit -- session` 통과.
- 미실행: 전체 E2E 테스트. 로컬 브라우저 의존성 미설치.
- 다음: 로그인 화면 리다이렉트 수동 확인.
```

좋은 기록은 다음 행동을 바로 만들 수 있습니다.

## 팀 작업에서의 주의점

`progress.md`를 저장소에 커밋할지 여부는 팀 성격에 따라 다릅니다.

- 개인 작업 기록이면 커밋하지 않고 로컬에만 둘 수 있습니다.
- 장기 기능 브랜치의 공유 기록이면 커밋해도 됩니다.
- 민감한 로그, 고객 정보, 토큰 값은 절대 넣지 않습니다.

공유할 경우 파일 위치를 명확히 정하세요.

```text
docs/progress/[feature-name].md
```

또는 개인 기록으로:

```text
.claude/progress.md
```

단, `.claude/` 아래 파일을 커밋할지 여부는 팀 설정과 저장소 정책에 맞춰야 합니다.

## 실습

현재 진행 중인 작업 하나를 골라 `progress.md` 초안을 만들어 보세요.

1. 목표를 한 문장으로 쓴다.
2. 완료한 일을 체크박스로 쓴다.
3. 변경 파일과 이유를 적는다.
4. 실행한 테스트와 결과를 적는다.
5. 다음 세션에서 시작할 작업 3개를 적는다.

## Claude Code에 입력할 프롬프트

```text
현재 작업 상태를 progress.md 형식으로 정리해줘.

확인할 것:
- 현재 변경된 파일
- 지금까지의 대화에서 결정한 내용
- 완료한 작업
- 남은 작업
- 실행한 명령과 테스트 결과
- 실행하지 못한 검증

제약:
- 비밀정보나 고객 데이터는 기록하지 마.
- 모르는 내용은 추정하지 말고 "확인 필요"라고 써줘.
- 아직 파일은 수정하지 말고 초안만 보여줘.
```

## 체크리스트

- [ ] 목표와 현재 상태가 적혀 있다.
- [ ] 완료, 진행 중, 다음 작업이 분리되어 있다.
- [ ] 변경 파일과 이유가 기록되어 있다.
- [ ] 테스트 실행 여부와 결과가 기록되어 있다.
- [ ] 다음 세션에서 바로 시작할 수 있다.

## 흔한 실수

- 대화 기록만 믿고 파일 기록을 남기지 않는다.
- 실패한 테스트를 기록하지 않는다.
- 민감한 로그나 토큰 값을 붙여 넣는다.
- 오래된 TODO를 정리하지 않아 현재 상태를 알 수 없게 만든다.
