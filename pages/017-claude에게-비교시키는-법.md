# 017. Claude에게 비교시키는 법

난이도: 초급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude는 비교를 잘하지만, 비교 기준을 주지 않으면 일반적인 장단점 목록을 만들 가능성이 큽니다. 실무에서 필요한 비교는 “무엇이 더 좋은가?”가 아니라 “우리 상황에서 어떤 선택이 더 적합한가?”입니다.

## 비교 요청의 기본 구조

```text
선택지:
비교 기준:
현재 상황:
우선순위:
결론 형식:
```

## 나쁜 비교 요청

```text
React랑 Vue 중 뭐가 좋아?
```

이 질문에는 정답이 없습니다. 팀의 경험, 프로젝트 규모, 생태계, 기존 코드, 채용, 유지보수 기준에 따라 달라집니다.

## 좋은 비교 요청

```text
React와 Vue를 비교해줘.

현재 상황:
- 팀은 React 경험이 조금 있다.
- 프로젝트는 관리자 대시보드다.
- 개발 기간은 6주다.
- 장기 유지보수가 중요하다.

비교 기준:
1. 학습 비용
2. 생태계
3. 채용 가능성
4. 대시보드 개발 생산성
5. 장기 유지보수

결론:
우리 상황에서 추천안을 하나 고르고, 반대 선택을 해야 하는 조건도 알려줘.
```

## 개발에서 자주 하는 비교

- 라이브러리 선택
- 프레임워크 선택
- 아키텍처 선택
- 데이터베이스 선택
- 테스트 전략 선택
- 배포 방식 선택
- 모델 선택
- MCP 서버 도입 여부
- Skill로 만들지, Hook으로 만들지, Subagent로 만들지

## 비교표 형식

```text
다음 열을 가진 표로 비교해줘.

기준 | 선택지 A | 선택지 B | 우리 상황에서 중요한 이유 | 추천
```

이 형식은 단순 장단점보다 실무 판단에 더 적합합니다.

## Claude Code에서 활용하기

Claude Code를 사용하다 보면 “직접 구현할까, 라이브러리를 쓸까?”, “테스트를 어디까지 만들까?”, “리팩터링을 먼저 할까, 버그만 고칠까?” 같은 판단이 자주 필요합니다. 이때 Claude에게 비교를 시키되, 최종 결정은 사용자가 해야 합니다.

## 실습

다음 선택지를 비교해 보세요.

```text
버그를 고치는 두 방법을 비교해줘.

선택지 A:
현재 코드에 최소 수정만 적용한다.

선택지 B:
문제가 생긴 모듈을 리팩터링한 뒤 수정한다.

현재 상황:
- 릴리스가 내일이다.
- 테스트가 부족하다.
- 이 모듈은 자주 변경된다.

비교 기준:
릴리스 위험, 장기 유지보수, 테스트 필요성, 작업 시간
```

## Claude에게 입력할 프롬프트

```text
다음 두 선택지를 비교해줘.

선택지:
A. [선택지 A]
B. [선택지 B]

현재 상황:
[상황 설명]

비교 기준:
1. 비용
2. 위험
3. 유지보수
4. 구현 시간
5. 검증 난이도

출력:
표로 비교하고, 마지막에 추천안과 추천하지 않는 조건을 함께 작성해줘.
```

## 체크리스트

- [ ] 비교 기준을 명시했다.
- [ ] 현재 상황을 제공했다.
- [ ] 추천안뿐 아니라 반대 조건도 요청했다.
- [ ] 단기 효과와 장기 비용을 구분했다.
- [ ] 결론을 그대로 따르지 않고 검토했다.
