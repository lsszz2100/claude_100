# 096. 보안 리뷰어 에이전트

난이도: 고급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

보안 리뷰어 에이전트는 인증, 권한, 입력 검증, 비밀정보, SSRF, dependency 위험을 검토합니다. 보안 에이전트는 기본적으로 읽기 전용으로 시작해야 합니다.

보안 리뷰의 목적은 모든 가능한 위험을 길게 나열하는 것이 아니라, 실제 공격 가능성과 영향이 있는 문제를 우선순위로 드러내는 것입니다.

## 설정 예시

````markdown
---
name: security-reviewer
description: Review code and configuration for security risks including authorization, authentication, injection, secrets exposure, SSRF, and unsafe dependencies.
tools: Read, Grep, Glob
---

You are a security reviewer.

Review priorities:
1. Broken access control
2. Authentication/session flaws
3. Injection and unsafe input handling
4. Secrets or PII exposure
5. SSRF and unsafe URL fetching
6. Security misconfiguration
7. Missing security tests

Output each finding with:
- Severity
- File/Line
- Attack scenario
- Impact
- Suggested fix
- Test case
````

## 사용 예시

```text
보안 리뷰어 에이전트를 사용해 인증 관련 변경사항을 검토해줘.

초점:
- 로그인하지 않은 사용자 접근
- 다른 사용자 데이터 접근
- 관리자 권한 우회
- 세션/토큰 처리
- 입력 검증
- 민감 정보 로그 출력

파일은 수정하지 말고 findings만 작성해줘.
```

## 심각도 기준

| 심각도 | 기준 |
| --- | --- |
| Critical | 인증 우회, 대량 데이터 노출, 원격 코드 실행 가능성 |
| High | 다른 사용자 데이터 접근, 권한 상승, secret 노출 |
| Medium | 특정 조건에서 악용 가능한 입력 검증 문제 |
| Low | 방어 심화, 로그 개선, 설정 명확화 |

## 검토 범위

보안 리뷰어 에이전트는 넓게 훑기보다 변경과 가까운 공격면을 먼저 봐야 합니다.

| 변경 유형 | 우선 확인 |
| --- | --- |
| 인증 | 세션 만료, 토큰 검증, refresh 흐름 |
| 권한 | 객체 소유권, 관리자 예외, tenant 격리 |
| 입력 처리 | injection, 파일 업로드, URL fetch |
| 로그/분석 | PII, secret, 토큰 마스킹 |
| dependency | 알려진 취약점, postinstall, supply chain |
| 설정 | CORS, CSP, cookie flag, debug mode |

보안 리뷰 결과가 유용하려면 “가능한 위험”보다 “이 변경에서 실제로 열리는 경로”가 먼저 나와야 합니다.

## 좋은 finding 예시

```text
Severity: High
File/Line: src/routes/users.ts:42
Attack scenario: 로그인한 사용자가 userId를 바꿔 다른 사용자의 프로필을 조회할 수 있음.
Impact: 개인정보 노출.
Suggested fix: 요청 userId와 세션 userId를 비교하고 관리자만 예외 허용.
Test case: 다른 사용자의 userId로 조회하면 403을 반환해야 함.
```

## 피해야 할 행동

- “OWASP를 검토하세요” 같은 일반론만 남긴다.
- 실제 코드 근거 없이 위험을 단정한다.
- 심각도 기준 없이 모든 항목을 High로 둔다.
- 수정 권한을 넓게 주고 보안 에이전트가 임의로 코드를 바꾼다.

## 위협 모델 질문

```text
이 변경의 위협 모델을 짧게 정리해줘.

확인:
1. 공격자는 누구인가?
2. 어떤 권한을 이미 가지고 있는가?
3. 어떤 데이터나 동작을 노리는가?
4. 현재 방어선은 어디인가?
5. 테스트로 확인할 수 있는 악용 경로는 무엇인가?
```

## 체크리스트

- [ ] 읽기 전용 도구만으로 시작했다.
- [ ] 공격 시나리오와 영향이 포함된다.
- [ ] 심각도 기준이 일관된다.
- [ ] 보안 테스트 케이스가 함께 제안된다.
- [ ] 변경 유형별 공격면을 먼저 좁혔다.
- [ ] 공격자와 기존 권한을 명시했다.
