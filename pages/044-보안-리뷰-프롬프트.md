# 044. 보안 리뷰 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

보안 리뷰는 일반 코드 리뷰보다 좁고 엄격합니다. Claude Code에게 보안 리뷰를 맡길 때는 OWASP Top 10과 OWASP API Security Top 10 같은 기준을 바탕으로 인증, 권한, 입력 검증, 비밀정보, 의존성, 로깅을 점검하게 해야 합니다.

보안 리뷰의 목표는 “불안한 느낌”을 말하는 것이 아니라, 공격 가능한 경로와 영향을 구체적으로 찾는 것입니다.

## 우선순위

웹 애플리케이션과 API에서 특히 먼저 볼 항목은 다음입니다.

| 영역 | 확인할 질문 |
| --- | --- |
| 접근 제어 | 다른 사용자의 데이터에 접근할 수 있는가? |
| 인증 | 토큰, 세션, 만료, 재발급 처리가 안전한가? |
| 입력 검증 | 사용자 입력이 SQL, command, template, path 등에 안전하게 들어가는가? |
| 민감 정보 | 토큰, 키, 개인정보가 로그나 응답에 노출되는가? |
| 리소스 사용 | rate limit, pagination, 파일 크기 제한이 있는가? |
| 서버 요청 | URL 입력으로 SSRF가 가능한가? |
| 설정 | CORS, 디버그 모드, 에러 응답이 과도하게 열려 있지 않은가? |

API에서는 특히 객체 단위 권한 확인이 중요합니다. `GET /users/{id}` 같은 엔드포인트는 “로그인 여부”뿐 아니라 “그 id에 접근할 권한”을 검사해야 합니다.

## 보안 리뷰 요청

```text
현재 변경사항을 보안 관점으로 리뷰해줘.

기준:
- OWASP Top 10
- OWASP API Security Top 10

우선 확인:
1. 인증 우회 가능성
2. 객체/기능 단위 권한 누락
3. 입력 검증과 injection 가능성
4. 민감 정보 로그/응답 노출
5. SSRF 또는 외부 URL 처리 위험
6. rate limit, pagination, 파일 크기 제한 누락
7. 보안 관련 테스트 공백

제약:
- 파일은 수정하지 마.
- 추측만으로 단정하지 마.
- 각 finding에 파일, 위치, 공격 시나리오, 영향, 수정 방향을 포함해줘.
```

## 출력 형식

```text
## Security Findings

- Severity: Critical / High / Medium / Low
- Category: Auth / Authorization / Injection / Secrets / SSRF / Config / Dependency
- File/Line:
- Problem:
- Attack scenario:
- Impact:
- Suggested fix:
- Test to add:

## No Finding Areas

- [확인했지만 중대한 문제가 없었던 영역]

## Open Questions

- [코드만으로 판단할 수 없는 질문]
```

보안 리뷰는 “중대한 문제 없음”도 의미 있는 결과입니다. 억지 finding을 만들지 않게 해야 합니다.

## API 권한 점검 프롬프트

```text
API 엔드포인트의 권한 검사를 점검해줘.

확인할 것:
- 로그인 여부만 확인하고 소유권을 확인하지 않는 곳
- role이 필요한데 일반 사용자도 호출할 수 있는 곳
- 요청 body의 userId, orgId, role 값을 신뢰하는 곳
- 목록 조회에서 tenant 또는 organization 필터가 빠진 곳
- 삭제/수정 API의 권한 테스트 공백

아직 파일은 수정하지 마.
```

## 민감 정보 점검 프롬프트

```text
민감 정보 노출 가능성을 점검해줘.

확인할 것:
- .env 또는 credential 파일 읽기/출력
- 로그에 토큰, 세션, 비밀번호, 개인정보 출력
- 에러 응답에 stack trace 또는 내부 경로 노출
- 테스트 fixture에 실제 키나 고객 데이터 포함
- 프론트엔드 번들에 서버 전용 secret 포함
```

## 보안 테스트 요청

보안 리뷰 후에는 테스트 공백을 요구하세요.

```text
위 보안 finding을 막기 위한 테스트 케이스를 제안해줘.
각 테스트는 Given/When/Then 형식으로 작성하고,
어느 테스트 파일에 추가할지 근거를 포함해줘.
아직 파일은 수정하지 마.
```

## 실습

아래 기능을 보안 리뷰 대상으로 가정해 프롬프트를 작성해 보세요.

```text
관리자가 사용자 role을 변경할 수 있는 API
```

반드시 인증, 관리자 권한, 대상 사용자 검증, audit log, 테스트 공백을 포함하세요.

## Claude Code에 입력할 프롬프트

```text
현재 변경사항을 보안 리뷰해줘.

범위:
[파일/기능/엔드포인트]

기준:
- OWASP Top 10
- OWASP API Security Top 10

출력:
1. Critical/High/Medium/Low finding
2. 파일과 위치
3. 공격 시나리오
4. 영향
5. 수정 방향
6. 추가할 테스트

제약:
- 아직 파일은 수정하지 마.
- 근거 없는 추측은 Open Questions로 분리해줘.
- 비밀정보 값은 출력하지 마.
```

## 체크리스트

- [ ] 인증과 권한을 구분해 검토한다.
- [ ] 객체 단위 권한과 기능 단위 권한을 확인한다.
- [ ] 사용자 입력이 위험한 sink로 들어가는지 본다.
- [ ] 민감 정보 로그/응답 노출을 확인한다.
- [ ] 보안 finding마다 테스트를 요구한다.

## 흔한 실수

- 로그인 여부만 보고 권한 검사를 했다고 판단한다.
- 보안 리뷰를 스타일 리뷰처럼 진행한다.
- 공격 시나리오 없이 “위험할 수 있음”만 적는다.
- 비밀정보를 그대로 로그나 리뷰 결과에 출력한다.
