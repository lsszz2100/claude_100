# 023. 작은 버그 고치기

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

Claude Code로 버그를 고칠 때는 “고쳐줘”보다 “재현, 원인 분석, 수정, 검증 순서로 진행해줘”가 안전합니다. 작은 버그라도 바로 수정부터 시작하면 증상만 가리고 원인을 놓칠 수 있습니다.

버그 수정의 기본 흐름은 네 단계입니다.

1. 증상을 정확히 적는다.
2. 재현 방법을 확인한다.
3. 원인을 좁힌다.
4. 작은 수정 후 테스트한다.

## 좋은 버그 요청의 재료

버그를 요청할 때는 다음 정보를 주면 좋습니다.

| 항목 | 예시 |
| --- | --- |
| 증상 | 저장 버튼을 눌러도 화면이 갱신되지 않는다 |
| 기대 동작 | 저장 후 목록에 새 항목이 보인다 |
| 실제 동작 | API는 성공하지만 목록은 그대로다 |
| 재현 절차 | 로그인, 새 항목 입력, 저장 클릭 |
| 오류 메시지 | 콘솔 또는 터미널 로그 |
| 최근 변경 | 어제 상태 관리 코드를 수정했다 |
| 제한 조건 | 먼저 원인 분석만 하고 파일은 수정하지 않는다 |

이 중 전부를 몰라도 됩니다. 모르는 항목은 “모름”이라고 적고, Claude Code에게 어떤 정보를 더 확인해야 하는지 물어보면 됩니다.

## 나쁜 요청

```text
버그 고쳐줘.
```

이 요청은 Claude Code가 어디서부터 봐야 할지 알 수 없습니다. 관련 없는 파일까지 넓게 읽거나, 잘못된 원인을 가정해 수정할 수 있습니다.

## 좋은 요청

```text
저장 버튼을 눌러도 새 항목이 목록에 바로 나타나지 않는다.

기대 동작:
- 저장 성공 후 목록에 새 항목이 보인다.

실제 동작:
- 새로고침하면 보이지만 저장 직후에는 보이지 않는다.

제약:
- 아직 파일은 수정하지 마.
- 먼저 관련 파일을 찾고 가능한 원인 후보를 정리해줘.
- 가장 가능성 높은 원인 1개만 선택한 뒤 수정 계획을 제안해줘.
```

## 원인 후보를 좁히는 질문

처음부터 수정하지 말고 후보를 좁히게 하세요.

```text
가능한 원인을 3개 이하로 정리해줘.
각 원인마다 확인할 파일, 확인할 코드 위치, 맞다면 나타날 증거를 적어줘.
```

이 질문은 Claude Code가 막연히 코드를 바꾸는 것을 줄입니다.

## 수정 범위를 작게 유지하기

원인을 찾았으면 수정 범위를 제한합니다.

```text
좋아. 지금은 목록 갱신 문제만 고쳐줘.
스타일 변경, 리팩터링, 파일명 변경은 하지 마.
수정 후 관련 테스트나 최소 검증 명령을 실행해줘.
```

작은 버그 수정에서 범위 제한은 매우 중요합니다. 버그 하나를 고치다가 주변 구조까지 바꾸면 검토하기 어려워집니다.

## 검증 요청

수정 후에는 반드시 검증을 요청합니다.

```text
수정이 끝나면 다음을 요약해줘.
1. 원인
2. 변경한 파일
3. 변경한 이유
4. 실행한 테스트 명령
5. 남은 위험
```

테스트를 실행할 수 없는 상황이라면, 왜 실행하지 못했는지와 수동 확인 방법을 받아야 합니다.

## 실습

연습용으로 다음 가짜 버그 리포트를 Claude Code 요청으로 바꿔 보세요.

```text
검색창에 단어를 입력해도 결과가 늦게 바뀌고,
가끔 이전 검색 결과가 남아 있다.
```

요청에는 기대 동작, 실제 동작, 재현 절차, 수정 전 분석 조건, 검증 방법을 포함하세요.

## Claude Code에 입력할 프롬프트

```text
아래 버그를 조사해줘.

증상:
[여기에 증상 입력]

기대 동작:
[여기에 기대 동작 입력]

실제 동작:
[여기에 실제 동작 입력]

재현 절차:
[여기에 재현 절차 입력]

제약:
- 아직 파일은 수정하지 마.
- 관련 파일을 먼저 찾고 원인 후보를 3개 이하로 좁혀줘.
- 각 후보의 근거 파일과 확인 방법을 적어줘.
- 마지막에 가장 가능성 높은 수정 계획을 제안해줘.
```

## 체크리스트

- [ ] 증상과 기대 동작을 구분한다.
- [ ] 재현 절차를 적는다.
- [ ] 수정 전에 원인 후보를 확인한다.
- [ ] 수정 범위를 작게 제한한다.
- [ ] 수정 후 테스트 또는 수동 검증 방법을 확인한다.

## 흔한 실수

- 오류 메시지를 일부만 붙여넣는다.
- 재현 절차 없이 결과만 말한다.
- 버그 수정과 리팩터링을 한 번에 요청한다.
- 테스트가 실패했는데 “대충 된 것 같다”고 넘어간다.
