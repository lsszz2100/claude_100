# 019. Claude에게 계획시키는 법

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

Claude Code에서 가장 중요한 습관 중 하나는 “바로 구현하지 말고 먼저 계획시키기”입니다. 계획은 시간 낭비가 아닙니다. 계획은 잘못된 파일을 수정하거나, 요구사항을 오해하거나, 테스트 없이 큰 변경을 하는 일을 줄이는 안전장치입니다.

## 언제 계획이 필요한가

다음 작업에는 계획이 필요합니다.

- 여러 파일을 수정하는 기능 개발
- 인증, 결제, 권한, 보안 관련 변경
- 데이터베이스 스키마 변경
- 대규모 리팩터링
- 외부 API 연동
- 배포 자동화
- MCP, Hooks, Skills 설정 변경

반대로 오타 수정이나 간단한 설명 요청에는 긴 계획이 필요하지 않습니다.

## 좋은 계획 요청

```text
다음 기능을 구현하려고 한다.
아직 파일은 수정하지 말고 구현 계획만 작성해줘.

기능:
사용자가 비밀번호를 재설정할 수 있어야 한다.

계획에 포함할 것:
1. 관련 파일 조사
2. 필요한 API 또는 화면
3. 데이터베이스 영향
4. 보안 고려사항
5. 테스트 전략
6. 예상 변경 파일
7. 단계별 구현 순서
8. 내가 확인해야 할 질문
```

## 계획에서 확인해야 할 것

Claude가 만든 계획을 그대로 승인하지 마세요. 다음을 확인해야 합니다.

- 요구사항을 정확히 이해했는가?
- 변경 범위가 과하게 넓지 않은가?
- 기존 패턴을 따르는가?
- 테스트 계획이 있는가?
- 보안 위험을 고려했는가?
- 되돌릴 방법이 있는가?
- 모르는 부분을 질문했는가?

좋은 계획은 모르는 것을 모른다고 말합니다. 나쁜 계획은 모든 것을 아는 척합니다.

## 계획 승인 문장

계획이 괜찮다면 이렇게 승인할 수 있습니다.

```text
계획의 1~3단계만 먼저 진행해줘.
파일 변경은 최소화하고, 변경 후 테스트 방법을 알려줘.
```

전체 계획을 한 번에 승인하기보다 일부 단계만 진행시키는 것이 안전합니다.

## 실습

아래 기능을 구현하기 전 계획을 요청해 보세요.

```text
관리자만 접근할 수 있는 사용자 관리 페이지를 추가하고 싶다.
```

계획에는 인증, 권한, 라우팅, UI, API, 테스트가 포함되어야 합니다.

## Claude에게 입력할 프롬프트

```text
다음 작업을 바로 구현하지 말고 계획만 작성해줘.

작업:
[작업 설명]

계획 형식:
1. 현재 구조에서 확인할 파일
2. 요구사항 해석
3. 구현 단계
4. 예상 변경 파일
5. 테스트 전략
6. 위험 요소
7. 내가 먼저 답해야 할 질문

추측이 필요한 부분은 추측이라고 표시해줘.
```

## 체크리스트

- [ ] 파일 수정 전 계획을 요구했다.
- [ ] 예상 변경 파일을 확인했다.
- [ ] 테스트 전략이 포함되어 있다.
- [ ] 위험 요소가 포함되어 있다.
- [ ] 모호한 요구사항에 대해 질문하게 했다.

## 흔한 실수

- 계획을 읽지 않고 승인한다.
- 너무 큰 계획을 한 번에 실행시킨다.
- 테스트 계획이 없는 계획을 승인한다.
- Claude가 추측한 부분을 확인하지 않는다.
