# 061. 세션 resume 전략

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code 세션을 다시 이어갈 때는 이전 대화만 믿으면 안 됩니다. 시간이 지나는 동안 파일이 바뀌었거나, 테스트 결과가 오래되었거나, 사용자가 직접 수정한 변경이 섞였을 수 있습니다.

resume의 첫 단계는 “기억을 이어받는 것”이 아니라 “현재 상태를 다시 확인하는 것”입니다.

## resume 첫 프롬프트

```text
이전 작업을 이어가자.

먼저 현재 상태를 확인해줘.
확인할 것:
1. 현재 작업 목표 요약
2. 현재 변경 파일 목록
3. 마지막으로 실행한 테스트와 결과
4. 남은 TODO
5. 이전 대화에서 추정한 내용 중 현재 파일로 다시 확인해야 할 것

아직 파일은 수정하지 마.
```

Git 저장소라면 `git status`, `git diff`를 먼저 확인하게 합니다. Git 저장소가 아니라면 최근 수정 파일 목록을 확인합니다.

## progress.md와 함께 쓰기

장기 작업에서는 `progress.md`를 resume 기준으로 삼습니다.

```text
progress.md를 읽고 현재 파일 상태와 비교해줘.

출력:
- progress.md와 일치하는 내용
- 현재 파일 상태와 달라진 내용
- 이어서 할 수 있는 작업
- 먼저 확인해야 할 위험

아직 파일은 수정하지 마.
```

## 오래된 테스트 결과 갱신

며칠 전 테스트 결과는 현재 상태를 보장하지 않습니다.

```text
이전 테스트 결과가 아직 유효한지 판단해줘.
변경된 파일을 기준으로 다시 실행해야 할 최소 테스트를 제안해줘.
```

## resume 후 바로 하면 안 되는 일

- 곧바로 파일 수정
- 오래된 계획 그대로 실행
- 이전 테스트 통과를 현재 통과로 가정
- 사용자가 만든 변경을 되돌리기
- unresolved conflict를 무시하기

## 인수인계 요약 요청

세션을 끝내기 전 다음 resume을 위해 요약을 남깁니다.

```text
다음 세션에서 바로 이어갈 수 있게 handoff 요약을 작성해줘.

포함할 것:
1. 목표
2. 완료한 일
3. 변경 파일
4. 테스트 결과
5. 남은 TODO
6. 주의할 점
7. 다음 첫 명령
```

## 체크리스트

- [ ] resume 직후 현재 파일 상태를 확인한다.
- [ ] 이전 계획을 현재 파일과 비교한다.
- [ ] 오래된 테스트 결과를 갱신한다.
- [ ] 사용자가 만든 변경을 되돌리지 않는다.
- [ ] 다음 resume을 위한 handoff를 남긴다.
