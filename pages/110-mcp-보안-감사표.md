# 110. MCP 보안 감사표

저자: AI_Innovation_Studio

MCP 서버는 Claude Code가 외부 도구와 데이터에 접근하게 해 줍니다. 연결 전에 무엇을 읽고 쓸 수 있는지, 어떤 인증을 쓰는지, 로그가 어디에 남는지 확인해야 합니다.

## 감사표

| 항목 | 질문 | 결과 |
| --- | --- | --- |
| 출처 | 서버의 제작자와 저장소를 신뢰할 수 있는가? |  |
| 권한 | 읽기와 쓰기 권한이 분리되어 있는가? |  |
| 인증 | 토큰 저장 위치와 갱신 방식이 명확한가? |  |
| 범위 | 접근 가능한 조직, 저장소, DB, 채널이 제한되어 있는가? |  |
| 네트워크 | 외부로 전송되는 데이터가 있는가? |  |
| 로그 | 요청과 응답이 어디에 기록되는가? |  |
| 업데이트 | 버전 고정과 변경 이력 확인이 가능한가? |  |
| 비상 대응 | 토큰 철회와 서버 비활성화 절차가 있는가? |  |

## 도입 전 질문

```text
이 MCP 서버를 도입하기 전에 보안 관점으로 검토해줘.

자료:
- 서버 설명:
- 필요한 권한:
- 접근 대상:
- 실행 환경:

출력:
1. 허용 가능한 권한
2. 과도한 권한
3. 민감 데이터 노출 위험
4. 도입 전 추가 질문
5. 최소 권한 설정안
```

## 운영 기준

- 읽기 전용으로 시작한다.
- 쓰기 권한은 명확한 사용 사례가 있을 때만 연다.
- 개인 토큰보다 조직이 관리하는 권한 체계를 우선한다.
- 사용하지 않는 MCP 서버는 제거한다.
- 권한 변경은 기록으로 남긴다.

## 위험도 분류

| 위험도 | 예시 | 기준 |
| --- | --- | --- |
| 낮음 | 공개 문서 읽기 | 민감 정보와 쓰기 권한이 없음 |
| 중간 | 내부 문서 검색 | 접근 범위와 로그 정책 확인 필요 |
| 높음 | GitHub issue, Jira, Slack 쓰기 | 사람 승인과 감사 로그 필요 |
| 매우 높음 | 운영 DB, 결제, 배포 도구 | 기본 차단, 별도 승인 절차 필요 |

MCP는 편리하지만 권한 경계가 흐려지기 쉽습니다. “읽을 수 있다”와 “쓸 수 있다”를 분리하고, 쓰기 작업은 초안 생성과 실제 반영 단계를 나누세요.

## 승인 등급

| 작업 | 승인 기준 |
| --- | --- |
| 공개 문서 읽기 | 사전 승인 없이 허용 가능 |
| 내부 문서 읽기 | 프로젝트 범위와 로그 정책 확인 |
| 댓글/메시지 초안 | 읽기 권한으로 허용 |
| 댓글/메시지 게시 | 게시 내용과 대상 확인 후 승인 |
| DB 쿼리 | read-only, row limit, PII 마스킹 확인 |
| 삭제/배포/결제 | 기본 차단, 별도 운영 절차 필요 |

## 정기 점검

- 사용 중인 MCP 서버 목록을 확인한다.
- 각 서버의 권한 범위가 현재 업무에 필요한지 확인한다.
- 더 이상 쓰지 않는 인증은 철회한다.
- 서버 버전과 변경 이력을 확인한다.
- 로그에 민감 데이터가 남지 않는지 샘플을 점검한다.

## 체크리스트

- [ ] 최소 권한으로 설정했다.
- [ ] 인증 정보 저장 위치를 확인했다.
- [ ] 민감 데이터 전송 가능성을 검토했다.
- [ ] 철회와 비활성화 절차를 문서화했다.
- [ ] 쓰기 도구는 초안과 승인 단계를 분리했다.
- [ ] 정기 권한 점검 주기를 정했다.
- [ ] 작업 유형별 승인 등급을 정했다.
