# 041. 테스트 생성 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

테스트 생성은 “테스트 좀 만들어줘”가 아니라 어떤 동작을 어떤 수준에서 검증할지 정하는 작업입니다. Claude Code에게 테스트를 맡길 때는 핵심 경로, 경계 조건, 실패 조건, 기존 테스트 스타일을 함께 확인하게 해야 합니다.

좋은 테스트는 구현 세부사항보다 사용자에게 중요한 동작을 검증합니다.

## 먼저 기존 테스트 스타일 확인

```text
아직 테스트를 추가하지 마.
먼저 이 프로젝트의 테스트 구조와 스타일을 확인해줘.

확인할 것:
1. 테스트 프레임워크
2. 테스트 파일 위치
3. 이름 규칙
4. mock/stub 사용 방식
5. 실행 명령
6. 비슷한 기능의 기존 테스트
```

기존 스타일을 따르는 것이 새 테스트를 유지보수하기 쉽게 만듭니다.

## 테스트 케이스 설계

구현 전에 테스트 케이스 목록을 먼저 받습니다.

```text
아래 기능에 대한 테스트 케이스를 설계해줘.

기능:
[기능 설명]

포함할 것:
- 정상 경로
- 경계 조건
- 실패 조건
- 권한/인증 조건
- 회귀 가능성이 높은 조건

아직 파일은 수정하지 마.
```

## Given/When/Then 형식

테스트 케이스는 Given/When/Then으로 쓰면 명확합니다.

```text
- Given 로그인하지 않은 사용자가 댓글 API를 호출할 때,
  When 댓글 작성을 요청하면,
  Then 401 응답을 반환한다.

- Given 댓글 작성자가 자신의 댓글을 볼 때,
  When 삭제를 요청하면,
  Then 댓글이 삭제되고 목록에서 보이지 않는다.
```

이 형식은 PRD의 수용 기준과 테스트를 연결하는 데 좋습니다.

## 테스트 추가 승인

설계가 괜찮으면 작은 범위부터 추가합니다.

```text
좋아. 정상 경로와 인증 실패 케이스만 먼저 테스트로 추가해줘.

제약:
- 기존 테스트 스타일을 따라줘.
- 제품 코드는 아직 수정하지 마.
- 테스트 추가 후 관련 테스트 명령을 실행해줘.
```

테스트가 실패하면 그 실패가 기대한 실패인지 확인합니다.

```text
테스트 실패가 새 테스트가 잡아낸 제품 코드 문제인지,
테스트 작성 오류인지 구분해줘.
아직 제품 코드는 수정하지 마.
```

## 테스트 품질 점검

테스트를 만든 뒤에는 품질을 검토합니다.

```text
방금 추가한 테스트를 검토해줘.

확인할 것:
1. 구현 세부사항에 과하게 의존하는지
2. 실제 사용자 동작을 검증하는지
3. 실패했을 때 원인이 잘 드러나는지
4. 중복이 과한지
5. 빠진 핵심 케이스가 있는지
```

## 실습

다음 기능에 대한 테스트 케이스를 설계해 보세요.

```text
사용자는 자신의 프로필 이름을 수정할 수 있다.
이름은 2자 이상 30자 이하이며, 공백만 있는 이름은 허용하지 않는다.
```

정상, 경계, 실패, 권한 조건을 각각 하나 이상 포함하세요.

## Claude Code에 입력할 프롬프트

```text
다음 기능의 테스트를 설계해줘.

기능:
[기능 설명]

제약:
- 아직 파일은 수정하지 마.
- 먼저 기존 테스트 구조와 스타일을 확인해줘.
- 테스트 케이스를 Given/When/Then 형식으로 제안해줘.
- 정상 경로, 경계 조건, 실패 조건, 권한 조건을 포함해줘.
- 각 테스트를 어느 파일에 둘지 근거와 함께 제안해줘.
```

## 체크리스트

- [ ] 기존 테스트 프레임워크와 스타일을 확인한다.
- [ ] 테스트 케이스를 먼저 설계한다.
- [ ] 정상, 경계, 실패, 권한 조건을 포함한다.
- [ ] 구현 세부사항보다 동작을 검증한다.
- [ ] 테스트 실행 명령과 결과를 확인한다.

## 흔한 실수

- 기존 테스트 스타일을 무시한다.
- 정상 케이스만 테스트한다.
- 테스트가 구현 내부 구조에 너무 의존한다.
- 실패한 테스트가 제품 문제인지 테스트 문제인지 구분하지 않는다.
