# 094. 테스트 엔지니어 에이전트

난이도: 고급  
기준일: 2026년 05월 03일

## 핵심 개념

테스트 엔지니어 에이전트는 테스트 전략, 실패 분석, 테스트 공백 찾기, 최소 검증 명령 제안을 전문으로 합니다. 코드 수정까지 맡길지, 읽기/실행 중심으로 둘지는 팀 정책에 따라 정합니다.

핵심은 “테스트를 많이 실행하는 것”이 아니라 “현재 변경을 검증하는 가장 작은 테스트부터 정확히 고르는 것”입니다.

## 설정 예시

````markdown
---
name: test-engineer
description: Use proactively after implementation or test failures to find relevant tests, analyze failures, and identify missing test coverage.
tools: Read, Grep, Glob, Bash
---

You are a test engineer.

Responsibilities:
- Identify the smallest relevant test command.
- Analyze failures by finding the first failing cause.
- Separate product bugs from test bugs.
- Propose missing test cases in Given/When/Then form.
- Never delete or weaken tests to make them pass.

Safety:
- Ask before running long test suites.
- Do not run deployment or migration commands.
- Summarize every command and result.
````

## 사용 예시

```text
테스트 엔지니어 에이전트를 사용해 이번 변경의 검증 계획을 세워줘.

출력:
1. 가장 작은 관련 테스트 명령
2. 그 다음 실행할 통합 테스트
3. 수동 확인이 필요한 항목
4. 추가해야 할 테스트 케이스
5. 실행 비용이 큰 테스트와 승인 필요 여부
```

실패 로그 분석:

```text
테스트 엔지니어 에이전트로 아래 실패 로그를 분석해줘.

요청:
- 최초 실패 원인
- 파생 오류
- 제품 코드 문제인지 테스트 코드 문제인지
- 가장 작은 수정 후보
- 수정 후 재실행할 명령
```

## 좋은 테스트 엔지니어의 판단

| 상황 | 판단 |
| --- | --- |
| 작은 UI 문구 변경 | 스냅샷 또는 화면 확인 중심 |
| 인증 로직 변경 | 단위, 통합, 권한 실패 케이스 |
| DB migration | dry run, rollback, 데이터 검증 쿼리 |
| dependency 업데이트 | 관련 테스트와 smoke test |
| 성능 개선 | 변경 전/후 측정 기준 |

## 테스트 비용 기준

테스트 엔지니어 에이전트는 항상 가장 작은 검증부터 제안해야 합니다.

| 단계 | 예시 | 목적 |
| --- | --- | --- |
| 정적 확인 | typecheck, lint, compile | 빠른 구조 오류 확인 |
| 단일 테스트 | 변경 파일과 직접 연결된 테스트 | 수정 영향 확인 |
| 통합 테스트 | API, DB, UI 흐름 | 모듈 간 계약 확인 |
| 전체 테스트 | release 전, shared module 변경 후 | 회귀 확인 |
| 수동 확인 | 시각적 UI, 외부 연동 | 자동화가 어려운 동작 확인 |

작은 변경에 전체 테스트만 반복하면 시간이 낭비되고, 큰 변경에 단일 테스트만 돌리면 위험이 남습니다. 변경 범위와 테스트 비용을 함께 설명하게 하세요.

## 피해야 할 행동

- 실패한 테스트를 삭제하거나 기대값을 무리하게 낮춘다.
- 전체 테스트만 반복 실행한다.
- 첫 오류보다 마지막 오류를 먼저 고친다.
- flake와 실제 버그를 구분하지 않는다.

## 실패 로그 요약 형식

```text

Next smallest command:
```

긴 로그를 그대로 붙여 넣기보다 첫 실패 지점과 다음 검증 명령을 분리하면 메인 작업자가 바로 판단할 수 있습니다.

## 체크리스트

- [ ] 최소 테스트 명령을 찾았다.
- [ ] 실패 원인을 최초 실패 지점부터 읽었다.
- [ ] 제품 버그와 테스트 버그를 구분했다.
- [ ] 추가 테스트가 Given/When/Then으로 표현된다.
- [ ] 테스트 비용과 범위가 함께 설명된다.
- [ ] 실패 로그는 첫 실패와 다음 명령 중심으로 요약된다.
