# 025. Git 변경사항 확인하기

난이도: 초급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code가 파일을 수정한 뒤에는 Git 변경사항을 확인해야 합니다. 변경된 파일 목록과 diff를 보면 Claude가 요청한 범위 안에서 작업했는지, 예상하지 못한 파일을 건드렸는지 알 수 있습니다.

초급자는 최소한 다음 두 가지 명령의 의미를 알아야 합니다.

```text
git status
git diff
```

`git status`는 어떤 파일이 바뀌었는지 보여주고, `git diff`는 파일 안에서 무엇이 바뀌었는지 보여줍니다.

## 먼저 파일 목록 확인

수정 후 Claude Code에게 이렇게 요청하세요.

```text
현재 Git 변경사항을 확인해줘.
변경된 파일 목록을 목적별로 분류하고,
내가 요청하지 않은 변경이 있는지 알려줘.
```

중요한 것은 변경 파일 수입니다. 작은 버그 하나를 고쳤는데 파일 20개가 바뀌었다면 이유를 확인해야 합니다.

## diff 요약 요청

파일 목록을 본 뒤 diff를 요약하게 합니다.

```text
git diff 기준으로 변경 내용을 요약해줘.

출력 형식:
1. 파일별 변경 의도
2. 동작 변경 여부
3. 테스트 변경 여부
4. 위험한 변경
5. 되돌리면 안 되는 변경과 되돌려도 되는 변경
```

Claude Code에게 diff를 요약하게 하면 검토 시간이 줄어듭니다. 하지만 최종 판단은 사용자가 해야 합니다.

## 요청 범위와 비교하기

Git 변경사항을 볼 때는 처음 요청과 비교하세요.

| 확인 질문 | 의미 |
| --- | --- |
| 요청한 파일만 바뀌었나? | 범위 이탈 여부 |
| 설정 파일이 바뀌었나? | 빌드나 실행 방식 변화 가능성 |
| 잠금 파일이 바뀌었나? | 의존성 변화 가능성 |
| 테스트 파일이 추가/수정됐나? | 검증 보강 여부 |
| 삭제된 파일이 있나? | 회귀 위험 |

특히 `package-lock.json`, `pnpm-lock.yaml`, `gradle.lockfile` 같은 잠금 파일이 바뀌었다면 의존성 변경이 있었는지 확인해야 합니다.

## Claude Code에 되묻기

변경사항이 예상보다 크면 바로 커밋하지 말고 묻습니다.

```text
이번 변경에서 요청 범위를 벗어난 부분이 있는지 스스로 검토해줘.
있다면 파일별로 이유와 되돌릴 수 있는지 알려줘.
아직 되돌리지는 마.
```

“아직 되돌리지는 마”를 붙이는 이유는 사용자가 만든 변경이 섞여 있을 수 있기 때문입니다. Git 작업에서는 남의 변경을 함부로 되돌리면 안 됩니다.

## 변경사항을 검토하는 순서

추천 순서는 다음과 같습니다.

1. `git status`로 파일 목록 확인
2. 새 파일, 삭제 파일, 설정 파일 확인
3. `git diff`로 실제 변경 내용 확인
4. Claude Code에게 파일별 요약 요청
5. 테스트 실행 결과와 연결
6. 커밋 또는 추가 수정 판단

## Git 저장소가 아닐 때

모든 폴더가 Git 저장소는 아닙니다. 이 경우 `git status`는 실패합니다. 그때는 최근 수정 파일 목록을 확인하고, 가능하면 작업 전 백업이나 버전 관리를 준비해야 합니다.

```text
이 폴더가 Git 저장소인지 확인해줘.
Git 저장소가 아니라면 최근 수정된 파일 목록을 보여주고,
지금부터 변경 이력을 안전하게 관리하는 방법을 제안해줘.
```

## 실습

작은 문서 파일 하나를 수정한 뒤 다음을 연습해 보세요.

1. `git status`로 변경 파일을 확인한다.
2. `git diff`로 실제 변경 내용을 확인한다.
3. Claude Code에게 변경 의도와 위험도를 요약하게 한다.
4. 요청 범위를 벗어난 변경이 있는지 묻는다.
5. 테스트 또는 검토 결과를 함께 정리한다.

## Claude Code에 입력할 프롬프트

```text
현재 변경사항을 검토해줘.

요청:
1. git status로 변경된 파일 목록을 확인해줘.
2. git diff를 기준으로 파일별 변경 의도를 요약해줘.
3. 내가 처음 요청한 범위를 벗어난 변경이 있는지 표시해줘.
4. 테스트가 필요한 변경과 문서 검토만 필요한 변경을 구분해줘.
5. 커밋 전에 확인해야 할 위험을 알려줘.

제약:
- 아직 커밋하지 마.
- 아직 파일을 되돌리지 마.
- 사용자가 만든 변경일 수 있는 파일은 따로 표시해줘.
```

## 체크리스트

- [ ] 수정 후 `git status`를 확인한다.
- [ ] `git diff`로 실제 변경 내용을 본다.
- [ ] 요청 범위를 벗어난 파일을 확인한다.
- [ ] 설정 파일과 잠금 파일 변경을 주의 깊게 본다.
- [ ] 커밋 전 테스트 결과를 함께 확인한다.

## 흔한 실수

- 변경 파일 목록만 보고 diff를 확인하지 않는다.
- AI가 만든 변경과 사용자가 만든 변경을 구분하지 않는다.
- 예상하지 못한 설정 파일 변경을 무시한다.
- 테스트 실패 상태에서 커밋 메시지만 먼저 작성한다.
