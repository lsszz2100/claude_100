# 052. 린트와 포맷 자동 수정

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

린트와 포맷은 코드 품질을 돕지만, 모든 오류가 같은 의미는 아닙니다. 포맷 오류는 자동 수정해도 안전한 경우가 많고, 린트 오류 중 일부는 실제 버그나 설계 문제를 드러냅니다.

Claude Code에게는 자동 수정 가능한 문제와 사람이 판단해야 하는 문제를 분리하게 해야 합니다.

## 먼저 명령 찾기

```text
이 프로젝트의 lint/format 명령을 찾아줘.

확인할 것:
- package 또는 build 설정
- lint 명령
- format 명령
- 자동 수정 명령
- CI에서 실행되는 명령

아직 실행하지 마.
```

## 요청 템플릿

```text
린트와 포맷 문제를 정리해줘.

단계:
1. lint/format 명령을 찾는다.
2. 먼저 check 명령만 실행한다.
3. 자동 수정 가능한 항목과 수동 판단 항목을 분리한다.
4. 자동 수정 전 변경 예상 범위를 설명한다.
5. 내가 승인하면 자동 수정 명령을 실행한다.

제약:
- 자동 수정 전에는 파일을 수정하지 마.
- 의미 있는 코드 변경과 포맷 변경을 섞지 마.
```

## 자동 수정 승인

```text
좋아. 포맷 자동 수정만 실행해줘.

제약:
- lint rule을 끄지 마.
- 테스트를 삭제하거나 무시하지 마.
- 자동 수정 후 git diff를 요약해줘.
- 남은 lint 오류는 별도로 설명해줘.
```

포맷 변경은 diff를 크게 만들 수 있으므로 기능 변경과 분리하는 편이 좋습니다.

## 남은 린트 오류 처리

자동 수정 후 남은 오류는 원인을 분류합니다.

```text
남은 lint 오류를 분류해줘.

분류:
1. 실제 버그 가능성
2. 타입 안정성 문제
3. 접근성 문제
4. 사용하지 않는 코드
5. 규칙 조정이 필요한 문제

각 항목의 수정 방향을 제안해줘.
```

## 금지할 행동

```text
금지:
- lint rule 비활성화로 문제 숨기기
- `eslint-disable` 남발
- 타입 오류를 any로 우회
- 테스트 삭제
- 관련 없는 파일 대량 포맷
```

규칙을 끄는 것은 마지막 선택입니다.

## 실습

현재 프로젝트에서 lint 명령만 찾고, 실행 전 계획을 세워 보세요. 자동 수정은 바로 하지 말고 변경 예상 범위를 먼저 확인합니다.

## Claude Code에 입력할 프롬프트

```text
린트/포맷 상태를 확인해줘.

요청:
1. 프로젝트의 lint/format 명령을 찾아줘.
2. check 명령과 fix 명령을 구분해줘.
3. 먼저 check만 실행할 계획을 제안해줘.
4. 자동 수정 가능한 오류와 수동 판단이 필요한 오류를 구분해줘.
5. 내가 승인하기 전에는 fix 명령을 실행하지 마.
```

## 체크리스트

- [ ] check 명령과 fix 명령을 구분한다.
- [ ] 자동 수정 전 변경 예상 범위를 확인한다.
- [ ] 포맷 변경과 기능 변경을 분리한다.
- [ ] 남은 lint 오류를 의미별로 분류한다.
- [ ] 규칙 비활성화로 문제를 숨기지 않는다.

## 흔한 실수

- 처음부터 fix 명령을 실행한다.
- 포맷 변경과 기능 변경을 한 커밋에 섞는다.
- lint rule을 꺼서 통과시킨다.
- 남은 오류를 전부 스타일 문제로 취급한다.
