# 030. 초급자가 피해야 할 실수

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

Claude Code 초급자가 가장 조심해야 할 것은 AI의 능력이 아니라 작업 범위입니다. Claude Code는 파일을 읽고 수정하고 명령을 실행할 수 있으므로, 애매한 요청은 실제 프로젝트에 애매한 변경을 만들 수 있습니다.

안전한 사용의 기본은 단순합니다.

1. 먼저 읽고 분석한다.
2. 바로 수정하지 않는다.
3. 범위를 작게 잡는다.
4. 변경사항을 확인한다.
5. 테스트하거나 검토한다.

## 실수 1: “전체를 알아서 고쳐줘”

초급자가 가장 많이 하는 위험한 요청입니다.

```text
전체 코드 정리해줘.
```

이 요청은 범위도, 목적도, 완료 기준도 없습니다. 결과적으로 리팩터링, 스타일 변경, 파일 이동, 동작 변경이 섞일 수 있습니다.

더 나은 요청:

```text
아직 파일은 수정하지 마.
src/components 폴더에서 중복이 큰 파일 후보만 찾아줘.
각 후보에 대해 개선 이유와 위험도를 표로 정리해줘.
```

## 실수 2: 테스트를 생략한다

AI가 수정한 코드도 테스트가 필요합니다. 설명이 그럴듯해도 실행이 실패할 수 있습니다.

수정 후에는 다음을 요청하세요.

```text
수정한 변경과 관련 있는 최소 테스트를 찾아 실행해줘.
실행할 수 없다면 이유와 수동 확인 방법을 알려줘.
```

테스트를 못 돌린 경우에는 그 사실을 기록해야 합니다.

## 실수 3: 권한 승인을 무비판적으로 누른다

Claude Code는 명령 실행이나 파일 변경 과정에서 승인을 요청할 수 있습니다. 초급자는 승인 전에 명령의 목적과 영향 범위를 확인해야 합니다.

승인 전에 물어볼 질문:

- 이 명령은 어떤 파일이나 폴더를 바꾸는가?
- 삭제, 이동, 덮어쓰기가 포함되는가?
- 네트워크 접근이나 패키지 설치가 필요한가?
- 운영 데이터나 비밀정보에 접근하는가?
- 되돌릴 방법이 있는가?

이해하지 못한 명령은 승인하지 말고 설명을 요청하세요.

## 실수 4: 비밀정보를 그대로 넣는다

API 키, 토큰, 비밀번호, 고객 개인정보, 내부 계약 정보는 조심해야 합니다. Claude에게 문제를 설명할 때는 민감한 값을 가려도 됩니다.

나쁜 예:

```text
실제 API 키 값을 그대로 붙여넣음
이거 왜 안 돼?
```

좋은 예:

```text
API_KEY가 설정되어 있지만 인증 오류가 난다.
실제 키 값은 공유하지 않겠다.
환경 변수 이름, 오류 메시지, 설정 파일 구조를 기준으로 원인을 봐줘.
```

비밀정보는 문제 해결에 필요한 구조만 제공하고 값은 숨기세요.

## 실수 5: 사용자가 만든 변경을 덮어쓴다

작업 중인 폴더에는 사용자가 직접 바꾼 파일이 있을 수 있습니다. Claude Code에게 되돌리기나 대규모 수정 작업을 맡길 때는 주의해야 합니다.

```text
현재 변경사항 중 내가 만든 변경과 네가 만든 변경이 섞였을 수 있다.
어떤 파일이 바뀌었는지 먼저 확인하고,
되돌릴 후보만 제안해줘. 아직 되돌리지는 마.
```

이 습관은 협업 프로젝트에서 특히 중요합니다.

## 실수 6: 오래된 맥락을 계속 끌고 간다

긴 세션에서는 이전 작업의 맥락이 새 작업에 섞일 수 있습니다. 주제가 바뀌면 `/clear`, 같은 작업을 이어가면 `/compact`를 사용하세요.

```text
/compact 지금까지의 결정, 변경 파일, 남은 TODO, 테스트 결과만 요약해줘.
```

또는 새 작업을 시작할 때:

```text
/clear
```

## 안전한 기본 프롬프트

초급자는 다음 문장을 자주 사용해도 됩니다.

```text
아직 파일은 수정하지 마.
먼저 현재 상태를 분석하고,
관련 파일과 가능한 접근 방법,
수정 시 위험한 부분을 정리해줘.
```

수정을 승인할 때도 범위를 제한하세요.

```text
좋아. 제안한 계획 중 1단계만 진행해줘.
수정 범위는 [파일/폴더]로 제한하고,
끝나면 변경 파일과 테스트 결과를 요약해줘.
```

## 실습

아래 위험한 요청을 안전한 요청으로 바꿔 보세요.

```text
내 프로젝트 전체를 최신 스타일로 정리해줘.
```

포함해야 할 것:

- 읽기 전용 분석 조건
- 작업 범위
- 수정 금지 조건
- 위험도 확인
- 테스트 또는 검토 방법

## Claude Code에 입력할 프롬프트

```text
나는 Claude Code 초급자다.
아래 요청이 위험한지 검토하고 더 안전한 요청으로 바꿔줘.

원래 요청:
[여기에 요청 입력]

출력:
1. 위험한 이유
2. 빠진 정보
3. 먼저 확인해야 할 질문
4. 읽기 전용 분석 프롬프트
5. 수정 승인용 프롬프트
6. 검증 체크리스트
```

## 체크리스트

- [ ] 큰 작업은 먼저 분석만 요청한다.
- [ ] 수정 범위를 파일 또는 폴더 단위로 제한한다.
- [ ] 승인 전 명령의 목적과 영향 범위를 이해한다.
- [ ] 비밀정보는 값이 아니라 구조만 공유한다.
- [ ] 수정 후 Git 변경사항과 테스트 결과를 확인한다.
- [ ] 주제가 바뀌면 세션 맥락을 정리한다.

## 흔한 실수

- “알아서”, “전체”, “최신식” 같은 모호한 표현을 쓴다.
- 테스트 없이 설명만 믿고 넘어간다.
- 삭제나 이동 명령을 이해하지 못한 채 승인한다.
- 비밀정보를 그대로 붙여넣는다.
- `/clear`와 `/compact`를 구분하지 못한다.
