# Part 5. 실무 개발 워크플로

이 장은 중급 단계 독자를 위한 흐름입니다. 각 페이지는 하나의 주제를 다루며, 개념 설명과 실습 프롬프트를 함께 제공합니다.

## 처음 읽는 분께

이 파트는 용어를 외우는 장이 아니라 실제 작업에서 판단 기준을 세우는 장입니다. 각 항목을 읽을 때는 “언제 쓰는가”, “무엇을 조심해야 하는가”, “어떻게 검증하는가”를 함께 확인하세요.

초보자는 본문을 읽은 뒤 바로 실습 프롬프트를 실행해 보는 것이 좋습니다. 실행 결과가 예상과 다르면 실패로 보지 말고, 어떤 맥락이 부족했는지 기록하세요. 그 기록이 다음 요청을 더 정확하게 만드는 재료가 됩니다.

## 이 장에서 다루는 100선 항목

- 038. 기능 개발 프롬프트: 기능 구현을 요구사항, 영향 범위, 테스트, 문서까지 포함해 요청한다.
- 039. 버그 조사 프롬프트: 증상, 재현 절차, 기대 동작, 실제 동작, 로그를 제공해야 한다.
- 040. 리팩터링 프롬프트: 리팩터링은 동작 유지와 검증 계획이 핵심임을 배운다.
- 041. 테스트 생성 프롬프트: 핵심 경로, 경계 조건, 실패 조건을 테스트하게 한다.
- 042. 문서 생성 프롬프트: 문서는 코드의 현재 동작과 동기화되어야 한다.
- 043. 코드 리뷰 프롬프트: 리뷰는 심각도, 파일 위치, 재현 가능성, 수정 제안으로 구성한다.
- 044. 보안 리뷰 프롬프트: 입력 검증, 인증, 권한, 비밀정보, 의존성 위험을 점검한다.
- 045. 성능 개선 프롬프트: 측정 없는 성능 개선은 추측이므로 병목 측정부터 요구한다.
- 046. UI 개선 프롬프트: UI 요청에는 대상 사용자, 정보 밀도, 반응형 조건, 접근성을 포함한다.
- 047. API 설계 요청법: 엔드포인트, 요청/응답, 오류, 인증, 테스트를 함께 설계한다.
- 048. 데이터베이스 변경 요청법: 스키마 변경은 마이그레이션, 기존 데이터, 롤백을 함께 고려한다.
- 049. 마이그레이션 작업 요청법: 데이터 변경 작업은 dry run, 백업, 검증 쿼리가 필요하다.
- 050. 의존성 업데이트 요청법: 라이브러리 업데이트는 changelog, breaking changes, 테스트가 핵심이다.
- 051. 실패한 빌드 고치기: 빌드 로그를 읽고 최초 원인과 파생 오류를 구분한다.
- 052. 린트와 포맷 자동 수정: 스타일 문제와 의미 있는 코드 문제를 분리한다.
- 053. PR 생성 워크플로: 브랜치, 커밋, 테스트, PR 설명, 리뷰 포인트를 연결한다.
- 054. 릴리스 노트 자동화: 커밋과 PR에서 사용자 관점 변경사항을 추출한다.
- 055. 반복 작업 자동화: 반복적이고 기준이 명확한 작업은 자동화 후보가 된다.

## 학습 방식

각 항목을 읽은 뒤 바로 Claude 또는 Claude Code에 실습 프롬프트를 입력해 보세요. 읽기만 하면 기능 목록으로 남고, 직접 실행하면 작업 습관으로 바뀝니다.
