# 011. 좋은 질문과 나쁜 질문

난이도: 초급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude를 잘 쓰는 첫 번째 기술은 좋은 질문을 하는 것입니다. 좋은 질문은 길기만 한 질문이 아닙니다. 좋은 질문은 목표가 분명하고, 필요한 맥락이 있으며, 결과 형식과 제약 조건이 명확한 질문입니다.

나쁜 질문은 Claude가 추측하게 만듭니다. 좋은 질문은 Claude가 판단할 기준을 갖게 만듭니다. 이 차이는 글쓰기에서는 문체 차이로, 개발에서는 코드 품질 차이로, Claude Code에서는 실제 파일 변경의 안전성 차이로 나타납니다.

## 나쁜 질문의 특징

나쁜 질문은 보통 다음 중 하나에 해당합니다.

- 목표가 없다.
- 맥락이 없다.
- 범위가 없다.
- 원하는 출력 형식이 없다.
- 완료 기준이 없다.
- 금지해야 할 행동이 없다.

예를 들어 다음 요청은 위험합니다.

```text
이 코드 좀 좋게 고쳐줘.
```

이 요청은 “좋게”의 기준이 없습니다. 성능을 좋게 하라는 것인지, 가독성을 높이라는 것인지, 버그를 고치라는 것인지, 타입 안정성을 높이라는 것인지 알 수 없습니다.

## 좋은 질문의 구조

좋은 질문은 다음 구조를 가집니다.

```text
목표:
맥락:
작업 범위:
제약 조건:
원하는 출력:
검증 방법:
```

위의 나쁜 질문을 좋은 질문으로 바꾸면 다음처럼 됩니다.

```text
목표:
이 함수의 가독성을 높이고 중복을 줄이고 싶다.

맥락:
현재 동작은 유지해야 한다. 성능 최적화보다 유지보수성이 중요하다.

작업 범위:
src/utils/date.ts 파일 안의 formatDate 함수만 검토해줘.

제약 조건:
아직 파일은 수정하지 말고 개선 계획만 제안해줘.

원하는 출력:
문제점, 개선 방향, 수정 후 테스트 방법을 표로 정리해줘.
```

## Claude Code에서 더 중요한 이유

일반 채팅에서는 나쁜 질문을 해도 잘못된 답변이 텍스트로 끝납니다. 하지만 Claude Code에서는 실제 파일이 수정될 수 있습니다. 그래서 Claude Code에서는 좋은 질문이 안전장치입니다.

Claude Code에 처음 요청할 때는 다음 문장을 자주 사용하세요.

```text
아직 파일은 수정하지 말고 먼저 분석과 계획만 작성해줘.
```

이 한 문장만으로도 불필요한 변경을 크게 줄일 수 있습니다.

## 실습

아래 나쁜 질문을 좋은 질문으로 바꿔 보세요.

```text
내 앱을 더 좋게 만들어줘.
```

예시 답안:

```text
목표:
사용자가 첫 화면에서 핵심 기능을 더 빨리 이해하도록 개선하고 싶다.

맥락:
이 앱은 개인 일정 관리 앱이다. 주요 사용자는 바쁜 직장인이다.

작업 범위:
홈 화면 UI와 온보딩 문구만 검토해줘.

제약 조건:
아직 코드는 수정하지 마. 디자인 개선안만 제안해줘.

원하는 출력:
문제점 5개, 개선안 5개, 우선순위, 예상 효과를 표로 정리해줘.
```

## Claude에게 입력할 프롬프트

```text
아래 요청을 Claude Code에 적합한 좋은 요청으로 바꿔줘.

원래 요청:
[여기에 내가 쓴 요청 입력]

다음 구조로 다시 작성해줘.
1. 목표
2. 맥락
3. 작업 범위
4. 제약 조건
5. 원하는 출력 형식
6. 검증 방법
7. Claude Code에서 바로 쓰기 좋은 최종 프롬프트
```

## 체크리스트

- [ ] 요청에 목표가 있다.
- [ ] Claude가 알아야 할 맥락이 있다.
- [ ] 작업 범위가 너무 넓지 않다.
- [ ] 수정 금지나 읽기 전용 조건을 필요할 때 명시한다.
- [ ] 원하는 출력 형식을 지정한다.
- [ ] 검증 방법을 포함한다.

## 흔한 실수

- “좋게”, “깔끔하게”, “알아서” 같은 표현만 사용한다.
- 한 번의 요청에 설계, 구현, 테스트, 배포를 모두 넣는다.
- Claude가 현재 프로젝트 맥락을 이미 알고 있다고 가정한다.
- 결과 형식을 지정하지 않아 장황한 답변을 받는다.
