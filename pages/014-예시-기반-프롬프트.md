# 014. 예시 기반 프롬프트

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

Claude에게 원하는 결과의 예시를 보여주면 답변 품질이 안정됩니다. 사람에게 일을 맡길 때도 “이런 느낌으로 해주세요”라고 예시를 주면 결과가 좋아집니다. Claude도 마찬가지입니다.

예시 기반 프롬프트는 특히 문체, 문서 구조, 코드 리뷰 형식, JSON 출력, 테스트 케이스 작성, 커밋 메시지 작성에 유용합니다.

## 좋은 예시의 조건

좋은 예시는 다음 조건을 갖습니다.

- 짧고 명확하다.
- 원하는 구조를 보여준다.
- 좋은 기준과 나쁜 기준을 구분한다.
- 실제 작업과 비슷하다.
- 복사 가능한 형식이다.

## 예시 없는 요청

```text
커밋 메시지를 작성해줘.
```

이 요청도 작동하지만 원하는 스타일과 다를 수 있습니다.

## 예시가 있는 요청

```text
다음 스타일로 커밋 메시지를 작성해줘.

예시:
feat(auth): add email validation to signup form
fix(api): handle empty search query
docs(readme): update local setup guide

규칙:
- 영어로 작성
- Conventional Commit 형식 사용
- 제목은 72자 이하
- 본문에는 변경 이유와 테스트 여부 포함
```

이 요청은 훨씬 안정적입니다. Claude는 형식과 문체를 동시에 이해합니다.

## 좋은 예시와 나쁜 예시를 함께 주기

때로는 원하는 것뿐 아니라 원하지 않는 것도 보여주는 편이 효과적입니다.

```text
좋은 예:
- 심각도: 높음
- 위치: src/api/login.ts
- 문제: 로그인 실패 시 동일한 오류 메시지를 반환하지 않아 사용자 열거 위험이 있음
- 제안: 실패 사유를 일반화하고 테스트 추가

나쁜 예:
- 로그인 코드가 좀 이상함
- 고쳐야 할 듯
```

Claude에게 이런 예시를 주면 답변이 더 구체적이고 검토 가능한 형태로 나옵니다.

## 실습

아래 요청에 예시를 추가해 보세요.

```text
README를 더 좋게 만들어줘.
```

예시 기반으로 바꾸면 다음과 같습니다.

```text
README의 "로컬 실행" 섹션을 개선하고 싶다.

좋은 예:
## 로컬 실행
1. 의존성 설치
2. 환경 변수 설정
3. 개발 서버 실행
4. 테스트 실행

나쁜 예:
대충 npm install 하고 실행하면 됨

규칙:
- 초급자도 따라 할 수 있게 작성
- 명령어는 코드 블록으로 제공
- 운영 비밀키는 포함하지 않음
```

## Claude에게 입력할 프롬프트

```text
내 요청에 예시를 추가해서 더 좋은 프롬프트로 바꿔줘.

원래 요청:
[여기에 요청 입력]

출력:
1. 좋은 예시
2. 나쁜 예시
3. 최종 프롬프트
4. Claude가 헷갈릴 수 있는 부분
```

## 체크리스트

- [ ] 원하는 결과 예시를 하나 이상 제공한다.
- [ ] 필요하면 나쁜 예시도 제공한다.
- [ ] 예시가 실제 작업과 너무 동떨어져 있지 않다.
- [ ] 형식, 길이, 문체 규칙을 함께 준다.

## 흔한 실수

- 예시가 너무 길어 핵심이 흐려진다.
- 예시와 실제 요청의 형식이 다르다.
- 나쁜 예시만 주고 좋은 예시를 주지 않는다.
- 예시를 주면서도 출력 형식을 따로 지정하지 않는다.
