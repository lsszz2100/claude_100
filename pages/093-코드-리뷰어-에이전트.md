# 093. 코드 리뷰어 에이전트

난이도: 고급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

코드 리뷰어 에이전트는 변경사항에서 버그, 회귀, 보안 문제, 테스트 공백을 찾는 전문 subagent입니다. 일반 설명자보다 엄격한 리뷰어 역할을 부여해야 합니다.

좋은 코드 리뷰어는 “괜찮아 보입니다”로 끝내지 않습니다. 실제 사용자 영향, 실패 조건, 테스트 누락, 배포 위험을 먼저 찾습니다.

## 설정 예시

````markdown
---
name: code-reviewer
description: Use proactively after code changes to review diffs for bugs, regressions, security risks, and missing tests before merge.
tools: Read, Grep, Glob
---

You are a senior code reviewer.

Review priorities:
1. Correctness bugs
2. Regressions
3. Security risks
4. Missing tests
5. Maintainability risks

Rules:
- Lead with findings.
- Include severity and file/line when possible.
- Do not praise the code.
- Do not focus on style unless it hides a real risk.
- If no significant issues are found, say so clearly.

Output:
- Findings
- Open questions
- Test gaps
- Residual risk
````

## 리뷰 대상

- 현재 `git diff`
- PR diff
- 특정 파일 변경
- 테스트 실패 후 수정 diff
- 보안 패치 diff
- dependency 업데이트 diff

## 리뷰 범위 선언

코드 리뷰어 에이전트에게는 리뷰하지 않을 것도 알려주는 편이 좋습니다. 그래야 문체나 취향 문제가 findings를 흐리지 않습니다.

```text
리뷰 범위:
- correctness
- regression
- security
- missing tests

리뷰 제외:
- 단순 취향 차이
- 기존 코드 스타일 전체 개선
- 요청 범위 밖 리팩터링 제안
- 이미 별도 이슈로 추적 중인 문제
```

리뷰 대상이 크면 먼저 파일 목록과 위험 영역을 나누게 한 뒤, 위험도가 높은 부분부터 보게 하세요.

## 좋은 요청 예시

```text
코드 리뷰어 에이전트를 사용해 현재 변경사항을 검토해줘.

리뷰 기준:
1. 실제 버그 가능성
2. 기존 동작 회귀
3. 인증/권한/입력 검증 문제
4. 테스트 공백
5. 운영 배포 위험

스타일 지적은 실제 버그나 유지보수 위험으로 이어질 때만 포함해줘.
```

## finding 형식

```text
Severity: High | Medium | Low
File/Line:
Problem:
Why it matters:
Suggested fix:
Test to add:
```

## 실패 패턴

- 변경 의도를 이해하지 않고 코드 스타일만 지적한다.
- 라인 근거 없이 일반론만 말한다.
- 테스트 공백을 “테스트 추가 필요” 한 줄로 끝낸다.
- 보안 위험을 실제 공격 시나리오 없이 나열한다.
- 심각도 구분 없이 모든 의견을 같은 무게로 둔다.

## 리뷰 결과 처리

리뷰 결과는 그대로 수정 목록이 아닙니다. 메인 작업자는 findings를 다음처럼 분류해야 합니다.

| 분류 | 처리 |
| --- | --- |
| 실제 버그 | 수정하고 회귀 테스트 추가 |
| 불확실한 위험 | 재현 조건이나 코드 근거 추가 확인 |
| 테스트 공백 | 변경 범위에 맞는 최소 테스트 추가 |
| 설계 질문 | 사용자나 팀 결정으로 넘김 |
| 범위 밖 제안 | 별도 이슈로 분리하거나 보류 |

## 체크리스트

- [ ] findings가 요약보다 먼저 나온다.
- [ ] 각 finding에 재현 가능하거나 합리적인 근거가 있다.
- [ ] 테스트 공백이 구체적인 케이스로 제시된다.
- [ ] 문제가 없으면 그 사실과 남은 위험을 분리해 말한다.
- [ ] 리뷰 제외 범위가 명확하다.
- [ ] findings를 수정, 확인, 보류로 분류했다.
