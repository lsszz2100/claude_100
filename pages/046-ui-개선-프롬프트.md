# 046. UI 개선 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

UI 개선은 “예쁘게 해줘”가 아니라 사용자가 더 쉽게 목표를 달성하게 만드는 작업입니다. Claude Code에게 UI 개선을 맡길 때는 대상 사용자, 핵심 작업, 정보 우선순위, 기존 디자인 시스템, 접근성, 반응형 조건을 함께 줘야 합니다.

좋은 UI 개선 요청은 취향이 아니라 사용성과 검증 기준을 포함합니다.

## 먼저 확인할 것

```text
아직 UI를 수정하지 마.
먼저 현재 화면 구조와 디자인 패턴을 분석해줘.

확인할 것:
1. 대상 사용자
2. 사용자가 이 화면에서 하는 핵심 작업
3. 현재 정보 구조
4. 반복되는 컴포넌트와 디자인 시스템
5. 접근성 문제
6. 모바일/데스크톱 레이아웃 문제
7. 개선 우선순위
```

기존 디자인 시스템이 있으면 새 스타일을 만들기보다 기존 패턴을 따라야 합니다.

## UI 개선 요청 템플릿

```text
아래 화면의 UI를 개선하고 싶다.

화면:
[화면 이름 또는 경로]

대상 사용자:
[사용자 유형]

핵심 작업:
- [작업 1]
- [작업 2]

문제:
- [현재 사용자가 겪는 문제]

제약:
- 기존 디자인 시스템을 우선 사용한다.
- 새 라이브러리는 추가하지 않는다.
- 접근성과 키보드 사용성을 유지한다.
- 모바일과 데스크톱 모두 확인한다.
- 먼저 개선 계획만 작성한다.
```

## 접근성 기준 포함

W3C WCAG 기준을 그대로 외울 필요는 없지만, UI 요청에는 다음 항목을 포함하세요.

```text
접근성 확인:
- 키보드만으로 주요 작업을 할 수 있는가
- 포커스 표시가 보이는가
- 버튼과 입력에 접근 가능한 이름이 있는가
- 색상만으로 상태를 전달하지 않는가
- 오류 메시지가 텍스트로 설명되는가
- 모바일에서 터치 대상이 너무 작지 않은가
```

이 조건은 “보기 좋은 UI”보다 “사용 가능한 UI”를 만드는 데 중요합니다.

## 반응형 조건

```text
반응형 조건:
- 360px 모바일 폭에서 텍스트가 잘리지 않아야 한다.
- 768px 태블릿에서 주요 CTA가 화면 밖으로 밀리지 않아야 한다.
- 1280px 데스크톱에서 정보 밀도가 지나치게 낮아지지 않아야 한다.
```

구체적인 폭을 주면 Claude Code가 더 정확히 설계할 수 있습니다.

## 구현 승인

계획을 본 뒤 작은 단위로 승인합니다.

```text
좋아. 레이아웃 구조와 상태 표시 개선만 먼저 적용해줘.

제약:
- 색상 팔레트 전체 변경은 하지 마.
- 새 컴포넌트 라이브러리는 추가하지 마.
- 기존 Button, Input, Card 컴포넌트를 사용해줘.
- 완료 후 변경 파일과 확인할 화면 크기를 알려줘.
```

## UI 검증 요청

```text
수정 후 UI를 검토해줘.

확인할 것:
1. 모바일/데스크톱에서 텍스트가 겹치지 않는지
2. 버튼과 입력의 접근 가능한 이름
3. 키보드 포커스 이동
4. 로딩/빈 상태/오류 상태
5. 기존 디자인 시스템과의 일관성
```

프론트엔드 변경은 가능하면 브라우저에서 직접 확인해야 합니다.

## 실습

아래 요청을 더 좋은 UI 개선 프롬프트로 바꿔 보세요.

```text
관리자 대시보드를 더 예쁘게 만들어줘.
```

대상 사용자, 핵심 작업, 정보 우선순위, 접근성, 반응형 조건, 기존 디자인 시스템 사용 조건을 포함하세요.

## Claude Code에 입력할 프롬프트

```text
[화면 이름] UI 개선 계획을 세워줘.

대상 사용자:
[사용자]

핵심 작업:
[사용자가 이 화면에서 끝내야 하는 일]

현재 문제:
[문제 목록]

제약:
- 아직 파일은 수정하지 마.
- 기존 디자인 시스템과 컴포넌트를 먼저 확인해줘.
- 접근성, 키보드 사용성, 모바일/데스크톱 레이아웃을 포함해줘.
- 로딩, 빈 상태, 오류 상태도 검토해줘.

출력:
1. 현재 UI 문제
2. 개선 우선순위
3. 변경할 컴포넌트
4. 접근성 체크
5. 반응형 체크
6. 구현 단계
```

## 체크리스트

- [ ] 대상 사용자와 핵심 작업이 명확하다.
- [ ] 기존 디자인 시스템을 먼저 확인한다.
- [ ] 접근성과 키보드 사용성을 포함한다.
- [ ] 모바일과 데스크톱 조건을 지정한다.
- [ ] 로딩, 빈 상태, 오류 상태를 확인한다.

## 흔한 실수

- “예쁘게”처럼 취향 중심으로 요청한다.
- 기존 컴포넌트 체계를 무시한다.
- 모바일에서 텍스트 겹침을 확인하지 않는다.
- 색상만으로 상태를 표현한다.
