# 062. context window 관리

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code는 긴 대화와 많은 파일을 다룰 수 있지만, 컨텍스트는 무한하지 않습니다. 대화가 길어지고 파일이 많이 들어오면 중요한 정보가 흐려질 수 있습니다.

context window 관리는 “더 많이 넣기”가 아니라 “필요한 정보만 남기기”입니다.

## 컨텍스트를 많이 쓰는 행동

- 전체 코드베이스를 한 번에 요약시키기
- 대용량 로그 전체 붙여넣기
- 관련 없는 파일을 많이 열어 두기
- 긴 대화에서 주제를 계속 바꾸기
- 같은 설명을 반복해서 붙여넣기
- 테스트 실패 로그를 정리 없이 누적하기

## 줄이는 방법

| 방법 | 효과 |
| --- | --- |
| `/compact [instructions]` | 긴 대화를 초점 있는 요약으로 압축 |
| `/clear` | 작업 주제가 바뀔 때 새 출발 |
| `progress.md` | 중요한 상태를 파일로 고정 |
| 범위 제한 | 읽을 파일과 폴더를 줄임 |
| 로그 요약 | 최초 오류와 관련 구간만 사용 |
| docs 분리 | 긴 정책은 필요할 때만 읽음 |

## 좋은 compact 요청

```text
/compact 지금까지의 작업을 다음 기준으로 요약해줘:
1. 목표
2. 변경 파일
3. 중요한 결정
4. 테스트 결과
5. 실패한 명령
6. 남은 TODO
7. 다음에 읽어야 할 파일
```

그냥 `/compact`만 쓰는 것보다 초점을 주는 편이 좋습니다.

## clear를 쓸 때

작업 주제가 크게 바뀌면 `/clear`가 낫습니다.

```text
/clear
```

단, clear 전에 중요한 내용은 `progress.md`, PR 설명, 이슈, 메모리에 남깁니다.

## 파일 읽기 범위 제한

```text
이번 작업은 로그인 버그 조사다.
먼저 auth, session, middleware, 관련 테스트 파일만 확인해줘.
전체 프로젝트 요약은 하지 마.
```

범위 제한은 정확도와 비용을 동시에 개선합니다.

## 체크리스트

- [ ] 긴 세션에서는 `/compact`에 초점 지시를 붙인다.
- [ ] 주제가 바뀌면 `/clear`를 쓴다.
- [ ] 중요한 진행 상태는 파일로 남긴다.
- [ ] 로그와 파일 범위를 줄인다.
- [ ] 전체 프로젝트 분석을 반복하지 않는다.
