# 036. PRD.md 작성법

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

`PRD.md`는 제품 요구사항 문서입니다. Claude Code에게 기능을 구현하게 하기 전에 목표, 사용자, 범위, 비범위, 수용 기준을 정리해 두면 “알아서 만들어줘”라는 위험한 요청을 줄일 수 있습니다.

PRD의 목적은 문서를 예쁘게 만드는 것이 아닙니다. 구현자가 추측해야 하는 부분을 줄이고, 완료 여부를 판단할 기준을 만드는 것입니다.

## 언제 필요한가

다음 작업은 PRD를 먼저 쓰는 편이 좋습니다.

- 사용자에게 보이는 새 기능
- 데이터베이스 스키마 변경
- 결제, 인증, 권한 관련 기능
- 여러 화면이나 API가 함께 바뀌는 작업
- 팀원과 리뷰해야 하는 작업
- 여러 세션에 걸쳐 진행할 작업

반대로 오타 수정이나 단일 테스트 이름 변경처럼 작은 작업에는 PRD가 과할 수 있습니다.

## PRD.md 템플릿

```markdown
# PRD: [기능 이름]

## 1. 배경

- 현재 문제:
- 왜 지금 필요한가:
- 관련 문서/이슈:

## 2. 목표

- 사용자가 무엇을 할 수 있어야 하는가:
- 비즈니스 또는 운영상 기대 효과:

## 3. 대상 사용자

- 주요 사용자:
- 보조 사용자:
- 관리자 또는 운영자 영향:

## 4. 범위

이번 작업에 포함:

- [ ]
- [ ]
- [ ]

이번 작업에 포함하지 않음:

- [ ]
- [ ]

## 5. 사용자 흐름

1. 사용자가 [행동]한다.
2. 시스템은 [응답]한다.
3. 사용자는 [결과]를 확인한다.

## 6. 기능 요구사항

- FR-1:
- FR-2:
- FR-3:

## 7. 비기능 요구사항

- 성능:
- 접근성:
- 보안:
- 로깅/모니터링:

## 8. 수용 기준

- [ ] Given [상황], When [행동], Then [결과]
- [ ] Given [상황], When [행동], Then [결과]

## 9. 검증 계획

- 단위 테스트:
- 통합 테스트:
- 수동 확인:

## 10. 리스크와 열린 질문

- 리스크:
- 열린 질문:
```

## Claude Code와 함께 쓰는 흐름

PRD를 먼저 만들고 바로 구현하지 마세요. 먼저 검토를 요청합니다.

```text
PRD.md를 읽고 구현 전에 보완해야 할 모호한 요구사항을 찾아줘.

출력:
1. 모호한 부분
2. 빠진 수용 기준
3. 구현 위험
4. 테스트 전략
5. 구현을 시작해도 되는 최소 조건

아직 파일은 수정하지 마.
```

PRD가 충분해진 뒤 구현 계획을 요청합니다.

```text
이 PRD를 기준으로 구현 계획을 작성해줘.
관련 파일 후보, 단계별 작업, 테스트 계획, 위험 요소를 포함해줘.
아직 구현하지 마.
```

## 좋은 수용 기준

나쁜 수용 기준:

```markdown
- 사용자가 편하게 검색할 수 있다.
```

좋은 수용 기준:

```markdown
- [ ] Given 로그인한 사용자가 고객 목록 화면에 있을 때, When 검색어를 입력하면, Then 이름 또는 이메일에 검색어가 포함된 고객만 표시된다.
- [ ] Given 검색 결과가 없을 때, When 검색이 완료되면, Then 빈 상태 메시지와 검색어 초기화 버튼을 보여준다.
```

수용 기준은 테스트로 바꿀 수 있을 만큼 구체적이어야 합니다.

## 비범위가 중요한 이유

PRD에서 “하지 않을 일”은 “할 일”만큼 중요합니다.

```markdown
## 비범위

- 이번 작업에서는 검색 결과 정렬 기능을 추가하지 않는다.
- 서버 API 페이지네이션 방식은 변경하지 않는다.
- 모바일 전용 UI 개편은 별도 작업으로 분리한다.
```

비범위가 없으면 Claude Code가 기능을 넓게 해석할 수 있습니다.

## 실습

작은 기능 하나를 골라 PRD를 작성해 보세요.

예시:

- 댓글 삭제 기능
- 관리자용 사용자 검색
- CSV 내보내기
- 프로필 이미지 변경

작성 후 Claude Code에게 모호한 요구사항과 빠진 수용 기준을 찾게 합니다.

## Claude Code에 입력할 프롬프트

```text
다음 기능을 위한 PRD.md 초안을 작성해줘.

기능:
[기능 이름]

현재 문제:
[문제 설명]

대상 사용자:
[사용자]

제약:
- 아직 구현하지 마.
- 모르는 부분은 추정하지 말고 열린 질문으로 남겨줘.
- 수용 기준은 Given/When/Then 형식으로 작성해줘.
- 범위와 비범위를 반드시 분리해줘.

출력:
PRD.md에 바로 넣을 수 있는 Markdown 문서
```

## 체크리스트

- [ ] 목표와 배경이 명확하다.
- [ ] 범위와 비범위가 분리되어 있다.
- [ ] 수용 기준이 테스트 가능한 문장이다.
- [ ] 보안, 성능, 접근성 같은 비기능 요구사항을 확인했다.
- [ ] 열린 질문을 숨기지 않고 남겼다.

## 흔한 실수

- PRD를 기능 목록만 적은 문서로 만든다.
- 비범위를 적지 않는다.
- 수용 기준이 “잘 동작한다” 수준에 머문다.
- 모르는 요구사항을 Claude가 추정하게 둔다.
