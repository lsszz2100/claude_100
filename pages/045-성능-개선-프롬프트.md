# 045. 성능 개선 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

성능 개선은 추측으로 시작하면 위험합니다. Claude Code에게 “빠르게 해줘”라고 요청하기보다 현재 증상, 측정 기준, 재현 조건, 병목 후보, 검증 방법을 정해야 합니다.

웹 프론트엔드에서는 Core Web Vitals처럼 사용자 경험과 연결된 지표를 사용할 수 있습니다. 백엔드에서는 응답 시간, 처리량, DB 쿼리 수, 메모리, CPU, 오류율 같은 지표를 봅니다.

## 먼저 측정하기

성능 요청의 첫 문장은 보통 이렇게 시작하는 것이 좋습니다.

```text
아직 최적화하지 마.
먼저 성능 문제를 측정하고 병목 후보를 찾아줘.
```

측정 없는 최적화는 코드 복잡도만 늘릴 수 있습니다.

## 성능 요청 템플릿

```text
성능 문제를 조사해줘.

증상:
[느린 화면/API/작업]

재현 조건:
[데이터 크기, 사용자 행동, 환경]

현재 지표:
- 응답 시간:
- LCP/INP/CLS:
- 쿼리 수:
- 메모리/CPU:

목표:
[예: p95 응답 800ms 이하, LCP 2.5초 이하]

제약:
- 아직 코드는 수정하지 마.
- 먼저 측정 방법과 병목 후보를 제안해줘.
- 변경 전후 비교 방법을 포함해줘.
```

## 프론트엔드 성능 기준

웹 화면에서는 다음 질문을 던질 수 있습니다.

```text
이 화면의 프론트엔드 성능 병목을 조사해줘.

확인할 것:
- 초기 번들 크기
- 불필요한 재렌더링
- 이미지 크기와 lazy loading
- LCP 후보 요소
- 사용자 입력 후 INP에 영향을 줄 수 있는 긴 작업
- CLS를 유발하는 레이아웃 이동

아직 파일은 수정하지 마.
```

Core Web Vitals 기준으로는 LCP는 로딩, INP는 상호작용, CLS는 시각적 안정성을 봅니다.

## 백엔드 성능 기준

API나 서버 코드에서는 다음을 봅니다.

```text
이 API의 성능 병목을 조사해줘.

확인할 것:
- N+1 쿼리 가능성
- pagination 누락
- 불필요한 전체 테이블 조회
- 반복문 안의 외부 API 호출
- 캐싱 가능성
- 동시성 문제
- timeout과 retry 설정
```

특히 데이터 크기가 커질 때 느려지는지 확인해야 합니다.

## 최적화 승인

조사 후 수정은 작은 단위로 승인하세요.

```text
가장 영향이 크고 위험이 낮은 개선 1개만 먼저 적용해줘.

제약:
- 동작은 바꾸지 마.
- 새 의존성은 추가하지 마.
- 변경 전후 비교 가능한 측정 방법을 남겨줘.
- 관련 테스트를 실행해줘.
```

성능 개선은 기능 변경과 섞이면 검증하기 어렵습니다.

## 결과 보고 형식

```text
완료 후 다음 형식으로 보고해줘.

1. 측정한 기준
2. 발견한 병목
3. 적용한 변경
4. 변경 전/후 예상 또는 실제 수치
5. 실행한 테스트
6. 남은 성능 리스크
```

수치가 없으면 “개선했다”는 말의 의미가 약합니다.

## 실습

아래 증상을 성능 조사 프롬프트로 바꿔 보세요.

```text
고객 목록 화면이 느리다.
```

데이터 크기, 측정 지표, 프론트엔드/백엔드 후보, 수정 전 조사 조건을 포함하세요.

## Claude Code에 입력할 프롬프트

```text
성능 문제를 조사해줘.

대상:
[화면/API/작업]

증상:
[느린 상황]

재현 조건:
[데이터 크기, 입력, 환경]

목표 지표:
[응답 시간, Core Web Vitals, 쿼리 수 등]

제약:
- 아직 최적화하지 마.
- 먼저 측정 방법과 병목 후보를 정리해줘.
- 가장 영향이 큰 개선 3개를 위험도와 함께 제안해줘.
- 변경 전후 검증 방법을 포함해줘.
```

## 체크리스트

- [ ] 성능 목표 지표가 있다.
- [ ] 수정 전 측정 방법을 정한다.
- [ ] 프론트엔드와 백엔드 병목을 구분한다.
- [ ] 가장 작은 개선부터 적용한다.
- [ ] 변경 전후 비교를 남긴다.

## 흔한 실수

- 측정 없이 코드부터 바꾼다.
- 사용자 체감 지표 없이 내부 구현만 최적화한다.
- 캐싱으로 데이터 정합성 문제를 만든다.
- 성능 개선과 대규모 리팩터링을 섞는다.
