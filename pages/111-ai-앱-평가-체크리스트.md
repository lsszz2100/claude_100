# 111. AI 앱 평가 체크리스트

저자: AI_Innovation_Studio

AI 앱은 동작한다고 끝이 아닙니다. 정확성, 안정성, 비용, 지연 시간, 보안, 사용자 만족도를 지속적으로 평가해야 운영 가능한 제품이 됩니다.

## 평가 축

| 평가 축 | 질문 |
| --- | --- |
| 정확성 | 사용자의 질문에 맞는 답을 하는가? |
| 근거성 | 출처나 입력 자료에 기반해 답하는가? |
| 안정성 | 같은 유형의 요청에서 품질이 일정한가? |
| 안전성 | 민감 정보나 금지된 행동을 피하는가? |
| 비용 | 요청당 비용이 예측 가능한가? |
| 지연 | 사용자가 기다릴 수 있는 시간 안에 응답하는가? |
| 회복 | 실패했을 때 재시도나 대체 경로가 있는가? |

## 평가셋 만들기

평가셋은 실제 사용자의 대표 질문으로 구성해야 합니다.

```text
case_id:
input:
expected_behavior:
must_include:
must_not_include:
risk_level:
```

평가셋은 한 번 만들고 끝내는 파일이 아닙니다. 운영 중 들어온 실패 사례, 사용자가 직접 고친 응답, 고객 문의를 주기적으로 반영해야 합니다. 특히 비용을 줄이거나 모델을 바꾸거나 프롬프트를 수정할 때는 같은 평가셋으로 전후 품질을 비교해야 합니다.

## 점수표 예시

| 항목 | 0점 | 1점 | 2점 |
| --- | --- | --- | --- |
| 요구 충족 | 질문을 벗어남 | 일부만 답함 | 필요한 내용을 답함 |
| 근거 사용 | 근거 없음 | 근거가 약함 | 입력 자료에 기반함 |
| 형식 준수 | 형식 불일치 | 일부 누락 | 지정 형식 준수 |
| 안전성 | 위험한 출력 | 주의 문구 부족 | 민감 정보와 위험 행동 회피 |
| 실행 가능성 | 바로 쓸 수 없음 | 수정 필요 | 바로 검토 가능한 수준 |

점수표는 복잡할 필요가 없습니다. 중요한 것은 팀원이 같은 기준으로 평가할 수 있어야 한다는 점입니다.

## 평가 운영 주기

평가셋은 제품 변경과 함께 운영해야 합니다.

| 시점 | 확인할 것 |
| --- | --- |
| 프롬프트 변경 | 기존 평가셋 점수 유지 여부 |
| 모델 변경 | 품질, 비용, 지연 시간 비교 |
| 검색 설정 변경 | 검색 결과와 답변 품질 분리 평가 |
| 실패 사례 발생 | 같은 유형의 평가 케이스 추가 |
| 릴리스 전 | 고위험 케이스 재실행 |

점수가 좋아져도 특정 고위험 케이스가 실패하면 배포를 멈출 수 있어야 합니다.

## Claude에게 줄 요청

```text
다음 AI 기능을 평가하기 위한 체크리스트와 평가셋 초안을 만들어줘.

기능 설명:
[설명]

요청:
1. 정상 케이스 5개
2. 경계 케이스 5개
3. 실패해야 하는 케이스 5개
4. 평가 기준과 점수표
5. 운영 중 모니터링해야 할 지표
```

## 운영 지표

- 요청 수
- 성공률과 실패율
- 평균 지연 시간
- 요청당 비용
- 사용자 재시도율
- 사람이 수정한 응답 비율
- 민감 정보 차단 횟수

## 체크리스트

- [ ] 평가셋이 실제 사용 사례를 반영한다.
- [ ] 실패해야 하는 케이스가 포함되어 있다.
- [ ] 비용과 지연 시간도 평가한다.
- [ ] 평가 결과를 다음 개선 작업으로 연결한다.
- [ ] 모델, 프롬프트, 검색 설정을 바꿀 때 같은 평가셋으로 비교한다.
- [ ] 사람이 수정한 응답을 다음 평가 후보로 모은다.
- [ ] 고위험 케이스는 릴리스 전 반드시 재평가한다.
