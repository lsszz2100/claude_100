# 018. Claude에게 검토시키는 법

난이도: 초급  
기준일: 2026년 05월 03일

## 핵심 개념

검토는 칭찬을 듣는 과정이 아닙니다. 검토는 결함, 위험, 빠진 조건, 테스트 공백을 찾는 과정입니다. Claude에게 “어때?”라고 물으면 부드러운 피드백이 나올 수 있습니다. 실무 검토를 원한다면 검토 기준과 출력 형식을 명확히 해야 합니다.

## 나쁜 검토 요청

```text
이 코드 괜찮아?
```

이 요청은 너무 모호합니다. Claude는 일반적인 인상평을 줄 수 있습니다.

## 좋은 검토 요청

```text
이 변경사항을 코드 리뷰 관점으로 검토해줘.

중점:
1. 실제 버그 가능성
2. 보안 위험
3. 회귀 가능성
4. 빠진 테스트
5. 유지보수 문제

출력 형식:
- 심각도
- 위치
- 문제
- 영향
- 수정 제안
- 테스트 필요 여부

칭찬보다 수정이 필요한 문제를 우선해줘.
```

## 검토 관점의 종류

### 코드 품질 검토

가독성, 중복, 복잡도, 책임 분리, 네이밍을 봅니다.

### 버그 검토

경계 조건, null, 빈 입력, 예외 흐름, 상태 불일치를 봅니다.

### 보안 검토

입력 검증, 인증, 권한, 비밀정보, 주입 공격, 의존성 위험을 봅니다.

### 테스트 검토

핵심 흐름, 실패 흐름, 회귀 테스트, 통합 테스트 필요성을 봅니다.

### 문서 검토

사용자가 따라 할 수 있는지, 명령이 최신인지, 빠진 전제가 없는지 봅니다.

## Claude Code에서 활용하기

Claude Code에서는 Git diff를 기준으로 검토시키는 방식이 유용합니다.

```text
현재 git diff를 기준으로 코드 리뷰를 해줘.
실제 수정은 하지 말고 리뷰 의견만 작성해줘.
심각도 높은 문제부터 정렬해줘.
```

이 요청은 PR 리뷰 전에 매우 유용합니다.

## 실습

작은 변경을 만든 뒤 다음 요청을 사용해 보세요.

```text
현재 변경사항을 리뷰해줘.
다음 기준으로만 봐줘.

1. 버그 가능성
2. 테스트 누락
3. 보안 위험
4. 불필요하게 넓은 변경

문제가 없으면 "중대한 문제 없음"이라고 말하고, 남은 리스크를 적어줘.
```

## Claude에게 입력할 프롬프트

```text
너는 엄격한 코드 리뷰어다.
아래 변경사항을 검토해줘.

검토 기준:
- 실제 버그 가능성
- 보안 위험
- 회귀 가능성
- 빠진 테스트
- 유지보수 문제

출력 형식:
1. 발견 사항
2. 심각도
3. 근거
4. 수정 제안
5. 추가 테스트

칭찬은 생략하고 문제와 리스크를 우선해줘.
```

## 체크리스트

- [ ] 검토 기준을 명확히 했다.
- [ ] 출력 형식을 지정했다.
- [ ] 심각도 순 정렬을 요청했다.
- [ ] 테스트 공백을 포함했다.
- [ ] 문제가 없을 때도 남은 리스크를 요청했다.

## 흔한 실수

- “괜찮아?”처럼 인상평을 묻는다.
- 리뷰 기준 없이 전체 검토를 맡긴다.
- Claude의 리뷰 결과를 다시 검토하지 않는다.
- 보안과 테스트 관점을 빼먹는다.
