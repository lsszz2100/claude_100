# 107. 추천 PRD.md 템플릿

PRD는 제품 요구사항을 구현 가능한 단위로 정리하는 문서입니다. Claude Code에게 구현을 맡기기 전에 PRD를 만들면 목표, 범위, 비범위, 수용 기준을 명확히 할 수 있습니다.

초보자가 PRD에서 가장 자주 놓치는 부분은 “하지 않을 일”과 “완료 판단 기준”입니다. 기능 설명만 길게 쓰면 Claude Code는 구현 범위를 넓게 해석할 수 있습니다. PRD는 아이디어 문서가 아니라 작업 범위를 조절하는 문서라는 점을 기억하세요.

## 템플릿

```markdown
# PRD: 기능 이름

## 배경

왜 이 기능이 필요한가?

## 목표

- 목표 1
- 목표 2

## 비목표

- 이번 작업에서 하지 않을 것

## 사용자

- 주요 사용자:
- 사용 상황:

## 요구사항

1. 사용자는 ...
2. 시스템은 ...
3. 관리자는 ...

## 수용 기준

- Given ...
  When ...
  Then ...

## 영향 범위

- UI:
- API:
- DB:
- 테스트:
- 문서:

## 위험과 대응

- 위험:
- 대응:
```

## Claude에게 줄 요청

```text
다음 기능 설명을 PRD.md 초안으로 바꿔줘.

기능 설명:
[여기에 작성]

요청:
1. 목표와 비목표를 분리해줘.
2. 모호한 요구사항은 질문 목록으로 빼줘.
3. 수용 기준은 Given/When/Then 형식으로 작성해줘.
4. 구현 범위와 테스트 범위를 나눠줘.
```

## 작성 순서

1. 사용자가 겪는 문제를 한 문장으로 씁니다.
2. 이번 작업에서 바뀌어야 하는 화면, API, 데이터, 문서를 나눠 적습니다.
3. 하지 않을 일을 비목표에 넣습니다. 이 항목이 있어야 범위가 커지는 것을 막을 수 있습니다.
4. 수용 기준은 사람이 읽는 설명이 아니라 테스트 가능한 조건으로 씁니다.
5. 모르는 내용은 추정하지 말고 질문으로 남깁니다.

## 나쁜 예와 고친 예

나쁜 예:

```text
로그인을 더 좋게 만든다.
```

고친 예:

```text
사용자가 이메일과 비밀번호를 잘못 입력했을 때, 어떤 항목이 잘못됐는지 알 수 있는 오류 메시지를 보여준다.
비밀번호 재설정 기능은 이번 범위에서 제외한다.
```

## 체크리스트

- [ ] 왜 필요한 기능인지 설명된다.
- [ ] 이번에 하지 않을 일이 명확하다.
- [ ] 수용 기준이 테스트 가능한 문장이다.
- [ ] 구현 전 질문이 분리되어 있다.
