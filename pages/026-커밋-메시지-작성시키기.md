# 026. 커밋 메시지 작성시키기

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

커밋 메시지는 “무엇을 바꿨는지”보다 “왜 바꿨는지”를 남기는 기록입니다. Claude Code는 diff를 읽고 커밋 메시지 초안을 만들 수 있지만, 사용자가 변경 의도와 범위를 검토해야 합니다.

좋은 커밋 메시지는 나중에 문제를 추적할 때 도움이 됩니다. 반대로 “fix”, “update”, “작업함” 같은 메시지는 변경 이력을 거의 쓸모없게 만듭니다.

## 커밋 전 확인

커밋 메시지를 작성하기 전에 먼저 세 가지를 확인하세요.

1. 변경 파일 목록이 의도한 범위인지
2. 테스트 또는 검토를 했는지
3. 변경이 하나의 주제로 묶이는지

여러 주제가 섞여 있다면 커밋을 나누는 편이 좋습니다. 예를 들어 버그 수정, 문서 수정, 포맷 변경이 한 커밋에 섞이면 리뷰와 되돌리기가 어려워집니다.

## 기본 구조

실무에서는 다음 구조가 무난합니다.

```text
type(scope): summary

body
```

예시:

```text
fix(auth): handle expired session redirect

Redirect users to the login page when the session refresh fails.
This prevents the profile page from rendering with stale user state.
```

초급자는 영어 커밋 메시지가 부담스럽다면 한국어로 써도 됩니다. 중요한 것은 일관성과 의미입니다.

```text
fix(auth): 만료된 세션 리다이렉트 처리

세션 갱신 실패 시 프로필 화면에 오래된 사용자 정보가 남지 않도록
로그인 화면으로 이동하게 수정했다.
```

## 자주 쓰는 type

| type | 의미 |
| --- | --- |
| `feat` | 사용자에게 보이는 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `test` | 테스트 추가 또는 수정 |
| `refactor` | 동작 변경 없는 구조 개선 |
| `style` | 포맷, 세미콜론 등 의미 없는 스타일 변경 |
| `chore` | 빌드, 설정, 관리 작업 |

팀 규칙이 있다면 그 규칙을 우선합니다. 없다면 위 정도만 써도 충분합니다.

## Claude Code에게 요청하기

커밋 메시지를 만들 때는 diff를 근거로 삼게 하세요.

```text
git diff를 기준으로 커밋 메시지 후보를 작성해줘.

조건:
- 변경 의도를 반영해줘.
- Conventional Commit 형식 후보 3개를 제안해줘.
- 각 후보가 적합한 이유를 한 줄로 설명해줘.
- 아직 커밋은 하지 마.
```

커밋이 여러 개로 나뉘어야 하는지도 물어보면 좋습니다.

```text
현재 변경사항이 하나의 커밋으로 적절한지 검토해줘.
분리하는 편이 좋다면 커밋 단위와 각 커밋 메시지를 제안해줘.
```

## 좋은 커밋 메시지의 기준

좋은 메시지는 다음 질문에 답합니다.

- 어떤 영역을 바꿨나?
- 사용자 또는 개발자에게 어떤 의미가 있나?
- 왜 이 변경이 필요한가?
- 나중에 되돌릴 때 어떤 단위로 이해할 수 있나?

나쁜 메시지는 단순히 작업 행위를 적습니다.

```text
수정
업데이트
코드 정리
버그
```

이런 메시지는 나중에 `git log`를 봐도 도움이 되지 않습니다.

## 본문이 필요한 경우

작은 오타 수정은 한 줄이면 충분합니다.

```text
docs(readme): fix setup command typo
```

하지만 다음 상황에서는 본문을 쓰는 편이 좋습니다.

- 버그 원인이 복잡하다.
- 동작 변경이 있다.
- 마이그레이션이나 설정 변경이 있다.
- 테스트 범위를 설명해야 한다.
- 의도적으로 하지 않은 일이 있다.

본문에는 “무엇을 하지 않았는지”도 남길 수 있습니다.

```text
인증 흐름 자체는 바꾸지 않고, 만료된 토큰 처리 분기만 추가했다.
```

## 실습

작은 변경사항을 만든 뒤 Claude Code에게 다음을 요청해 보세요.

1. 변경사항이 하나의 커밋으로 적절한지 판단한다.
2. 커밋 메시지 후보 3개를 만든다.
3. 가장 적합한 후보를 고른 이유를 설명한다.
4. 본문이 필요한지 판단한다.
5. 커밋 전에 빠진 검증이 있는지 확인한다.

## Claude Code에 입력할 프롬프트

```text
현재 git diff를 기준으로 커밋 메시지를 제안해줘.

요청:
1. 변경사항이 하나의 커밋으로 적절한지 판단해줘.
2. 아니라면 커밋을 어떻게 나눌지 제안해줘.
3. Conventional Commit 형식의 메시지 후보 3개를 작성해줘.
4. 각 후보의 장단점을 설명해줘.
5. 커밋 전에 확인해야 할 테스트나 위험을 알려줘.

제약:
- 아직 커밋하지 마.
- diff에 없는 내용을 메시지에 넣지 마.
- 사용자가 만든 변경과 AI가 만든 변경이 섞였을 가능성을 고려해줘.
```

## 체크리스트

- [ ] 커밋 전 `git status`와 `git diff`를 확인한다.
- [ ] 변경사항이 하나의 주제로 묶이는지 확인한다.
- [ ] 메시지에 변경 의도와 범위가 드러난다.
- [ ] diff에 없는 내용을 과장해서 쓰지 않는다.
- [ ] 필요한 경우 본문에 테스트 결과나 주의사항을 남긴다.

## 흔한 실수

- 커밋 메시지를 작성하기 전에 변경사항을 검토하지 않는다.
- 여러 주제를 한 커밋에 섞는다.
- “수정”, “업데이트”처럼 의미 없는 제목을 쓴다.
- 테스트하지 않은 내용을 커밋 메시지에 통과한 것처럼 쓴다.
