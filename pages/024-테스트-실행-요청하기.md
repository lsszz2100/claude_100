# 024. 테스트 실행 요청하기

난이도: 초급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code로 코드를 수정했다면 테스트 실행까지 요청해야 합니다. AI가 만든 변경은 그럴듯해 보여도 실제 빌드, 타입 검사, 단위 테스트, 통합 테스트에서 실패할 수 있습니다.

테스트 실행 요청의 핵심은 “무슨 테스트를 왜 실행하는지”를 Claude Code가 찾고 설명하게 만드는 것입니다.

## 테스트 명령 찾기

테스트 명령을 모를 때는 바로 실행을 요구하지 말고 먼저 찾게 하세요.

```text
이 프로젝트의 테스트 명령을 찾아줘.
README, package 설정, 빌드 설정 파일을 확인하고
실행 가능한 테스트 명령 후보와 근거 파일을 정리해줘.
아직 테스트는 실행하지 마.
```

그 다음 적절한 명령을 선택해 실행하게 합니다.

```text
가장 작은 관련 테스트부터 실행해줘.
전체 테스트가 필요하다면 이유를 먼저 설명해줘.
```

## 테스트의 종류

프로젝트마다 이름은 다르지만 초급자는 다음 구분만 알아도 충분합니다.

| 종류 | 목적 | 예시 |
| --- | --- | --- |
| 포맷 | 코드 스타일 확인 | `npm run format:check` |
| 린트 | 잠재 오류와 규칙 확인 | `npm run lint` |
| 타입 검사 | 타입 오류 확인 | `npm run typecheck` |
| 단위 테스트 | 작은 함수나 컴포넌트 검증 | `npm test` |
| 빌드 | 앱이 실제로 묶이는지 확인 | `npm run build` |
| 통합/E2E | 사용자 흐름 검증 | `npm run e2e` |

모든 작업에 모든 테스트가 필요한 것은 아닙니다. README 오타 수정에 E2E 테스트를 돌릴 필요는 없습니다. 반대로 인증 로직을 고쳤다면 단위 테스트만으로 부족할 수 있습니다.

## 실패 로그 읽기

테스트가 실패하면 Claude Code에게 “전체 로그를 다시 설명”시키기보다 최초 원인을 찾게 하세요.

```text
테스트 실패 로그에서 가장 먼저 고쳐야 할 원인을 찾아줘.
파생 오류와 최초 오류를 구분해줘.
아직 파일은 수정하지 말고 수정 계획만 제안해줘.
```

긴 로그에서는 마지막 줄만 보면 안 됩니다. 많은 테스트 도구는 중간에 실제 원인을 출력하고 마지막에는 요약만 보여줍니다.

## 작은 테스트부터 실행하기

버그를 하나 고쳤다면 가장 관련 있는 테스트부터 실행하는 편이 효율적입니다.

```text
방금 수정한 파일과 가장 관련 있는 테스트만 먼저 실행해줘.
그 테스트가 통과하면 다음으로 필요한 검증 명령을 제안해줘.
```

이 흐름은 빠른 피드백을 줍니다. 작은 테스트가 통과한 뒤 전체 테스트나 빌드를 실행할지 판단하면 됩니다.

## 테스트를 실행할 수 없을 때

환경 문제로 테스트를 실행할 수 없는 경우도 있습니다. 이때는 실패를 숨기지 말고 기록해야 합니다.

```text
테스트를 실행할 수 없다면 이유를 정확히 적어줘.
그리고 내가 로컬에서 직접 확인할 명령과 기대 결과를 알려줘.
```

테스트 미실행은 실패가 아닙니다. 문제는 테스트를 실행하지 못했다는 사실을 모른 채 넘어가는 것입니다.

## 실습

현재 프로젝트에서 다음을 연습해 보세요.

1. 테스트 명령 후보를 찾게 한다.
2. 각 명령의 근거 파일을 확인한다.
3. 가장 작은 관련 테스트를 실행하게 한다.
4. 실패하면 최초 원인과 파생 오류를 구분하게 한다.
5. 마지막에 실행한 명령과 결과를 요약하게 한다.

## Claude Code에 입력할 프롬프트

```text
이 프로젝트의 테스트 실행 방법을 확인해줘.

단계:
1. README와 설정 파일에서 테스트 명령 후보를 찾아줘.
2. 각 명령이 무엇을 검증하는지 설명해줘.
3. 현재 변경사항과 가장 관련 있는 최소 테스트를 추천해줘.
4. 내가 승인하면 그 명령을 실행해줘.

제약:
- 근거 파일 없이 명령을 추정하지 마.
- 실패하면 최초 원인과 파생 오류를 구분해줘.
- 실행한 명령과 결과를 마지막에 요약해줘.
```

## 체크리스트

- [ ] 테스트 명령의 근거 파일을 확인한다.
- [ ] 가장 작은 관련 테스트부터 실행한다.
- [ ] 실패 로그에서 최초 원인을 찾는다.
- [ ] 테스트를 실행하지 못한 이유를 기록한다.
- [ ] 수정 후 실행한 명령과 결과를 요약한다.

## 흔한 실수

- 테스트 명령을 추측해서 실행한다.
- 실패 로그의 마지막 요약만 보고 원인을 판단한다.
- 작은 수정에도 무조건 전체 테스트만 실행한다.
- 테스트를 실행하지 못했는데 통과한 것처럼 말한다.
