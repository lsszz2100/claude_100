# 048. 데이터베이스 변경 요청법

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

데이터베이스 변경은 코드 변경보다 되돌리기 어렵습니다. Claude Code에게 DB 변경을 맡길 때는 스키마, 기존 데이터, 마이그레이션, 롤백, 백업, 검증 쿼리를 함께 요청해야 합니다.

가장 위험한 요청은 “컬럼 하나 추가해줘”처럼 쉬워 보이는 요청입니다. 컬럼 하나도 기본값, null 허용, 인덱스, 기존 데이터, 배포 순서에 영향을 줄 수 있습니다.

## DB 변경 전 확인할 것

```text
아직 DB 파일이나 마이그레이션은 수정하지 마.
먼저 다음을 조사해줘.

1. 현재 ORM 또는 migration 도구
2. 스키마 파일 위치
3. 기존 migration 폴더
4. 관련 모델과 테이블
5. 기존 데이터에 미치는 영향
6. 롤백 가능성
7. 필요한 테스트
```

프로젝트마다 Prisma, Django, Rails, Alembic, Flyway, Liquibase 등 도구가 다르므로 먼저 현재 도구를 확인해야 합니다.

## 안전한 요청 템플릿

```text
데이터베이스 변경을 준비해줘.

변경 목표:
[예: users 테이블에 display_name 컬럼 추가]

요구사항:
- [요구사항]

제약:
- 아직 migration 파일은 만들지 마.
- 기존 데이터 손실이 없어야 한다.
- 배포 중 서비스 중단을 최소화해야 한다.
- 롤백 또는 복구 전략을 포함해줘.

출력:
1. 현재 스키마 구조
2. 변경 설계
3. migration 전략
4. 기존 데이터 처리
5. 애플리케이션 코드 변경 범위
6. 검증 쿼리
7. 롤백 계획
8. 위험 요소
```

## 추가 컬럼 예시

```text
users 테이블에 `display_name` 컬럼을 추가하고 싶다.

조건:
- 기존 사용자는 name 값을 display_name 초기값으로 사용한다.
- 새 사용자는 display_name을 필수로 입력한다.
- 기존 API 응답은 당장 깨지면 안 된다.

먼저 안전한 migration 계획만 작성해줘.
```

이 경우 바로 `NOT NULL` 컬럼을 추가하면 기존 데이터 때문에 실패할 수 있습니다. 먼저 nullable 컬럼 추가, 데이터 backfill, 애플리케이션 반영, 이후 제약 강화 순서가 필요할 수 있습니다.

## 검증 쿼리

DB 변경 요청에는 검증 쿼리를 포함하세요.

```sql
-- display_name이 비어 있는 기존 사용자 확인
SELECT COUNT(*) AS missing_display_name
FROM users
WHERE display_name IS NULL OR display_name = '';
```

검증 쿼리는 마이그레이션 전과 후 모두 필요할 수 있습니다.

## 위험한 변경

다음 작업은 특히 조심해야 합니다.

- 컬럼 삭제
- 컬럼 이름 변경
- 타입 변경
- unique 제약 추가
- NOT NULL 제약 추가
- 대형 테이블 인덱스 추가
- 데이터 backfill
- 외래 키 추가

이런 변경은 작은 트래픽 환경에서는 쉬워 보여도 운영 DB에서는 잠금, 지연, 데이터 손실을 만들 수 있습니다.

## 실습

다음 변경을 안전한 요청으로 바꿔 보세요.

```text
orders 테이블에 status 컬럼을 추가하고 기존 주문 상태를 채워 넣고 싶다.
```

기존 데이터 처리, 기본값, 검증 쿼리, 롤백 계획을 포함하세요.

## Claude Code에 입력할 프롬프트

```text
다음 데이터베이스 변경을 설계해줘.

변경:
[스키마 변경 설명]

제약:
- 아직 파일은 수정하지 마.
- migration 도구와 기존 migration 패턴을 먼저 확인해줘.
- 데이터 손실 가능성을 분석해줘.
- 기존 데이터 backfill 전략을 제안해줘.
- 롤백 또는 복구 계획을 포함해줘.

출력:
1. 영향받는 테이블/모델
2. migration 단계
3. 애플리케이션 코드 변경
4. 검증 쿼리
5. 테스트 계획
6. 위험 요소
```

## 체크리스트

- [ ] migration 도구와 기존 패턴을 확인했다.
- [ ] 기존 데이터 처리 계획이 있다.
- [ ] 데이터 손실 가능성을 검토했다.
- [ ] 검증 쿼리와 테스트 계획이 있다.
- [ ] 롤백 또는 복구 방법을 적었다.

## 흔한 실수

- 운영 DB 크기와 잠금 영향을 고려하지 않는다.
- 기존 데이터가 있는데 바로 NOT NULL을 추가한다.
- 컬럼 rename을 안전한 단계로 나누지 않는다.
- 롤백 계획 없이 migration을 만든다.
