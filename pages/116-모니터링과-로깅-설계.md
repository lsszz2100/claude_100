# 116. 모니터링과 로깅 설계

AI 앱 운영에서는 일반 서버 지표뿐 아니라 입력, 출력, 비용, 지연 시간, 거절, 재시도, 사용자 수정 비율을 함께 봐야 합니다. 단, 로그에 민감 정보를 그대로 남기면 안 됩니다.

## 관찰할 지표

| 지표 | 이유 |
| --- | --- |
| 요청 수 | 사용량과 비용 예측 |
| 성공률 | 시스템 안정성 |
| 지연 시간 | 사용자 경험 |
| 요청당 비용 | 운영 비용 관리 |
| 오류 유형 | 장애 원인 분류 |
| 재시도율 | 품질 또는 UX 문제 탐지 |
| 차단/거절 횟수 | 안전 정책 점검 |

## 로그에 남길 것

- 요청 ID
- 사용자 또는 조직 ID의 비식별 값
- 기능 이름
- 모델 또는 처리 경로
- 토큰/비용 요약
- 지연 시간
- 오류 코드
- 평가 점수 또는 사용자 피드백

## 로그에 남기지 말 것

- API 키와 토큰
- 비밀번호
- 원본 개인정보
- 내부 인증 쿠키
- 민감한 프롬프트 전문
- 고객 데이터 전문

## 마스킹 기준

로그 마스킹은 나중에 붙이는 장식이 아니라 설계 단계에서 정해야 합니다. 운영자가 장애를 조사할 수 있을 만큼의 단서는 남기되, 원문을 그대로 복원할 수 없게 해야 합니다.

| 데이터 | 권장 처리 |
| --- | --- |
| 사용자 ID | 해시 또는 내부 식별자 |
| 이메일 | 도메인만 남기거나 마스킹 |
| 긴 입력 문장 | 길이, 언어, 기능 구분만 저장 |
| 파일명 | 민감 경로 제거 후 확장자와 유형만 저장 |
| 오류 | 내부 stack trace는 제한된 저장소에만 보관 |

## 알림 조건 예시

- 오류율이 평소보다 급격히 증가한다.
- 요청당 비용이 기준값을 넘는다.
- 지연 시간이 사용자 경험 기준을 넘는다.
- 차단 또는 거절 비율이 갑자기 바뀐다.
- 특정 기능에서 재시도율이 높아진다.

알림은 많을수록 좋은 것이 아닙니다. 바로 행동할 수 있는 기준만 남기고, 단순 참고 지표는 대시보드로 분리하세요.

## Claude에게 줄 요청

```text
다음 AI 기능의 모니터링과 로깅 설계를 작성해줘.

기능:
[설명]

요청:
1. 운영 지표
2. 로그 필드
3. 민감 정보 마스킹 기준
4. 알림 조건
5. 장애 조사 시 확인 순서
```

## 체크리스트

- [ ] 비용과 지연 시간이 관찰된다.
- [ ] 민감 정보가 로그에 남지 않는다.
- [ ] 장애 알림 기준이 있다.
- [ ] 사용자 피드백이 개선 작업으로 연결된다.
- [ ] 운영자가 장애를 재현할 최소 단서가 남는다.
- [ ] 로그 보관 기간과 접근 권한이 정해져 있다.
