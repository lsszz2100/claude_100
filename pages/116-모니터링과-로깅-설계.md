# 116. 모니터링과 로깅 설계

AI 앱 운영에서는 일반 서버 지표뿐 아니라 입력, 출력, 비용, 지연 시간, 거절, 재시도, 사용자 수정 비율을 함께 봐야 합니다. 단, 로그에 민감 정보를 그대로 남기면 안 됩니다.

## 관찰할 지표

| 지표 | 이유 |
| --- | --- |
| 요청 수 | 사용량과 비용 예측 |
| 성공률 | 시스템 안정성 |
| 지연 시간 | 사용자 경험 |
| 요청당 비용 | 운영 비용 관리 |
| 오류 유형 | 장애 원인 분류 |
| 재시도율 | 품질 또는 UX 문제 탐지 |
| 차단/거절 횟수 | 안전 정책 점검 |

## 로그에 남길 것

- 요청 ID
- 사용자 또는 조직 ID의 비식별 값
- 기능 이름
- 모델 또는 처리 경로
- 토큰/비용 요약
- 지연 시간
- 오류 코드
- 평가 점수 또는 사용자 피드백

## 로그에 남기지 말 것

- API 키와 토큰
- 비밀번호
- 원본 개인정보
- 내부 인증 쿠키
- 민감한 프롬프트 전문
- 고객 데이터 전문

## Claude에게 줄 요청

```text
다음 AI 기능의 모니터링과 로깅 설계를 작성해줘.

기능:
[설명]

요청:
1. 운영 지표
2. 로그 필드
3. 민감 정보 마스킹 기준
4. 알림 조건
5. 장애 조사 시 확인 순서
```

## 체크리스트

- [ ] 비용과 지연 시간이 관찰된다.
- [ ] 민감 정보가 로그에 남지 않는다.
- [ ] 장애 알림 기준이 있다.
- [ ] 사용자 피드백이 개선 작업으로 연결된다.
