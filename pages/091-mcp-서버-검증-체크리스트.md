# 091. MCP 서버 검증 체크리스트

난이도: 고급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

MCP 서버는 Claude에게 외부 시스템 접근 권한을 주는 통로입니다. 따라서 새 MCP 서버를 도입하기 전에는 코드, 권한, transport, 인증, 로그, 데이터 노출, 운영 주체를 검증해야 합니다.

## 5분 검증표

```text
MCP 서버 이름:
출처:
transport:
운영 주체:
필요 권한:
읽기 도구:
쓰기 도구:
인증 방식:
로그 위치:
민감 데이터 접근:
승인 필요 작업:
도입 결정:
```

## 코드 검토 항목

- 어떤 tools를 노출하는가
- tool description이 실제 동작과 맞는가
- 입력 validation이 있는가
- secret을 stdout/log로 출력하지 않는가
- HTTP 서버라면 Origin/auth 처리가 있는가
- stdio 서버라면 stdout에 MCP 메시지만 쓰는가
- dependency 취약점이 있는가

## tool description 검증

tool description은 Claude가 도구를 선택하는 기준입니다. 설명이 넓으면 도구가 잘못 호출될 수 있습니다.

확인할 것:

- 읽기 전용인지 쓰기 가능한지 설명되어 있는가?
- 접근 가능한 범위가 명시되어 있는가?
- 삭제, 전송, 게시 같은 부작용이 표시되어 있는가?
- 실패 시 되돌릴 수 있는지 알 수 있는가?
- 실제 코드 동작과 설명이 일치하는가?

## 권한 검토 항목

- 읽기와 쓰기가 분리되는가
- 최소 scope로 동작하는가
- 운영 데이터에 접근하는가
- PII를 반환하는가
- destructive tool이 있는가
- 사용자가 tool 호출을 거부할 수 있는가

## Claude Code에 요청할 검증 프롬프트

```text
이 MCP 서버 도입을 검토해줘.

확인할 것:
1. 노출하는 tools/resources/prompts
2. transport와 인증 방식
3. 필요한 권한과 과도한 권한
4. 민감 데이터 접근 가능성
5. destructive 동작
6. 로깅과 audit 가능성
7. 샌드박스 테스트 계획

아직 연결하거나 실행하지 말고 검토 결과만 작성해줘.
```

## 도입 결정 기준

| 결정 | 조건 |
| --- | --- |
| 도입 | 출처가 명확하고 최소 권한으로 필요한 작업만 수행 |
| 제한 도입 | 읽기 전용, sandbox, 특정 프로젝트로 제한 필요 |
| 보류 | 권한 범위나 로그 정책이 불명확 |
| 거절 | destructive tool, secret 노출, 검증 불가 코드 |

## 샌드박스 테스트 절차

1. 실제 운영 데이터가 없는 테스트 환경을 준비한다.
2. 읽기 도구만 먼저 호출한다.
3. 반환 데이터에 민감 정보가 섞이는지 확인한다.
4. 쓰기 도구가 있다면 초안/미리보기 모드가 있는지 확인한다.
5. 로그에 secret과 PII가 남는지 확인한다.
6. 연결 해제와 토큰 회수가 되는지 확인한다.

## 도입 후 점검

MCP 서버는 도입 후에도 주기적으로 확인해야 합니다.

- 서버 버전과 dependency 변경을 확인한다.
- 새 tool이 추가되었는지 확인한다.
- 권한 scope가 넓어지지 않았는지 확인한다.
- 로그 보관 위치와 접근 권한을 확인한다.
- 사용하지 않는 서버는 연결 해제한다.

## 서버 설명 검토

MCP 서버의 tool description은 Claude가 도구를 언제 쓸지 판단하는 중요한 정보입니다. 설명이 실제 동작보다 넓거나 모호하면 위험합니다.

나쁜 설명:

```text
Manage files.
```

좋은 설명:

```text
Read Markdown files from the configured docs directory. Does not write, delete, or access files outside the allowlisted root.
```

## 체크리스트

- [ ] 출처와 운영 주체를 확인했다.
- [ ] tools 목록과 권한을 확인했다.
- [ ] 읽기/쓰기/destructive 동작을 구분했다.
- [ ] 인증과 revoke 방법을 확인했다.
- [ ] 샌드박스에서 먼저 테스트한다.
- [ ] tool description과 실제 동작이 일치한다.
- [ ] 도입 후 정기 점검 주기를 정했다.
