# 020. Plan Mode 기초

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

Plan Mode는 Claude Code를 더 안전하게 쓰기 위한 핵심 작업 방식입니다. 복잡한 작업에서 Claude가 바로 파일을 수정하지 않고 먼저 문제를 분석하고 실행 계획을 제안하도록 만드는 흐름입니다.

Plan Mode의 목적은 속도를 늦추는 것이 아닙니다. 잘못된 방향으로 빠르게 가는 것을 막는 것입니다.

## Plan Mode가 필요한 상황

Plan Mode는 다음 상황에서 특히 유용합니다.

- 처음 보는 코드베이스
- 영향 범위가 큰 기능 개발
- 인증, 권한, 결제, 보안 관련 변경
- 데이터베이스 마이그레이션
- 대규모 리팩터링
- 테스트가 부족한 프로젝트
- 요구사항이 아직 모호한 작업

## Plan Mode가 덜 필요한 상황

다음 작업은 일반 요청으로 충분할 수 있습니다.

- 오타 수정
- 작은 README 문구 수정
- 단일 테스트 이름 변경
- 설명 요청
- 간단한 명령 확인

중요한 기준은 위험도입니다. 위험이 클수록 계획이 먼저입니다.

## 기본 사용 흐름

Claude Code에서 Plan Mode를 사용할 수 있는 환경이라면 `/plan`을 입력해 계획 중심 흐름으로 전환합니다.

```text
/plan
```

그 다음 작업을 설명합니다.

```text
사용자 프로필 수정 기능을 추가하고 싶다.
먼저 현재 구조를 분석하고 구현 계획을 작성해줘.
아직 파일은 수정하지 마.
```

Claude가 계획을 제시하면 사용자는 계획을 검토합니다. 계획이 너무 넓으면 범위를 줄이고, 빠진 테스트가 있으면 보완을 요청합니다.

## 좋은 Plan Mode 요청

```text
/plan

목표:
사용자가 자신의 프로필 이름과 소개를 수정할 수 있게 한다.

제약:
- 기존 인증 흐름은 변경하지 않는다.
- 새 라이브러리는 추가하지 않는다.
- 먼저 계획만 작성한다.

계획에 포함할 것:
1. 관련 파일
2. API 변경 여부
3. UI 변경 위치
4. 테스트 전략
5. 보안 고려사항
6. 단계별 구현 순서
```

## 계획 검토 질문

계획이 나온 뒤에는 이렇게 물어볼 수 있습니다.

```text
이 계획에서 가장 위험한 부분 3가지를 알려줘.
각 위험을 줄이는 방법도 함께 제안해줘.
```

또는:

```text
이 계획을 더 작게 나눠줘.
첫 번째 PR에 포함할 최소 범위만 제안해줘.
```

## Plan Mode와 승인

계획을 승인할 때도 범위를 제한하세요.

```text
좋아. 전체를 한 번에 하지 말고 1단계와 2단계만 진행해줘.
변경 후에는 테스트 명령을 실행하고 결과를 요약해줘.
```

이렇게 하면 큰 작업도 작은 단위로 안전하게 진행할 수 있습니다.

## 실습

다음 기능을 Plan Mode로 요청해 보세요.

```text
댓글 기능을 추가하고 싶다.
로그인한 사용자만 댓글을 작성할 수 있어야 한다.
댓글 작성, 목록 조회, 삭제 기능이 필요하다.
```

계획에 인증, API, UI, DB, 테스트, 보안이 포함되는지 확인하세요.

## 체크리스트

- [ ] 복잡한 작업 전에 Plan Mode를 사용한다.
- [ ] 계획에 예상 변경 파일이 포함되어 있다.
- [ ] 테스트 전략이 포함되어 있다.
- [ ] 위험 요소를 따로 확인한다.
- [ ] 전체 계획을 한 번에 승인하지 않고 작은 단계로 나눈다.

## 흔한 실수

- Plan Mode를 켜고도 “바로 구현해줘”라고 한다.
- 계획의 위험 요소를 확인하지 않는다.
- 너무 큰 작업을 한 번에 승인한다.
- 테스트 없는 계획을 그대로 실행한다.
