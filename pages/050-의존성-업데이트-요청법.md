# 050. 의존성 업데이트 요청법

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

의존성 업데이트는 단순히 최신 버전을 설치하는 일이 아닙니다. changelog, breaking changes, 보안 취약점, lockfile 변화, 테스트 범위를 함께 확인해야 합니다.

Claude Code에게 의존성 업데이트를 맡길 때는 “업데이트 후보 조사”와 “실제 설치”를 분리하세요.

## 먼저 조사하기

```text
아직 패키지를 설치하거나 lockfile을 수정하지 마.
먼저 의존성 업데이트 후보를 조사해줘.

확인할 것:
1. 현재 패키지 매니저
2. package 파일과 lockfile
3. 업데이트하려는 패키지의 현재 버전
4. 목표 버전
5. changelog와 breaking changes
6. 보안 취약점 여부
7. 영향을 받을 코드와 테스트
```

패키지 매니저는 npm, pnpm, yarn, Gradle, Maven, pip, Poetry 등 프로젝트마다 다릅니다.

## 요청 템플릿

```text
[패키지명]을 업데이트하고 싶다.

목표:
- [보안 패치 / 기능 필요 / 호환성 개선]

제약:
- 아직 설치하지 마.
- 현재 버전과 가능한 업데이트 버전을 조사해줘.
- changelog와 breaking changes를 확인해줘.
- lockfile 변경 영향을 설명해줘.
- 업데이트 후 실행할 테스트를 제안해줘.
```

## 보안 업데이트

보안 이슈라면 취약점과 영향 범위를 구분합니다.

```text
npm audit 결과를 바탕으로 조치 계획을 세워줘.

제약:
- 바로 npm audit fix를 실행하지 마.
- force 옵션이 필요한 수정은 위험도를 따로 설명해줘.
- production dependency와 dev dependency를 구분해줘.
- lockfile만 바뀌는지 코드 수정도 필요한지 알려줘.
```

`npm audit fix --force` 같은 명령은 major upgrade나 breaking change를 만들 수 있으므로 바로 승인하지 않습니다.

## 업데이트 승인

```text
좋아. [패키지명]만 [버전]으로 업데이트해줘.

제약:
- 다른 패키지는 업데이트하지 마.
- lockfile 변경을 요약해줘.
- 업데이트 후 관련 테스트와 빌드를 실행해줘.
- 실패하면 원인과 되돌릴 방법을 알려줘.
```

가능하면 한 번에 하나의 핵심 패키지만 업데이트합니다.

## 검증 항목

의존성 업데이트 후 확인할 것:

- lockfile이 예상대로 바뀌었는가
- 타입 오류가 생겼는가
- 빌드가 통과하는가
- 관련 테스트가 통과하는가
- 런타임 동작이 바뀌었는가
- 번들 크기가 크게 늘었는가
- 보안 경고가 줄었는가

## 실습

현재 프로젝트에서 의존성 하나를 골라 업데이트 계획만 작성해 보세요. 실제 설치는 하지 말고, changelog 확인과 테스트 계획까지 정리합니다.

## Claude Code에 입력할 프롬프트

```text
다음 의존성 업데이트 계획을 세워줘.

패키지:
[패키지명]

목표:
[보안/기능/호환성]

제약:
- 아직 설치하지 마.
- package 파일과 lockfile을 먼저 확인해줘.
- changelog와 breaking changes를 조사해줘.
- 업데이트 영향 파일과 테스트 범위를 제안해줘.
- 안전한 업데이트 명령을 제안하되 실행하지 마.

출력:
1. 현재 버전
2. 목표 버전
3. breaking changes
4. 업데이트 명령
5. 예상 변경 파일
6. 테스트 계획
7. 위험과 되돌리기 방법
```

## 체크리스트

- [ ] 패키지 매니저와 lockfile을 확인했다.
- [ ] changelog와 breaking changes를 확인했다.
- [ ] 한 번에 너무 많은 패키지를 업데이트하지 않는다.
- [ ] 보안 업데이트와 일반 업데이트를 구분한다.
- [ ] 업데이트 후 테스트와 빌드를 실행한다.

## 흔한 실수

- `npm audit fix --force`를 무비판적으로 실행한다.
- lockfile 변경을 검토하지 않는다.
- major version 변경인데 테스트 범위를 좁게 잡는다.
- 여러 패키지를 한 번에 올려 원인 추적을 어렵게 만든다.
