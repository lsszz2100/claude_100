## 핵심 개념

API 설계는 URL을 정하는 일이 아닙니다. 리소스, 권한, 요청/응답 스키마, 오류 형식, 페이지네이션, 버전 관리, 테스트, 문서화를 함께 설계하는 일입니다.

Claude Code에게 API 설계를 맡길 때는 OpenAPI, 기존 API 패턴, 인증/권한 규칙을 함께 확인하게 해야 합니다.

## 먼저 기존 API 패턴 확인

```text
아직 새 API를 만들지 마.
먼저 이 프로젝트의 기존 API 설계 패턴을 확인해줘.

확인할 것:
1. 라우트 구조
2. 요청/응답 형식
3. 오류 응답 형식
4. 인증/권한 처리 방식
5. validation 방식
6. pagination/sorting/filtering 패턴
7. OpenAPI 또는 API 문서 위치
```

새 API는 기존 API와 일관되어야 합니다.

## API 설계 템플릿

```text
다음 기능을 위한 API를 설계해줘.

기능:
[기능 이름]

사용자 흐름:
[클라이언트가 무엇을 해야 하는지]

리소스:
[예: posts, comments, users]

요구사항:
- [요구사항]

권한:
- [누가 무엇을 할 수 있는지]

제약:
- 기존 API 응답 형식을 따른다.
- OpenAPI 문서화 가능하게 설계한다.
- 아직 구현하지 말고 설계만 작성한다.

출력:
1. 엔드포인트 목록
2. 요청 스키마
3. 응답 스키마
4. 오류 응답
5. 인증/권한 규칙
6. 테스트 케이스
7. OpenAPI 초안
```

## 엔드포인트 설계 예시

댓글 기능이라면 다음처럼 리소스 중심으로 설계할 수 있습니다.

```text
GET    /posts/{postId}/comments
POST   /posts/{postId}/comments
DELETE /posts/{postId}/comments/{commentId}
```

수정 기능이 비범위라면 `PATCH`는 만들지 않습니다.

## 요청/응답 스키마

```json
{
  "content": "좋은 글입니다."
}
```

```json
{
  "id": "cmt_123",
  "postId": "post_123",
  "authorId": "user_123",
  "content": "좋은 글입니다.",
  "createdAt": "2026-05-03T12:00:00Z"
}
```

스키마에는 필수/선택 필드, 길이 제한, nullable 여부, 서버가 생성하는 필드를 구분해야 합니다.

## 오류 응답

오류 응답은 일관성이 중요합니다.

```json
{
  "error": {
    "code": "COMMENT_NOT_FOUND",
    "message": "Comment not found.",
    "requestId": "req_123"
  }
}
```

확인할 상태 코드:

- `400`: 잘못된 요청
- `401`: 인증 필요
- `403`: 권한 없음
- `404`: 리소스 없음
- `409`: 충돌
- `422`: 검증 실패
- `429`: 요청 제한
- `500`: 서버 오류

## 권한 설계

API 설계에서 권한은 별도 섹션으로 써야 합니다.

```text
권한 규칙:
- 댓글 목록은 게시글을 볼 수 있는 사용자만 조회할 수 있다.
- 댓글 작성은 로그인한 사용자만 가능하다.
- 댓글 삭제는 댓글 작성자 또는 관리자만 가능하다.
- 클라이언트가 보낸 authorId는 신뢰하지 않고 서버 세션에서 결정한다.
```

권한 규칙이 없으면 구현자가 각자 추정하게 됩니다.

## OpenAPI 초안 요청

```text
위 설계를 OpenAPI 3.1 YAML 초안으로 작성해줘.

포함할 것:
- paths
- requestBody
- responses
- components.schemas
- securitySchemes
- 주요 오류 응답

아직 파일은 만들지 말고 초안만 보여줘.
```

OpenAPI는 사람과 도구가 API를 이해할 수 있는 표준 형식입니다.

## 체크리스트

- [ ] 기존 API 패턴을 먼저 확인한다.
- [ ] 리소스와 엔드포인트가 일관적이다.
- [ ] 요청/응답/오류 스키마가 명확하다.
- [ ] 인증과 권한 규칙이 분리되어 있다.
- [ ] OpenAPI 문서화와 테스트 케이스가 포함되어 있다.

## 흔한 실수

- URL만 정하고 오류와 권한을 빠뜨린다.
- 클라이언트가 보낸 userId나 role을 신뢰한다.
- pagination 없이 목록 API를 만든다.
- 기존 API 응답 형식과 다르게 설계한다.
