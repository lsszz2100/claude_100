# 043. 코드 리뷰 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code에게 코드 리뷰를 맡길 때는 “어때?”라고 묻지 말고 버그, 회귀, 보안, 테스트 공백을 우선 찾게 해야 합니다. 리뷰의 목적은 칭찬이 아니라 출시 전에 문제를 줄이는 것입니다.

좋은 리뷰 요청은 diff 기준, 심각도, 파일 위치, 재현 가능성, 수정 제안을 요구합니다.

## 리뷰 전 준비

먼저 변경사항을 기준으로 리뷰하게 합니다.

```text
현재 git diff를 기준으로 코드 리뷰를 해줘.

초점:
- 실제 버그
- 회귀 가능성
- 보안 위험
- 테스트 공백
- 성능상 명백한 문제

스타일 취향이나 사소한 문구는 낮은 우선순위로 다뤄줘.
```

Git 저장소가 아니라면 변경 파일 목록을 먼저 제공하거나 최근 수정 파일을 기준으로 리뷰하게 합니다.

## 출력 형식

리뷰 결과는 심각도와 위치가 있어야 합니다.

```text
출력 형식:

## Findings

- Severity: High / Medium / Low
- File:
- Line:
- Problem:
- Why it matters:
- Suggested fix:

## Test Gaps

- [빠진 테스트]

## Open Questions

- [확인할 질문]
```

파일과 줄이 없는 리뷰는 실행하기 어렵습니다.

## 좋은 리뷰 요청

```text
현재 변경사항을 코드 리뷰해줘.

리뷰 기준:
1. 버그나 회귀를 먼저 찾는다.
2. 보안상 위험한 입력 처리나 권한 문제를 찾는다.
3. 테스트가 빠진 핵심 경로를 찾는다.
4. 성능 문제는 실제로 발생 가능성이 높은 것만 지적한다.
5. 단순 취향이나 스타일은 마지막에만 언급한다.

제약:
- 문제 없으면 억지로 찾지 마.
- 각 지적에는 파일과 위치를 포함해줘.
- 가능한 경우 수정 방향을 제안해줘.
- 아직 파일은 수정하지 마.
```

## 리뷰 범위 나누기

큰 변경은 한 번에 리뷰하지 말고 관점별로 나눕니다.

```text
1차로 버그와 회귀만 리뷰해줘.
보안, 성능, 스타일은 아직 보지 마.
```

그 다음:

```text
이제 같은 diff를 보안 관점으로만 리뷰해줘.
입력 검증, 인증, 권한, 비밀정보 노출을 중심으로 봐줘.
```

관점을 나누면 더 깊은 리뷰가 가능합니다.

## 테스트 공백 찾기

```text
현재 변경사항에서 테스트가 빠진 핵심 경로를 찾아줘.

출력:
- 빠진 테스트
- 왜 필요한지
- 어떤 파일에 추가하면 좋은지
- Given/When/Then 테스트 케이스
```

코드 리뷰에서 테스트 공백은 버그만큼 중요합니다.

## 리뷰 후 수정

리뷰 결과를 받은 뒤에는 모든 지적을 무조건 반영하지 않습니다.

```text
리뷰 지적을 다음 기준으로 분류해줘.

1. 반드시 수정해야 하는 버그
2. 테스트로 보강해야 하는 부분
3. 이번 PR 범위를 벗어난 개선
4. 무시해도 되는 취향성 의견

아직 파일은 수정하지 마.
```

이 분류를 거친 뒤 수정 범위를 승인합니다.

## 실습

작은 diff를 하나 만들고 Claude Code에게 다음 세 가지 리뷰를 각각 요청해 보세요.

1. 버그/회귀 리뷰
2. 보안 리뷰
3. 테스트 공백 리뷰

세 리뷰의 결과가 어떻게 다른지 비교합니다.

## Claude Code에 입력할 프롬프트

```text
현재 변경사항을 코드 리뷰해줘.

우선순위:
1. 실제 버그
2. 회귀 위험
3. 보안 문제
4. 테스트 공백
5. 유지보수 위험

출력 형식:
- Severity
- File/Line
- Problem
- Why it matters
- Suggested fix

제약:
- 문제 없으면 "중대한 문제 없음"이라고 말해줘.
- 스타일 취향은 주요 finding으로 올리지 마.
- 아직 파일은 수정하지 마.
```

## 체크리스트

- [ ] diff 기준으로 리뷰한다.
- [ ] 버그와 회귀를 우선한다.
- [ ] 파일과 위치를 요구한다.
- [ ] 테스트 공백을 별도로 확인한다.
- [ ] 리뷰 결과를 수정/보류/범위 밖으로 분류한다.

## 흔한 실수

- “전반적으로 어때?”라고만 묻는다.
- 스타일 의견을 버그와 같은 무게로 다룬다.
- 파일 위치 없는 추상적 리뷰를 받아들인다.
- 리뷰 지적을 검토 없이 모두 반영한다.
