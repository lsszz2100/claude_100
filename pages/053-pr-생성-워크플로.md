# 053. PR 생성 워크플로

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

Pull Request는 변경사항을 병합하기 전 팀이 검토하고 논의하는 단위입니다. Claude Code는 PR 본문 초안, 변경 요약, 테스트 결과, 리뷰 포인트를 만드는 데 유용합니다.

좋은 PR은 “무엇을 바꿨는지”뿐 아니라 “왜 바꿨고, 어떻게 검증했는지”를 보여줍니다.

## PR 전 확인

```text
PR을 만들기 전에 현재 변경사항을 점검해줘.

확인할 것:
1. 현재 브랜치
2. 변경 파일 목록
3. diff 요약
4. 테스트 실행 결과
5. 남은 TODO
6. 요청 범위를 벗어난 변경
7. 커밋 분리 필요 여부
```

GitHub 공식 문서 기준으로 PR은 브랜치의 변경을 제안하고 리뷰하는 협업 단위입니다. 그래서 PR 본문에는 리뷰어가 판단할 정보가 있어야 합니다.

## PR 본문 템플릿

```markdown
## Summary

- [변경 요약 1]
- [변경 요약 2]

## Why

- [문제 또는 요구사항]

## Changes

- [구체적 변경]

## Tests

- [x] [실행한 테스트 명령]
- [ ] [실행하지 못한 테스트와 이유]

## Screenshots

- [UI 변경이 있으면 첨부]

## Risks

- [리스크와 완화 방법]

## Review Focus

- [리뷰어가 특히 봐야 할 부분]
```

## Claude Code 요청

```text
현재 git diff를 기준으로 PR 본문 초안을 작성해줘.

포함할 것:
1. Summary
2. Why
3. Changes
4. Tests
5. Risks
6. Review Focus

제약:
- diff에 없는 내용을 쓰지 마.
- 테스트를 실행하지 않았으면 실행하지 않았다고 적어줘.
- 과장된 표현을 쓰지 마.
- 아직 PR은 만들지 마.
```

## draft PR 활용

작업이 끝나지 않았지만 팀원에게 방향을 공유하고 싶을 때는 draft PR이 유용합니다.

```text
이 변경사항을 draft PR로 공유할 때 적합한 본문을 작성해줘.

포함할 것:
- 아직 완료되지 않은 작업
- 리뷰 받고 싶은 설계 방향
- 실행한 테스트
- 병합 전 남은 조건
```

## 리뷰 포인트 분리

리뷰어가 쉽게 볼 수 있게 초점을 적습니다.

```markdown
## Review Focus

- `src/auth/session.ts`의 만료 세션 처리 분기가 기존 로그인 흐름을 깨지 않는지
- `src/auth/session.test.ts`의 실패 케이스가 충분한지
- API 오류 코드가 기존 클라이언트와 호환되는지
```

## 실습

작은 변경사항을 만든 뒤 Claude Code에게 PR 본문 초안을 만들게 하세요. 테스트를 실행하지 않았다면 그 사실을 명확히 적게 합니다.

## Claude Code에 입력할 프롬프트

```text
현재 변경사항으로 PR 본문 초안을 작성해줘.

먼저 확인:
- git status
- git diff
- 최근 커밋
- 실행한 테스트 또는 미실행 테스트

출력:
Markdown PR 본문

제약:
- 아직 PR을 생성하지 마.
- diff에 없는 내용은 쓰지 마.
- 리뷰어가 볼 파일과 위험 요소를 구체적으로 적어줘.
```

## 체크리스트

- [ ] 변경 요약과 이유가 있다.
- [ ] 테스트 결과가 정확히 적혀 있다.
- [ ] 리뷰 포인트가 구체적이다.
- [ ] 남은 TODO나 미실행 검증을 숨기지 않는다.
- [ ] PR 범위가 너무 넓지 않다.

## 흔한 실수

- PR 본문에 “수정함”만 적는다.
- 테스트하지 않았는데 테스트한 것처럼 쓴다.
- 리뷰어가 봐야 할 위험 파일을 숨긴다.
- 기능 변경, 포맷 변경, 의존성 변경을 한 PR에 섞는다.
