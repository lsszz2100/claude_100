# 040. 리팩터링 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

리팩터링은 동작을 바꾸지 않고 코드 구조를 개선하는 작업입니다. Claude Code에게 리팩터링을 맡길 때는 “동작 변경 금지”와 “검증 방법”을 명확히 해야 합니다.

리팩터링의 실패는 보통 코드가 덜 예뻐서가 아니라, 의도치 않게 동작을 바꾸었는데 검증하지 못해서 발생합니다.

## 리팩터링 전 확인

먼저 리팩터링 대상과 이유를 분리하세요.

```text
아직 파일은 수정하지 마.
아래 파일의 리팩터링 필요성을 검토해줘.

확인할 것:
1. 중복
2. 너무 긴 함수
3. 책임이 섞인 부분
4. 테스트 가능성을 떨어뜨리는 구조
5. 리팩터링하면 위험한 부분
6. 동작 유지 검증 방법
```

## 좋은 리팩터링 요청

```text
src/services/order.ts의 createOrder 흐름을 리팩터링하고 싶다.

목표:
- 함수 길이를 줄이고 검증 로직과 저장 로직을 분리한다.

제약:
- 외부 동작은 바꾸지 않는다.
- 함수의 public signature는 유지한다.
- 오류 메시지와 응답 형식은 변경하지 않는다.
- 새 라이브러리는 추가하지 않는다.

검증:
- 기존 단위 테스트를 통과해야 한다.
- createOrder 관련 테스트를 먼저 실행한다.

먼저 계획만 작성해줘.
```

## 리팩터링 단위 나누기

큰 리팩터링은 작은 단계로 나눕니다.

1. 테스트 또는 현재 동작 고정
2. 이름 변경 없이 내부 구조만 분리
3. 중복 제거
4. 타입 또는 인터페이스 정리
5. 테스트 실행

한 번에 파일 구조, 이름, 테스트, 동작을 모두 바꾸면 리뷰가 어려워집니다.

## 동작 고정 테스트

테스트가 부족한 코드는 리팩터링 전에 현재 동작을 고정하는 테스트를 먼저 추가할 수 있습니다.

```text
리팩터링 전에 현재 동작을 고정하는 characterization test가 필요한지 판단해줘.
필요하다면 테스트 케이스 후보를 먼저 제안해줘.
아직 제품 코드는 수정하지 마.
```

테스트가 없으면 리팩터링이 아니라 재작성에 가까워질 수 있습니다.

## 금지해야 할 행동

리팩터링 요청에는 금지 조건을 명시하세요.

```text
금지:
- 공개 API 이름 변경
- 오류 메시지 변경
- 데이터베이스 스키마 변경
- 새 의존성 추가
- 스타일 포맷 전체 변경
- 관련 없는 파일 수정
```

금지 조건은 Claude Code가 “더 나은 구조”를 과하게 해석하지 않도록 막습니다.

## 실습

긴 함수 하나를 대상으로 다음 요청을 만들어 보세요.

- 리팩터링 목적
- 유지해야 할 동작
- 바꾸면 안 되는 인터페이스
- 먼저 추가할 테스트
- 단계별 구현 순서

## Claude Code에 입력할 프롬프트

```text
아래 범위의 리팩터링 계획을 세워줘.

대상:
[파일 또는 함수]

목표:
[중복 제거 / 함수 분리 / 테스트 가능성 개선 등]

제약:
- 외부 동작은 바꾸지 마.
- public API, 응답 형식, 오류 메시지는 유지해.
- 관련 없는 포맷 변경은 하지 마.
- 아직 파일은 수정하지 마.

요청:
1. 현재 구조의 문제
2. 리팩터링 단계
3. 동작 유지 검증 방법
4. 먼저 추가할 테스트가 있는지
5. 위험 요소
```

## 체크리스트

- [ ] 리팩터링 목적이 명확하다.
- [ ] 동작 변경 금지 조건이 있다.
- [ ] 공개 인터페이스 유지 여부가 명시되어 있다.
- [ ] 관련 테스트 또는 동작 고정 테스트가 있다.
- [ ] 작은 단계로 나누어 승인한다.

## 흔한 실수

- “깔끔하게 정리해줘”라고만 요청한다.
- 테스트 없이 구조를 크게 바꾼다.
- 리팩터링 중 기능 변경을 함께 넣는다.
- 포맷 변경과 의미 있는 변경을 섞는다.
