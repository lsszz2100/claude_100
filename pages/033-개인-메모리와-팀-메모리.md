# 033. 개인 메모리와 팀 메모리

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code 메모리는 위치에 따라 목적이 다릅니다. 팀 전체가 공유해야 하는 규칙과 개인의 작업 취향을 섞으면 협업이 어려워집니다.

공식 문서 기준으로 대표적인 위치는 다음과 같습니다.

| 종류 | 위치 | 용도 |
| --- | --- | --- |
| 프로젝트 메모리 | `./CLAUDE.md` | 저장소와 함께 공유하는 팀 지침 |
| 사용자 메모리 | `~/.claude/CLAUDE.md` | 모든 프로젝트에 적용되는 개인 선호 |
| 엔터프라이즈 정책 | OS별 관리 경로 | 조직 전체 보안/표준 정책 |

`CLAUDE.local.md`는 예전에는 개인 프로젝트 메모리 용도로 쓰였지만, 현재는 개인 파일을 import하는 방식이 더 권장됩니다.

## 팀 메모리에 넣을 것

팀 메모리에는 팀원이 모두 따라야 하는 규칙만 넣습니다.

```markdown
# Team Project Rules

- Use `npm run typecheck` before opening a PR that changes TypeScript files.
- Do not edit files under `src/generated/`; regenerate them with the documented command.
- Follow existing API error response format in `docs/api-errors.md`.
- Ask before adding new runtime dependencies.
```

이 규칙들은 개인 취향이 아니라 프로젝트 품질과 협업에 직접 영향을 줍니다.

## 개인 메모리에 넣을 것

사용자 메모리에는 개인이 모든 프로젝트에서 선호하는 작업 방식을 넣습니다.

```markdown
# My Claude Code Preferences

- When explaining code, start with the high-level flow before line-by-line details.
- Prefer concise Korean summaries unless I ask for English.
- When tests fail, identify the first failing cause before suggesting broad fixes.
```

개인 메모리는 팀 저장소에 커밋하지 않습니다.

## 개인 프로젝트 지침 import

프로젝트마다 개인적으로 쓰는 샌드박스 URL이나 테스트 계정 메모가 필요할 수 있습니다. 이런 내용은 저장소에 넣지 말고 사용자 홈 아래 파일로 분리한 뒤 import합니다.

```markdown
# Project Instructions

## Team Rules

- Follow @docs/testing.md for test strategy.

## Individual Preferences

- @~/.claude/my-project-instructions.md
```

단, 이 import 줄 자체를 팀 저장소의 `CLAUDE.md`에 넣을지 여부는 팀과 합의해야 합니다. 개인 파일 경로가 모든 팀원에게 존재하지 않을 수 있기 때문입니다.

## 충돌 사례

나쁜 팀 메모리:

```markdown
- 항상 VS Code 기준으로 설명한다.
- 테스트는 내가 쓰는 로컬 DB 이름을 기준으로 실행한다.
- 답변은 무조건 반말로 한다.
```

이런 규칙은 팀 표준이 아니라 개인 환경입니다.

좋은 분리:

```markdown
# ./CLAUDE.md

- Tests must not depend on a developer's local database name.
- Use documented environment variables from `.env.example`.
```

```markdown
# ~/.claude/CLAUDE.md

- I use VS Code, so mention VS Code shortcuts when they are relevant.
- Answer me in Korean by default.
```

## /memory로 확인하기

Claude Code 세션에서 `/memory`를 사용하면 메모리 파일을 직접 편집하거나 어떤 메모리가 로드되는지 확인할 수 있습니다.

```text
/memory
```

팀 작업을 시작하기 전에는 현재 어떤 메모리가 적용되는지 확인하는 습관이 좋습니다.

## 실습

현재 프로젝트의 규칙을 다음 세 묶음으로 나눠 보세요.

1. 팀원이 모두 따라야 하는 규칙
2. 나만 선호하는 작업 방식
3. 저장소에 절대 넣으면 안 되는 민감 정보

그 다음 팀 규칙만 `CLAUDE.md` 초안에 남기고, 개인 규칙은 사용자 메모리 후보로 분리합니다.

## Claude Code에 입력할 프롬프트

```text
현재 프로젝트에서 CLAUDE.md에 넣을 규칙과 개인 메모리로 분리할 규칙을 나눠줘.

분류 기준:
1. 팀 전체가 지켜야 하는 규칙
2. 개인 취향 또는 개인 개발 환경
3. 저장소에 넣으면 안 되는 민감 정보
4. 별도 문서로 분리할 긴 설명

제약:
- 실제 파일을 근거로 판단해줘.
- 아직 파일은 수정하지 마.
- 개인 취향을 팀 규칙처럼 작성하지 마.
```

## 체크리스트

- [ ] 팀 규칙은 `./CLAUDE.md`에 둔다.
- [ ] 개인 취향은 `~/.claude/CLAUDE.md` 또는 개인 import로 분리한다.
- [ ] 샌드박스 URL, 개인 계정, 로컬 경로는 팀 메모리에 넣지 않는다.
- [ ] `/memory`로 로드된 메모리를 확인할 수 있다.
- [ ] 팀원이 동의하지 않은 규칙을 저장소에 커밋하지 않는다.

## 흔한 실수

- 개인 말투나 에디터 취향을 팀 `CLAUDE.md`에 넣는다.
- 팀 표준을 개인 메모리에만 넣어 다른 팀원이 알 수 없게 한다.
- 민감한 테스트 계정 정보를 메모리에 저장한다.
- `CLAUDE.local.md`에 의존해 worktree나 팀 환경에서 혼란을 만든다.
