# 051. 실패한 빌드 고치기

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

빌드 실패를 고칠 때는 로그 전체를 “에러가 많다”로 보지 말고 최초 원인과 파생 오류를 구분해야 합니다. Claude Code에게는 실패한 명령, 전체 로그의 핵심 구간, 최근 변경사항, 수정 금지 조건을 함께 줍니다.

빌드 오류는 하나의 원인에서 수십 개의 파생 오류가 나올 수 있습니다.

## 빌드 실패 요청 템플릿

```text
빌드 실패를 조사해줘.

실행한 명령:
[예: npm run build]

실패 로그:
[최초 실패 지점부터 요약까지]

최근 변경:
[관련 가능 변경]

제약:
- 아직 파일은 수정하지 마.
- 최초 원인과 파생 오류를 구분해줘.
- 수정 후보를 위험도 순서로 제안해줘.
- 가장 작은 수정부터 제안해줘.
```

## 로그 분석 기준

Claude Code에게 다음을 요구하세요.

```text
로그를 다음 기준으로 분해해줘.

1. 실제로 실행된 명령
2. 최초 실패 메시지
3. 실패한 파일 또는 모듈
4. 파생 오류
5. 최근 변경과의 관련성
6. 재현 명령
7. 최소 수정 계획
```

## 흔한 원인

| 원인 | 예시 |
| --- | --- |
| 타입 오류 | TypeScript 타입 불일치 |
| 누락된 import | 파일명 변경 후 import 미수정 |
| 환경 변수 | 빌드 시 필요한 값 없음 |
| 의존성 문제 | lockfile 불일치, peer dependency |
| 테스트 실패 | 빌드 과정에서 테스트 실행 |
| lint 실패 | CI에서 lint를 빌드 실패로 처리 |
| 경로 대소문자 | Windows에서는 통과, Linux CI에서 실패 |

CI에서만 실패한다면 로컬 OS와 CI 환경 차이를 확인해야 합니다.

## 수정 승인

```text
가장 가능성 높은 원인 1개만 수정해줘.

제약:
- 빌드 통과만을 위해 타입을 any로 우회하지 마.
- 테스트를 삭제하거나 건너뛰지 마.
- 관련 없는 포맷 변경은 하지 마.
- 수정 후 같은 빌드 명령을 다시 실행해줘.
```

빌드만 통과시키려고 검증을 약화하면 나중에 더 큰 문제가 됩니다.

## CI 실패일 때

```text
GitHub Actions 빌드 실패 로그를 분석해줘.

확인할 것:
- 실패한 job과 step
- 로컬과 CI의 Node/Java/Python 버전 차이
- 캐시 또는 lockfile 문제
- 환경 변수 누락
- OS 대소문자 경로 문제
- 재현 가능한 로컬 명령
```

## 실습

아래 로그 상황을 빌드 조사 요청으로 바꿔 보세요.

```text
CI에서 TypeScript build가 실패했고, 로컬 Windows에서는 통과한다.
```

OS 차이, 대소문자 경로, Node 버전, lockfile, 최초 오류를 포함하세요.

## Claude Code에 입력할 프롬프트

```text
빌드 실패를 조사해줘.

명령:
[빌드 명령]

로그:
[실패 로그]

환경:
[로컬/CI, OS, 런타임 버전]

제약:
- 아직 파일은 수정하지 마.
- 최초 원인과 파생 오류를 구분해줘.
- 재현 명령을 제안해줘.
- 가장 작은 수정 계획을 작성해줘.
- 수정 후 다시 실행할 검증 명령을 알려줘.
```

## 체크리스트

- [ ] 실패한 명령을 정확히 제공한다.
- [ ] 최초 오류와 파생 오류를 구분한다.
- [ ] 로컬과 CI 환경 차이를 확인한다.
- [ ] 검증을 약화하는 수정은 피한다.
- [ ] 수정 후 같은 명령을 다시 실행한다.

## 흔한 실수

- 로그 마지막 줄만 보고 원인을 판단한다.
- 테스트를 삭제해서 빌드를 통과시킨다.
- 타입 오류를 `any`로 숨긴다.
- CI 환경 차이를 확인하지 않는다.
