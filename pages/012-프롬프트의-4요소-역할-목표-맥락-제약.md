# 012. 프롬프트의 4요소: 역할, 목표, 맥락, 제약

난이도: 초급  
기준일: 2026년 05월 03일

## 핵심 개념

프롬프트의 기본은 네 가지입니다. 역할, 목표, 맥락, 제약입니다. 이 네 가지가 명확하면 Claude는 더 안정적인 답변을 합니다. 이 네 가지가 빠지면 Claude는 부족한 정보를 추측으로 채웁니다.

## 1. 역할

역할은 Claude가 어떤 관점으로 문제를 봐야 하는지 정합니다.

```text
너는 시니어 백엔드 개발자다.
```

```text
너는 보안 리뷰어다.
```

```text
너는 초급자에게 설명하는 튜터다.
```

역할은 답변의 깊이와 기준을 바꿉니다. 같은 코드라도 튜터는 설명을, 리뷰어는 위험을, 구현자는 수정 방법을 우선합니다.

## 2. 목표

목표는 무엇을 달성해야 하는지 알려줍니다.

```text
이 함수의 동작을 유지하면서 가독성을 높이고 싶다.
```

```text
로그인 실패 원인을 찾고 재발 방지 테스트를 추가하고 싶다.
```

목표가 없으면 Claude는 무엇을 최적화해야 할지 알 수 없습니다.

## 3. 맥락

맥락은 Claude가 판단할 배경입니다.

```text
이 프로젝트는 Next.js와 PostgreSQL을 사용한다.
현재 문제는 모바일에서만 발생한다.
최근 인증 라이브러리를 업데이트했다.
```

맥락은 길수록 좋은 것이 아니라, 판단에 필요한 만큼 정확해야 합니다.

## 4. 제약

제약은 Claude가 하지 말아야 할 것과 지켜야 할 조건입니다.

```text
아직 파일은 수정하지 마.
테스트 파일 외에는 변경하지 마.
외부 라이브러리는 추가하지 마.
현재 API 응답 형식은 유지해야 한다.
```

Claude Code에서는 제약이 특히 중요합니다. 제약은 변경 범위를 통제하는 안전장치입니다.

## 완성형 템플릿

```text
역할:
너는 [역할]이다.

목표:
[달성하고 싶은 결과]

맥락:
[프로젝트, 사용자, 오류, 제약 배경]

제약:
[수정 금지, 범위 제한, 라이브러리 제한, 호환성 조건]

원하는 출력:
[표, 단계별 계획, 코드 diff, 체크리스트 등]
```

## 예시

```text
역할:
너는 시니어 프론트엔드 개발자이자 코드 리뷰어다.

목표:
로그인 폼의 모바일 사용성을 개선하고 싶다.

맥락:
React 프로젝트이고, 사용자가 모바일에서 제출 버튼을 누르기 어렵다는 피드백이 있다.

제약:
아직 파일은 수정하지 마. 기존 디자인 시스템 컴포넌트만 사용해야 한다.

원하는 출력:
문제점, 개선안, 수정 대상 파일, 테스트 방법을 표로 정리해줘.
```

## Claude에게 입력할 프롬프트

```text
내가 작성한 프롬프트를 역할, 목표, 맥락, 제약 기준으로 리뷰해줘.

프롬프트:
[여기에 입력]

출력:
1. 빠진 요소
2. 모호한 표현
3. 위험한 지시
4. 개선된 최종 프롬프트
```

## 체크리스트

- [ ] 역할이 작업 목적과 맞다.
- [ ] 목표가 한 문장으로 명확하다.
- [ ] 맥락이 판단에 필요한 만큼 들어 있다.
- [ ] 제약 조건이 실행 범위를 제한한다.
- [ ] 출력 형식이 정해져 있다.

## 흔한 실수

- 역할을 과하게 많이 부여한다.
- 목표가 여러 개라 우선순위가 없다.
- 맥락 없이 결론만 요구한다.
- 제약 조건 없이 Claude Code에게 수정 작업을 맡긴다.
