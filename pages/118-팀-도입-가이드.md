# 118. 팀 도입 가이드

저자: AI_Innovation_Studio

Claude Code를 팀에 도입할 때는 “모두 쓰게 하기”보다 “작고 안전한 반복 작업 하나를 표준화하기”가 먼저입니다. 팀 규칙, 권한, 검토 기준이 없으면 생산성보다 혼란이 커질 수 있습니다.

팀 도입의 핵심은 도구 사용률이 아니라 반복 가능한 품질 기준입니다. 누가 Claude Code를 쓰더라도 변경 범위, 테스트 결과, 보안 기준, 리뷰 방식이 비슷하게 유지되어야 합니다.

처음부터 모든 업무를 자동화하려고 하면 실패하기 쉽습니다. 팀이 이미 자주 하고 있고, 실패해도 되돌릴 수 있으며, 결과를 사람이 검토할 수 있는 업무부터 시작하세요.

## 도입 단계

1. 개인 실험
2. 작은 팀 파일럿
3. 공통 `CLAUDE.md` 작성
4. 리뷰와 테스트 기준 합의
5. Skills 또는 Hooks 일부 도입
6. MCP와 외부 도구 연결 검토
7. 운영 표준 문서화

## 먼저 표준화할 작업

- PR 리뷰 초안
- 테스트 실패 로그 분석
- README와 운영 문서 갱신
- 릴리스 노트 초안
- 반복 체크리스트 생성

## 4주 파일럿 예시

1주차에는 개인 실험을 진행합니다. 각자 반복 업무 하나를 골라 Claude Code로 처리해 보고, 어떤 요청이 잘 통했는지 기록합니다.

2주차에는 작은 팀 파일럿을 진행합니다. PR 리뷰 초안, 테스트 실패 분석, README 갱신처럼 위험이 낮은 작업 하나를 정해 공통 프롬프트를 만듭니다.

3주차에는 `CLAUDE.md`와 기본 체크리스트를 정리합니다. 실행 명령, 테스트 기준, 금지 파일, 리뷰 방식, 커밋 전 확인 사항을 문서화합니다.

4주차에는 결과를 평가합니다. 시간이 줄었는지, 리뷰 품질이 흔들리지 않았는지, 보안상 불편한 지점이 있었는지 확인하고 계속 쓸 범위를 결정합니다.

## 팀 규칙 예시

- 운영 배포 명령은 자동 실행하지 않는다.
- secret, 고객 데이터, 내부 토큰은 프롬프트에 넣지 않는다.
- 대규모 변경은 계획 승인 후 진행한다.
- Claude가 작성한 코드는 사람이 리뷰한다.
- 테스트 결과와 미실행 검증을 PR에 남긴다.

## 성공 지표

도입 성공은 사용 횟수보다 결과 품질로 판단해야 합니다.

| 지표 | 확인 이유 |
| --- | --- |
| 리뷰에서 발견된 실제 문제 | 품질 향상 여부 |
| 테스트 실패 분석 시간 | 반복 작업 절감 |
| 문서 최신화 주기 | 유지보수 개선 |
| 미실행 검증 보고율 | 투명성 |
| 보안 위반 또는 위험 명령 차단 | 안전성 |

파일럿이 끝나면 계속 쓸 작업, 중단할 작업, 더 엄격한 승인 절차가 필요한 작업을 나누세요.

## Claude에게 줄 요청

```text
우리 팀에 Claude Code를 도입하기 위한 4주 파일럿 계획을 작성해줘.

팀 상황:
- 인원:
- 기술 스택:
- 반복 업무:
- 보안 제약:

출력:
1. 주차별 목표
2. 사용할 기능
3. 금지할 사용 방식
4. 성공 지표
5. 팀 문서 초안 목차
```

## 체크리스트

- [ ] 도입 목적이 명확하다.
- [ ] 보안과 권한 기준이 있다.
- [ ] 성공 지표를 정했다.
- [ ] 팀원이 거절하거나 중단할 기준도 정했다.
- [ ] 파일럿 종료 후 계속할 범위와 중단할 범위를 정한다.
