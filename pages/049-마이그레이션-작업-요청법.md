# 049. 마이그레이션 작업 요청법

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

마이그레이션은 스키마 변경만이 아니라 데이터와 애플리케이션 배포 순서를 함께 다루는 작업입니다. 운영 환경에서는 expand-and-contract 패턴처럼 “추가 → 이중 쓰기/읽기 전환 → 데이터 이동 → 정리” 순서가 필요할 수 있습니다.

Claude Code에게 마이그레이션을 요청할 때는 dry run, 백업, 검증, 모니터링, 롤백 조건을 반드시 포함하세요.

## expand-and-contract 개념

컬럼 이름 변경처럼 단순해 보이는 작업도 운영에서는 단계가 필요합니다.

1. Expand: 새 컬럼을 추가한다.
2. 애플리케이션이 새 컬럼도 쓰게 한다.
3. 기존 데이터를 새 컬럼으로 복사한다.
4. 읽기 경로를 새 컬럼으로 바꾼다.
5. 충분히 검증한 뒤 기존 컬럼을 제거한다.

이 방식은 배포 중 구버전 코드와 신버전 코드가 함께 떠 있는 상황을 고려합니다.

## 요청 템플릿

```text
다음 마이그레이션 계획을 작성해줘.

목표:
[예: users.name을 users.display_name으로 대체]

제약:
- 데이터 손실이 없어야 한다.
- 운영 중 무중단 또는 최소 중단을 목표로 한다.
- migration과 애플리케이션 배포 순서를 분리해줘.
- dry run 또는 staging 검증 절차를 포함해줘.
- 아직 파일은 수정하지 마.

출력:
1. 현재 상태 확인
2. expand 단계
3. 데이터 backfill 단계
4. 애플리케이션 전환 단계
5. contract 단계
6. 검증 쿼리
7. 롤백 조건
8. 모니터링 항목
```

## dry run과 staging

실제 운영 DB에 적용하기 전에 가능한 한 staging 또는 운영 복제본에서 검증합니다.

```text
이 migration을 운영에 적용하기 전에 staging에서 검증할 절차를 작성해줘.

포함할 것:
- 적용 전 row count 확인
- migration 실행
- backfill 결과 검증
- 애플리케이션 smoke test
- 성능 또는 lock 위험 확인
- 롤백 리허설
```

## 데이터 backfill 스크립트 요청

```text
데이터 backfill 스크립트 계획을 작성해줘.

제약:
- 한 번에 전체 테이블을 잠그지 않도록 batch 처리한다.
- 재실행해도 안전한 idempotent 방식이어야 한다.
- 진행 상황을 로그로 남긴다.
- 실패 시 중단 지점부터 재개할 수 있어야 한다.
- 아직 스크립트는 만들지 말고 설계만 보여줘.
```

대형 테이블은 한 번의 `UPDATE`가 위험할 수 있습니다. 배치 처리와 재실행 가능성이 중요합니다.

## 롤백보다 복구를 생각하기

모든 migration이 단순 롤백 가능한 것은 아닙니다. 특히 데이터 삭제나 타입 변경은 되돌리기 어렵습니다.

```text
이 migration은 단순 rollback이 가능한지 판단해줘.
불가능하다면 복구 전략과 백업 지점을 제안해줘.
```

## 실습

다음 작업을 expand-and-contract 방식으로 나눠 보세요.

```text
profiles.bio 컬럼을 profiles.biography로 이름 변경한다.
```

각 단계에서 DB 변경, 애플리케이션 변경, 검증 쿼리를 작성하세요.

## Claude Code에 입력할 프롬프트

```text
다음 마이그레이션을 안전하게 설계해줘.

목표:
[마이그레이션 목표]

운영 조건:
- 데이터 손실 금지
- 서비스 중단 최소화
- 구버전/신버전 코드가 동시에 떠 있을 수 있음

요청:
1. expand-and-contract 단계
2. migration 파일 생성 전 확인사항
3. backfill 전략
4. 검증 쿼리
5. staging dry run 절차
6. 롤백 또는 복구 계획
7. 모니터링 항목

제약:
- 아직 파일은 수정하지 마.
```

## 체크리스트

- [ ] 배포 순서가 DB와 앱 코드로 분리되어 있다.
- [ ] backfill이 재실행 가능하다.
- [ ] staging dry run 절차가 있다.
- [ ] 검증 쿼리가 있다.
- [ ] rollback 가능 여부와 복구 전략을 구분했다.

## 흔한 실수

- rename을 한 번의 migration으로 끝내려 한다.
- 대형 테이블을 한 번에 업데이트한다.
- backfill 재실행 가능성을 고려하지 않는다.
- 롤백 불가능한 변경을 rollback 가능하다고 가정한다.
