# 022. 파일 읽기와 코드베이스 요약

난이도: 초급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

Claude Code의 강점은 프로젝트 폴더 안의 파일을 읽고, 관계를 파악하고, 필요한 작업을 제안할 수 있다는 점입니다. 하지만 “전체 다 읽고 설명해줘”라고 요청하면 시간이 오래 걸리고 중요한 내용이 흐려질 수 있습니다.

좋은 코드베이스 요약은 모든 파일을 나열하는 것이 아닙니다. 프로젝트를 이해하는 데 필요한 핵심 파일을 먼저 읽고, 구조와 실행 흐름을 초급자가 판단할 수 있게 정리하는 것입니다.

## 먼저 읽히면 좋은 파일

대부분의 프로젝트에서는 다음 순서가 유용합니다.

| 우선순위 | 파일 또는 폴더 | 확인할 내용 |
| --- | --- | --- |
| 1 | `README.md` | 프로젝트 목적, 실행 방법, 배포 방법 |
| 2 | 패키지/빌드 설정 | 언어, 프레임워크, 스크립트, 의존성 |
| 3 | 폴더 구조 | 앱 코드, 테스트, 문서, 설정 위치 |
| 4 | 진입점 | 서버 시작, 앱 렌더링, CLI 시작점 |
| 5 | 테스트 파일 | 검증 방식과 주요 기능 |

JavaScript 프로젝트라면 `package.json`, Python 프로젝트라면 `pyproject.toml` 또는 `requirements.txt`, Android 프로젝트라면 `build.gradle` 계열 파일이 중요합니다.

## 나쁜 요청

```text
이 프로젝트 전부 분석해줘.
```

이 요청은 범위가 너무 넓습니다. Claude Code가 많은 파일을 읽을 수는 있지만, 초급자가 바로 쓸 수 있는 요약으로 나오지 않을 가능성이 큽니다.

## 좋은 요청

```text
아직 파일은 수정하지 마.
먼저 README, 패키지 설정 파일, 최상위 폴더 구조를 읽고
이 프로젝트를 신규 팀원에게 설명하듯 요약해줘.

포함할 것:
1. 프로젝트 목적
2. 기술 스택
3. 실행 명령
4. 테스트 명령
5. 주요 폴더 역할
6. 처음 봐야 할 핵심 파일 5개
```

이 요청은 읽을 범위와 출력 형식을 함께 줍니다. Claude Code가 더 안정적으로 움직입니다.

## 요약을 검증하는 법

Claude Code의 요약은 유용하지만 항상 맞는 것은 아닙니다. 다음 질문으로 검증하세요.

```text
방금 요약에서 실제 파일 근거가 있는 내용과 추정한 내용을 구분해줘.
각 항목 옆에 근거 파일 경로를 적어줘.
```

파일 근거를 요구하면 추측이 줄어듭니다. 특히 실행 명령, 테스트 명령, 배포 명령은 반드시 실제 설정 파일이나 README에 근거해야 합니다.

## 코드베이스 지도 만들기

조금 더 깊게 이해하려면 “지도”를 요청하세요.

```text
이 프로젝트의 코드베이스 지도를 만들어줘.

형식:
- 주요 폴더
- 각 폴더의 책임
- 서로 의존하는 흐름
- 데이터가 들어오고 나가는 경로
- 수정 전에 조심해야 할 파일

아직 파일은 수정하지 마.
```

지도는 이후 버그 수정, 기능 개발, 리팩터링 요청의 기반이 됩니다.

## 실습

현재 작업 중인 프로젝트에서 다음을 해 보세요.

1. Claude Code에게 README와 설정 파일만 읽고 요약하게 한다.
2. 그 다음 주요 폴더 3개만 추가로 읽게 한다.
3. 실행 명령과 테스트 명령의 근거 파일을 요구한다.
4. “추정한 내용”을 따로 표시하게 한다.
5. 마지막으로 신규 팀원용 10줄 요약을 작성하게 한다.

## Claude Code에 입력할 프롬프트

```text
현재 프로젝트를 처음 보는 사람이라고 가정하고 설명해줘.

제약:
- 아직 파일은 수정하지 마.
- 먼저 README, 설정 파일, 최상위 폴더 구조만 확인해줘.
- 모르는 내용은 추정하지 말고 "확인 필요"라고 표시해줘.

출력 형식:
1. 한 문장 요약
2. 기술 스택
3. 실행 명령
4. 테스트 명령
5. 주요 폴더 역할
6. 핵심 파일 5개
7. 다음에 읽어야 할 파일
8. 확인이 더 필요한 질문
```

## 체크리스트

- [ ] 처음부터 전체 파일을 다 읽히지 않는다.
- [ ] README와 설정 파일을 우선 확인한다.
- [ ] 실행 명령과 테스트 명령의 근거 파일을 확인한다.
- [ ] 요약과 추정을 구분하게 한다.
- [ ] 다음 작업 전에 핵심 파일 목록을 만든다.

## 흔한 실수

- “전부 분석”처럼 범위가 없는 요청을 한다.
- 테스트와 실행 명령을 실제로 확인하지 않는다.
- Claude의 추정을 프로젝트 사실로 받아들인다.
- 대용량 로그, 빌드 산출물, 의존성 폴더까지 읽히게 둔다.
