# 097. 구현 에이전트

난이도: 고급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

구현 에이전트는 실제 파일을 수정할 수 있는 subagent입니다. 가장 강력하지만 가장 위험하므로 파일 소유권, 범위, 금지사항, 테스트 기준을 명확히 해야 합니다.

구현 에이전트에게는 “무엇을 만들지”보다 “어디까지 해도 되는지”를 더 구체적으로 알려야 합니다.

## 설정 예시

````markdown
---
name: implementer
description: Implement small, well-scoped code changes after a plan is approved. Use when the task has clear requirements, file ownership, and test criteria.
tools: Read, Grep, Glob, Edit, Bash
---

You are an implementation agent.

Rules:
- Keep changes scoped to the assigned files.
- Do not refactor unrelated code.
- Do not add dependencies unless explicitly requested.
- Do not modify secrets, migrations, or deployment files without explicit approval.
- Run the smallest relevant test after changes.
- Summarize changed files and tests.
````

## 구현 요청 템플릿

```text
구현 에이전트를 사용해 아래 작업을 처리해줘.

목표:

담당 파일:
- 

읽어야 할 파일:
- 

변경 금지:
- 

수용 기준:
- 

검증:
- 실행할 테스트:
- 실행하지 못하면 이유를 남길 것:

주의:
- 다른 사람이 같은 코드베이스에서 작업 중일 수 있으니 관련 없는 변경은 하지 마.
```

## 파일 소유권 예시

좋은 범위:

```text
담당 파일:
- src/features/profile/ProfileForm.tsx
- src/features/profile/profileSchema.ts
- src/features/profile/ProfileForm.test.tsx
```

나쁜 범위:

```text
전체 프론트엔드를 적절히 고쳐줘.
```

## 작업 단위 기준

구현 에이전트에게 맡기기 좋은 작업은 작고 검증 가능해야 합니다.

| 적합한 작업 | 부적합한 작업 |
| --- | --- |
| 단일 컴포넌트 버그 수정 | 제품 방향을 정해야 하는 기능 |
| 명확한 API 필드 추가 | 여러 팀 계약이 필요한 API 재설계 |
| 실패 테스트 원인 수정 | 원인 불명 장애 전체 조사와 수정 |
| 문서화된 수용 기준 구현 | “UX를 좋게 개선”처럼 열린 요구 |

구현 범위가 커지면 먼저 계획 에이전트나 메인 세션에서 파일 소유권을 나누고, 구현 에이전트는 그 일부만 맡기는 편이 안전합니다.

## 구현 전 확인 질문

구현 에이전트가 바로 수정하면 안 되는 경우도 있습니다.

- 수용 기준이 없다.
- 수정 파일 범위가 넓다.
- DB schema 변경이 필요하다.
- 인증/권한 정책 결정이 필요하다.
- 새 dependency가 필요하다.
- 테스트 명령이 불명확하다.

이 경우에는 구현보다 계획을 먼저 작성하게 해야 합니다.

## 결과 요약 형식

```text
Changed files:
- 

Behavior changed:
- 

Tests run:
- 

Not run:
- 

Risks:
- 
```

## 동시 작업 주의

여러 구현 에이전트를 병렬로 쓸 때는 각 에이전트에게 다음 문장을 포함하세요.

```text
이 코드베이스에서 다른 작업자가 동시에 작업 중일 수 있다.
관련 없는 파일을 되돌리거나 정리하지 말고, 담당 범위 밖 변경을 발견하면 요약만 남겨라.
```

이는 충돌을 없애지는 못하지만, 한 에이전트가 다른 사람의 변경을 “불필요한 diff”로 판단해 되돌리는 일을 줄입니다.

## 체크리스트

- [ ] 담당 파일과 금지 파일이 분명하다.
- [ ] 새 dependency 추가 여부가 명시되어 있다.
- [ ] 테스트 기준이 있다.
- [ ] 결과 요약에 변경 파일과 검증이 포함된다.
- [ ] 작업 단위가 작고 수용 기준이 검증 가능하다.
- [ ] 동시 작업자의 변경을 되돌리지 말라는 지시가 있다.
