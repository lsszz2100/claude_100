## 핵심 개념

Notion, Jira, Slack 같은 업무 도구를 MCP로 연결하면 요구사항, 이슈, 회의 기록, 팀 논의를 Claude Code 작업에 바로 연결할 수 있습니다. 하지만 업무 도구에는 민감한 정보가 많기 때문에 범위와 권한 설계가 필수입니다.

## 도구별 적합한 사용

| 도구 | 적합한 작업 |
| --- | --- |
| Notion | PRD, 회의록, 운영 문서 요약 |
| Jira | 이슈 요구사항 분석, 작업 상태 정리 |
| Slack | 논의 요약, 결정사항 추출 |
| Linear | 이슈 기반 구현 계획 |

## 읽기 전용부터 시작

```text
Jira MCP로 ENG-123 이슈를 읽고 구현 계획만 작성해줘.

제약:
- 이슈 상태를 변경하지 마.
- 댓글을 작성하지 마.
- 저장소 파일도 아직 수정하지 마.
- 요구사항, 확인 질문, 테스트 전략만 정리해줘.
```

## 쓰기 작업은 별도 승인

```text
아래 댓글을 Jira 이슈에 남기기 전에 초안만 보여줘.
내가 승인하기 전에는 게시하지 마.
```

Slack도 마찬가지입니다. 메시지 전송은 팀 커뮤니케이션에 영향을 주므로 초안과 승인을 분리하세요.

## 권한 정책

- 개인 DM 읽기 금지
- 특정 프로젝트/채널만 허용
- 쓰기 권한은 최소화
- OAuth token 정기 점검
- 퇴사자/외부 협력자 권한 정리
- 민감 키워드 마스킹

## 데이터 분류

업무 도구 연결 전에 데이터 등급을 나누세요.

| 등급 | 예시 | 기본 정책 |
| --- | --- | --- |
| 공개 가능 | 공개 문서, 일반 가이드 | 읽기 허용 가능 |
| 내부 | 팀 회의록, 개발 이슈 | 프로젝트 범위 제한 |
| 민감 | 고객명, 계약 조건, 장애 보고 | 요약 전 마스킹 |
| 제한 | HR, 법무, 보안 사고, secret | 기본 차단 |

MCP가 읽을 수 있는 공간은 사람이 읽을 수 있는 공간보다 좁아야 합니다. “팀원이 볼 수 있다”가 곧 “AI 도구가 읽어도 된다”는 뜻은 아닙니다.

## 도구별 승인 기준

| 작업 | 기본 정책 |
| --- | --- |
| Notion 문서 읽기 | 프로젝트 문서만 허용 |
| Notion 문서 수정 | 초안 확인 후 승인 |
| Jira 이슈 읽기 | 특정 프로젝트만 허용 |
| Jira 상태 변경 | 별도 승인 필요 |
| Slack 채널 요약 | 공개/프로젝트 채널만 |
| Slack 메시지 발송 | 초안 승인 후 |

## 좋은 연결 시나리오

```text
Jira 이슈를 읽고 GitHub PR diff와 비교해줘.

출력:
1. 요구사항 충족 여부
2. 빠진 수용 기준
3. 테스트 공백
4. 리뷰어에게 남길 질문 초안

제약:
- Jira 상태 변경 금지
- Slack 메시지 발송 금지
- PR 코멘트 작성 금지
```

## 위험 신호

- 개인 DM이나 비공개 HR 채널을 읽으려 한다.
- 이슈 상태 변경과 코드 수정을 한 번에 요청한다.
- Slack 메시지 발송을 자동화한다.
- 회의록에 고객명, 계약 조건, secret이 그대로 포함되어 있다.
- 조직 전체 권한을 요구한다.

## 감사 기록

업무 도구 MCP는 다음 기록을 남기는 편이 좋습니다.

- 어떤 문서나 이슈를 읽었는가
- 어떤 초안을 만들었는가
- 어떤 댓글이나 메시지를 게시했는가
- 누가 승인했는가
- 게시 후 수정이나 삭제가 있었는가

## 체크리스트

- [ ] 읽기 전용부터 연결한다.
- [ ] 채널/프로젝트 범위를 제한한다.
- [ ] 댓글/메시지 작성은 승인 후 진행한다.
- [ ] 개인 DM과 민감 문서는 제외한다.
- [ ] OAuth 토큰과 권한을 주기적으로 점검한다.
- [ ] HR, 법무, 보안 사고 채널은 기본 차단한다.
- [ ] 게시 작업은 승인자와 게시 내용을 기록한다.
