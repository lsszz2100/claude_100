# 032. 프로젝트 메모리 설계

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

프로젝트 메모리 설계의 핵심은 “항상 필요한 정보”와 “필요할 때만 읽을 정보”를 분리하는 것입니다. 모든 규칙과 문서를 `CLAUDE.md`에 넣으면 Claude Code가 매번 큰 맥락을 들고 시작하게 됩니다.

좋은 설계는 `CLAUDE.md`를 얇고 정확하게 유지하고, 긴 문서는 `@import` 또는 일반 문서 링크로 분리합니다.

## 메모리에 넣을 정보

`CLAUDE.md`에 직접 넣기 좋은 정보는 다음과 같습니다.

| 정보 | 이유 |
| --- | --- |
| 프로젝트 한 문장 설명 | 모든 작업의 기본 전제 |
| 자주 쓰는 명령 | 매번 탐색하는 비용 절약 |
| 절대 지켜야 할 금지사항 | 사고 방지 |
| 테스트 우선순위 | 수정 후 검증 안정화 |
| 주요 폴더 역할 | 탐색 방향 제공 |
| 팀 코딩 규칙 | 일관성 유지 |

반대로 다음 정보는 직접 넣기보다 별도 문서로 분리하는 편이 낫습니다.

- 긴 API 명세
- 전체 데이터베이스 스키마 설명
- 과거 의사결정 기록 전체
- 긴 배포 절차
- 대량의 예제 코드
- 오래된 회의록

## 추천 구조

```text
project/
  CLAUDE.md
  README.md
  docs/
    architecture.md
    testing.md
    git-workflow.md
    deployment.md
    decisions/
      0001-auth-session.md
```

`CLAUDE.md`는 안내판 역할을 하고, 자세한 문서는 `docs/` 아래에 둡니다.

```markdown
# Project Instructions

## Read First

- Project overview: @README.md
- Architecture details: @docs/architecture.md
- Testing strategy: @docs/testing.md

## Always Follow

- Keep changes scoped to the requested task.
- Run the smallest relevant test before broader test suites.
- Do not change public API contracts without explicit approval.
```

## 자동 로드와 필요 시 참조

Claude Code는 메모리 파일을 시작 컨텍스트에 포함합니다. 그래서 `CLAUDE.md`에는 세션마다 필요한 규칙만 두는 편이 좋습니다.

작업별 세부 문서는 프롬프트에서 필요할 때 읽게 할 수 있습니다.

```text
이번 작업은 인증 흐름 변경이다.
먼저 CLAUDE.md와 docs/architecture.md, docs/testing.md를 읽고
인증 관련 수정 계획만 작성해줘.
아직 파일은 수정하지 마.
```

이렇게 하면 항상 모든 문서를 들고 다니지 않고도 필요한 순간에 근거를 확인할 수 있습니다.

## 좋은 메모리 설계 예시

```markdown
# Project Instructions

## Overview

- B2B SaaS admin dashboard for internal operations teams.
- Frontend: React + TypeScript.
- Backend API contract is owned by another repository.

## Commands

- Run: `npm run dev`
- Lint: `npm run lint`
- Type check: `npm run typecheck`
- Unit tests: `npm run test:unit`

## Scope Rules

- Do not edit generated files under `src/generated/`.
- Do not change API response shapes unless explicitly requested.
- Prefer existing components in `src/components/ui` before adding new UI primitives.

## References

- Architecture: @docs/architecture.md
- Testing guide: @docs/testing.md
- Release checklist: @docs/release.md
```

## 나쁜 메모리 설계 예시

```markdown
# Everything

[README 전체 복사]
[API 명세 전체 복사]
[회의록 전체 복사]
[코딩 스타일에 대한 개인 취향 50개]
```

이런 파일은 길지만 실무에는 약합니다. 중요한 규칙이 묻히고, 컨텍스트 비용만 커집니다.

## 실습

현재 프로젝트에서 정보를 세 그룹으로 나눠 보세요.

| 구분 | 예시 | 저장 위치 |
| --- | --- | --- |
| 항상 필요 | 테스트 명령, 금지 폴더 | `CLAUDE.md` |
| 작업별 필요 | 배포 절차, 상세 아키텍처 | `docs/*.md` |
| 개인 취향 | 선호 에디터, 개인 샌드박스 URL | 사용자 메모리 또는 개인 import |

## Claude Code에 입력할 프롬프트

```text
현재 프로젝트의 메모리 설계를 검토해줘.

요청:
1. CLAUDE.md에 직접 넣을 정보
2. docs/로 분리할 정보
3. 개인 메모리로 분리할 정보
4. 민감해서 메모리에 넣으면 안 되는 정보
5. 추천 파일 구조

제약:
- 실제 파일을 먼저 읽고 근거를 표시해줘.
- 아직 파일은 수정하지 마.
- 긴 문서를 CLAUDE.md에 붙여 넣는 방식은 피해서 제안해줘.
```

## 체크리스트

- [ ] `CLAUDE.md`는 짧고 구체적이다.
- [ ] 긴 문서는 `docs/`로 분리되어 있다.
- [ ] `@import`는 정말 필요한 문서에만 사용한다.
- [ ] 개인 취향과 팀 규칙이 섞이지 않는다.
- [ ] 민감 정보는 메모리에 저장하지 않는다.

## 흔한 실수

- 모든 문서를 `CLAUDE.md`에 넣는다.
- import를 과하게 사용해 시작 컨텍스트를 무겁게 만든다.
- 오래된 규칙을 정리하지 않고 계속 누적한다.
- 개인 개발 환경 정보를 팀 저장소에 커밋한다.
