# 038. 기능 개발 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일

## 핵심 개념

Claude Code에게 기능 개발을 맡길 때는 “기능 만들어줘”가 아니라 요구사항, 범위, 비범위, 관련 파일 탐색, 구현 계획, 테스트 기준을 함께 줘야 합니다.

기능 개발은 보통 여러 파일을 건드립니다. 그래서 바로 구현보다 “먼저 코드베이스를 이해하고 계획을 세우는 단계”가 중요합니다.

## 기본 흐름

추천 순서는 다음과 같습니다.

1. 요구사항을 설명한다.
2. 관련 파일을 찾게 한다.
3. 구현 계획을 작성하게 한다.
4. 위험과 테스트 전략을 확인한다.
5. 작은 단계부터 구현한다.
6. 변경사항과 테스트 결과를 검토한다.

처음부터 전체를 구현하게 하지 말고, 계획과 구현을 분리하세요.

## 나쁜 요청

```text
댓글 기능 만들어줘.
```

이 요청은 인증, API, UI, 데이터 저장, 삭제 권한, 테스트 범위가 모두 모호합니다.

## 좋은 요청

```text
댓글 기능을 추가하고 싶다.

요구사항:
- 로그인한 사용자만 댓글을 작성할 수 있다.
- 게시글 상세 화면에서 댓글 목록을 볼 수 있다.
- 댓글 작성자는 자신의 댓글을 삭제할 수 있다.

비범위:
- 댓글 수정 기능은 이번 작업에 포함하지 않는다.
- 대댓글 기능은 포함하지 않는다.

제약:
- 아직 파일은 수정하지 마.
- 먼저 관련 파일과 현재 구조를 조사해줘.
- API, UI, 데이터 모델, 테스트 영향 범위를 나눠서 구현 계획을 작성해줘.
```

## 기능 개발 요청 템플릿

```text
목표:
[사용자가 할 수 있어야 하는 일]

배경:
[왜 필요한지, 현재 문제는 무엇인지]

요구사항:
- [기능 요구사항 1]
- [기능 요구사항 2]
- [기능 요구사항 3]

비범위:
- [이번에 하지 않을 일]

제약:
- 기존 [인증/API/DB/UI] 흐름은 유지한다.
- 새 의존성은 추가하지 않는다.
- 먼저 계획만 작성한다.

확인할 것:
1. 관련 파일
2. 현재 구현 방식
3. 변경이 필요한 파일
4. 테스트 전략
5. 위험 요소
6. 단계별 구현 순서
```

## 구현 승인 프롬프트

계획을 받은 뒤에는 범위를 제한해서 승인합니다.

```text
좋아. 계획 중 1단계만 구현해줘.

범위:
- [파일 또는 폴더]

제약:
- API 응답 형식은 바꾸지 마.
- 리팩터링은 하지 마.
- 수정 후 관련 테스트를 실행해줘.

완료 후:
1. 변경 파일
2. 변경 이유
3. 실행한 테스트
4. 남은 작업
을 요약해줘.
```

## 수용 기준을 포함하기

기능 개발에서는 수용 기준이 중요합니다.

```text
수용 기준:
- Given 로그인한 사용자가 게시글 상세 화면에 있을 때,
  When 댓글 내용을 입력하고 등록을 누르면,
  Then 댓글 목록 최상단에 새 댓글이 표시된다.

- Given 댓글 작성자가 자신의 댓글을 볼 때,
  When 삭제 버튼을 누르면,
  Then 확인 후 댓글이 목록에서 사라진다.
```

수용 기준이 있으면 Claude Code가 테스트와 구현을 더 정확히 연결할 수 있습니다.

## 실습

아래 기능을 Claude Code용 요청으로 바꿔 보세요.

```text
사용자가 프로필 이름과 소개를 수정할 수 있게 한다.
```

요청에는 목표, 요구사항, 비범위, 제약, 수용 기준, 테스트 전략을 포함하세요.

## Claude Code에 입력할 프롬프트

```text
다음 기능 개발을 준비해줘.

기능:
[기능 이름]

요구사항:
[요구사항 목록]

비범위:
[이번에 하지 않을 일]

제약:
- 아직 파일은 수정하지 마.
- 먼저 관련 파일과 현재 구조를 조사해줘.
- 구현 계획, 테스트 전략, 위험 요소를 작성해줘.
- 모르는 요구사항은 추정하지 말고 질문으로 남겨줘.

출력:
1. 관련 파일
2. 현재 구조 요약
3. 구현 단계
4. 테스트 계획
5. 위험 요소
6. 확인 질문
```

## 체크리스트

- [ ] 요구사항과 비범위가 있다.
- [ ] 바로 구현하지 않고 관련 파일을 먼저 조사한다.
- [ ] 구현 계획과 테스트 전략을 분리한다.
- [ ] 한 번에 전체가 아니라 단계별로 승인한다.
- [ ] 수용 기준이 테스트 가능한 문장이다.

## 흔한 실수

- 기능 이름만 던지고 구현을 맡긴다.
- 비범위를 적지 않아 기능이 커진다.
- 테스트 기준 없이 UI만 완성한다.
- 계획 검토 없이 전체 구현을 승인한다.
