# 042. 문서 생성 프롬프트

난이도: 중급  
기준일: 2026년 05월 03일
저자: AI_Innovation_Studio

## 핵심 개념

문서 생성은 Claude Code가 잘하는 작업이지만, 코드와 어긋난 문서는 오히려 위험합니다. 문서를 만들 때는 반드시 현재 코드, 테스트, 설정 파일을 근거로 작성하게 해야 합니다.

좋은 문서 요청은 독자, 목적, 포함할 범위, 근거 파일, 검증 방법을 명확히 합니다.

## 문서 독자 정하기

먼저 독자를 정하세요.

| 독자 | 문서 초점 |
| --- | --- |
| 신규 개발자 | 설치, 실행, 테스트, 구조 |
| API 사용자 | 엔드포인트, 요청/응답, 오류 |
| 운영 담당자 | 배포, 환경 변수, 장애 대응 |
| 기획자 | 기능 흐름, 제약, 상태 |
| 코드 리뷰어 | 변경 의도, 영향 범위, 테스트 |

독자를 정하지 않으면 문서가 너무 일반적이 됩니다.

## 코드 근거 기반 요청

```text
현재 코드 기준으로 API 문서를 작성해줘.

먼저 확인할 것:
- 라우터 또는 컨트롤러 파일
- 요청/응답 타입
- validation schema
- 관련 테스트

제약:
- 코드에 없는 기능은 문서에 쓰지 마.
- 추정한 내용은 "확인 필요"로 표시해줘.
- 예시는 실제 타입과 일치해야 해.
- 아직 파일은 수정하지 말고 초안만 작성해줘.
```

## README 업데이트 요청

```text
README의 실행 방법이 현재 설정과 맞는지 확인하고 업데이트 초안을 만들어줘.

확인할 파일:
- README.md
- package.json 또는 빌드 설정
- 환경 변수 예시 파일

출력:
1. 현재 README에서 맞는 내용
2. 오래됐거나 틀린 내용
3. 누락된 내용
4. 수정 제안 Markdown

아직 README는 수정하지 마.
```

## 문서 템플릿

```markdown
# [문서 제목]

## 대상 독자

- [누가 읽는가]

## 목적

- [이 문서를 읽고 무엇을 할 수 있어야 하는가]

## 사전 조건

- [필요한 환경, 권한, 계정]

## 절차

1. [단계]
2. [단계]
3. [단계]

## 예시

```text
[명령 또는 입력 예시]
```

## 검증

- [성공 여부 확인 방법]

## 문제 해결

- [흔한 오류와 대응]

## 근거 파일

- `[path]`: [문서화에 사용한 이유]
```

문서 끝에 근거 파일을 적게 하면 추정이 줄어듭니다.

## 문서 검토 요청

문서 초안이 나온 뒤에는 검토를 시킵니다.

```text
방금 작성한 문서 초안을 검토해줘.

확인할 것:
1. 코드와 맞지 않는 내용
2. 근거가 없는 추정
3. 초급자가 막힐 수 있는 단계
4. 보안상 공개하면 안 되는 정보
5. 실행해볼 수 없는 명령
```

## 실습

현재 프로젝트에서 README 또는 API 파일 하나를 고르고 다음 문서를 작성해 보세요.

- 신규 개발자용 실행 가이드
- API 사용 예시
- 테스트 실행 가이드
- 배포 전 체크리스트

문서에는 독자, 목적, 절차, 검증 방법, 근거 파일을 포함합니다.

## Claude Code에 입력할 프롬프트

```text
현재 코드 기준으로 [문서 종류] 초안을 작성해줘.

대상 독자:
[신규 개발자/API 사용자/운영 담당자 등]

먼저 확인할 파일:
[파일 또는 폴더]

제약:
- 코드에 없는 내용을 추정하지 마.
- 근거 파일을 문서 끝에 표시해줘.
- 비밀정보나 실제 토큰 값은 포함하지 마.
- 아직 파일은 수정하지 말고 초안만 보여줘.

출력:
Markdown 문서 초안
```

## 체크리스트

- [ ] 문서 독자가 명확하다.
- [ ] 현재 코드와 설정 파일을 근거로 작성한다.
- [ ] 코드에 없는 내용은 추정하지 않는다.
- [ ] 예시 명령과 응답이 실제 구조와 맞다.
- [ ] 보안상 공개하면 안 되는 정보가 없다.

## 흔한 실수

- 코드 확인 없이 일반론 문서를 만든다.
- 오래된 README 내용을 그대로 요약한다.
- 실제로 동작하지 않는 명령을 문서에 넣는다.
- 예시 응답이 현재 타입과 다르다.
