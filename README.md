# Claude 기초부터 고급까지 100

![Claude 기초부터 고급까지 100 표지](assets/00-cover.png)

Claude, Claude Code, MCP, Skills, Hooks, Subagents, Agent Teams, RAG, 평가, 배포, 운영까지 한 권으로 연결하는 100선 실전서입니다.

- 기준일: 2026년 05월 03일
- 저자: AI_Innovation_Studio
- 대상 독자: 초급자, 실무 개발자, 자동화 담당자, AI 제품 기획자, 팀 리더
- 핵심 주제: AI 에이전트로 혼자서 앱, 자동화, 업무 시스템을 만드는 법

이 책은 단순한 기능 소개가 아니라 실제 프로젝트에서 Claude와 Claude Code를 어떻게 도입하고, 어떻게 안전하게 확장하며, 어떻게 운영 가능한 결과물로 만드는지를 단계별로 다룹니다.

## 책의 목표

이 저장소는 WikiDocs 출판을 염두에 둔 Claude Code 실전 원고입니다. 독자가 Claude를 단순한 대화 도구로만 쓰는 단계에서 벗어나, 실제 프로젝트를 읽고 수정하고 검증하는 작업 흐름을 익히도록 구성했습니다.

책의 핵심 목표는 다음과 같습니다.

- Claude Web, Desktop, CLI, IDE의 차이를 이해한다.
- Claude Code로 프로젝트 구조를 읽고 안전하게 작업을 맡기는 법을 익힌다.
- 프롬프트를 역할, 목표, 맥락, 제약, 완료 기준으로 나누어 작성한다.
- `CLAUDE.md`, PRD, progress 문서로 긴 작업의 맥락을 관리한다.
- Skills, Hooks, MCP, Subagents를 필요한 범위에서만 도입한다.
- 테스트, 보안, 배포, 모니터링까지 포함한 운영 관점으로 AI 작업 결과를 검토한다.

이 책은 “AI가 알아서 해준다”는 접근을 권하지 않습니다. 사용자가 목표와 범위를 명확히 정하고, Claude가 만든 결과를 테스트와 리뷰로 확인하는 방식을 기준으로 삼습니다.

## 저장소 구조

```text
README.md
TOC.md
pages/
assets/
tools/
```

- `README.md`: GitHub에서 먼저 보이는 책 소개와 읽는 방법입니다.
- `TOC.md`: WikiDocs 목차입니다. 000번부터 120번, 900번 이미지 계획, 999번 마무리로 이어집니다.
- `pages/`: 실제 본문 원고입니다. 각 파일은 하나의 장 또는 파트 안내입니다.
- `assets/`: 표지와 장별 설명 이미지입니다.
- `tools/`: 원고 생성과 출판 전 검증을 위한 PowerShell 스크립트입니다.

`tools/` 폴더는 원고 관리용 스크립트만 담고 있으며, 토큰이나 개인 문서, PDF, 인증 파일을 넣지 않는 것을 원칙으로 합니다.

## 읽는 방법

처음 읽는 분은 001번부터 030번까지를 먼저 읽으세요. 이 구간은 Claude를 어떻게 질문하고, Claude Code에게 어떤 범위로 일을 맡기며, 결과를 어떻게 확인해야 하는지 설명합니다.

이미 개발 경험이 있다면 031번부터 읽어도 됩니다. 프로젝트 메모리, PRD, progress 문서, Git 검토 흐름을 먼저 잡으면 Claude Code를 더 안정적으로 사용할 수 있습니다.

팀 도입이나 자동화가 목표라면 066번 Skills, 075번 Hooks, 083번 MCP, 092번 Subagents를 순서대로 읽으세요. 이 기능들은 편리하지만 권한과 책임 범위가 커지므로, 작은 업무 하나에 적용한 뒤 점진적으로 넓히는 편이 좋습니다.

각 장의 실습 프롬프트는 그대로 복사해도 되지만, 대괄호 안의 예시나 프로젝트 이름은 반드시 자신의 상황에 맞게 바꾸세요. Claude의 답변은 초안으로 보고, 테스트와 검토를 거쳐야 실제 결과물로 사용할 수 있습니다.

## 단계별 학습 경로

초급자는 다음 순서를 권장합니다.

1. 001-010: Claude와 Claude Code의 기본 개념을 잡습니다.
2. 011-020: 좋은 프롬프트의 구조를 익힙니다.
3. 021-030: Claude Code 기본 명령과 안전한 작업 습관을 배웁니다.
4. 031-037: 프로젝트 메모리와 장기 작업 관리 방법을 익힙니다.
5. 038-055: 실제 개발 업무에 쓰는 프롬프트 패턴을 연습합니다.
6. 056-065: 작업 환경과 비용, 컨텍스트 관리 방법을 정리합니다.
7. 066-100: Skills, Hooks, MCP, Subagents, 운영 전략을 다룹니다.
8. 101-120: 템플릿, 체크리스트, 최종 프로젝트로 실습합니다.

개발 경험이 적은 독자는 001-030을 먼저 끝낸 뒤, 바로 101번 명령어 치트시트와 104번 WikiDocs 출판 가이드를 보는 편이 좋습니다. 팀 리더나 자동화 담당자는 118번 팀 도입 가이드와 120번 최종 프로젝트 목록을 함께 참고하면 도입 범위를 정하기 쉽습니다.

## 원고 작성 원칙

이 원고는 다음 기준을 따릅니다.

- 초보자가 읽을 수 있도록 용어를 먼저 설명하고 예시를 붙입니다.
- 위험한 자동화보다 작은 범위의 검증 가능한 작업을 우선합니다.
- 테스트하지 않은 결과는 확정된 사실처럼 쓰지 않습니다.
- 토큰, 비밀번호, 고객 데이터, 내부 문서, PDF 원본 같은 민감 자료는 저장소에 포함하지 않습니다.
- Markdown 본문에는 불필요한 강조 표시나 이모지를 쓰지 않습니다.
- 과장된 표현보다 검증 가능한 기준을 사용합니다.

## 출판 전 검증

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\check_wikidocs_book.ps1
```

이 스크립트는 `README.md`, `TOC.md`, `pages/`, `assets/` 존재 여부와 Markdown 링크, 이미지 경로, 빈 이미지 파일을 확인합니다. 또한 페이지 번호, TOC 순서, 금지된 강조 표시, 이모지, 과장 표현, 실제 토큰 패턴, 커밋하면 안 되는 문서 확장자를 함께 점검합니다.

검증을 통과한 뒤에만 커밋하는 것을 권장합니다.
